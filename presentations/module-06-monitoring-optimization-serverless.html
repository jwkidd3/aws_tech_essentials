☁️ Amazon CloudWatch Overview

What is CloudWatch?

A monitoring and observability service that collects metrics, logs, and events from AWS resources, applications, and on-premises servers.

  • Metrics — Numerical data points over time (CPU, network, disk)
  • Logs — Centralized log collection and analysis
  • Alarms — Automated actions based on thresholds
  • Dashboards — Custom visualizations of operational data
  • Events / EventBridge — React to state changes in real time
CloudWatch is the backbone of AWS observability. Most AWS services automatically publish metrics to CloudWatch at no additional cost. Explain that CloudWatch has evolved significantly — EventBridge is the successor to CloudWatch Events with broader capabilities.

CloudWatch Metrics: Built-in vs Custom

📦 Built-in Metrics

  • Published automatically by AWS services
  • EC2: CPUUtilization, NetworkIn/Out, DiskReadOps
  • RDS: DatabaseConnections, FreeStorageSpace
  • ELB: RequestCount, TargetResponseTime
  • Default period: 5 minutes (basic), 1 minute (detailed)

🔧 Custom Metrics

  • Published via PutMetricData API or CloudWatch Agent
  • Memory utilization (not built-in for EC2)
  • Disk space usage
  • Application-specific KPIs
  • Resolution: standard (60s) or high-resolution (1s)
This is a common gotcha for people new to AWS. EC2 hypervisor-level metrics like CPU are free, but OS-level metrics like memory require the CloudWatch Agent. The agent can also collect logs simultaneously. Important: EC2 does not natively report memory or disk utilization to CloudWatch. You must install the CloudWatch Agent to collect these metrics.

📊 CloudWatch Dashboards

Customizable pages in the CloudWatch console for visual monitoring of resources.

  • Global view — Dashboard widgets can display metrics from any AWS Region
  • Widget types — Line graphs, stacked areas, numbers, text, logs, alarms
  • Auto-refresh — Configurable intervals (10s, 1m, 2m, 5m, 15m)
  • Shareable — Share via URL or embed in external pages
  • Pricing — First 3 dashboards free, then $3/dashboard/month
Best Practice: Create separate dashboards for different audiences — an operational dashboard for engineers and a summary dashboard for leadership.
Demonstrate the concept of an operational dashboard versus a business dashboard. Engineers need granular metrics; leadership needs high-level health indicators and trends. Dashboards can also be set as the default landing page in the CloudWatch console.

🚨 CloudWatch Alarms

Watch a single metric and perform actions when a threshold is breached.

✅ OK

Metric is within the defined threshold

🚨 ALARM

Metric has breached the threshold

❓ INSUFFICIENT_DATA

Not enough data to determine state

Walk through a concrete example: set a CPUUtilization alarm at 80% for 3 consecutive 5-minute periods. If breached, send an SNS notification to the ops team and trigger an Auto Scaling policy. Explain that INSUFFICIENT_DATA is the initial state for a new alarm, or occurs if metric reporting stops. Alarm Actions include: SNS notification (Email, SMS, HTTP endpoint, Lambda trigger), Auto Scaling action (Scale out or scale in), and EC2 action (Stop, terminate, reboot, or recover an instance).

Alarm Configuration Details

Setting Description Example
Metric The CloudWatch metric to monitor CPUUtilization
Statistic Aggregation function Average, Sum, Max, p99
Period Evaluation interval 300 seconds (5 min)
Evaluation Periods Consecutive periods to evaluate 3 periods
Threshold Value that triggers the alarm >= 80%
Datapoints to Alarm How many periods must breach 3 out of 3
Tip: Use "M out of N" evaluation to reduce false alarms. For example, alarm only if 3 out of 5 evaluation periods breach the threshold.
Emphasize the difference between evaluation periods and datapoints to alarm. This "M out of N" approach prevents a single spike from triggering an alarm while still catching sustained issues. Also mention composite alarms that combine multiple alarm states with AND/OR logic.

📜 CloudWatch Logs

Centralized Log Management

Collect, store, and analyze log data from EC2 instances, Lambda functions, CloudTrail, Route 53, and other sources in one place.

Core Concepts

  • Log Groups — Container for log streams (e.g., /app/web-server)
  • Log Streams — Sequence of events from a single source
  • Retention — Configurable from 1 day to 10 years, or indefinite

Analysis Tools

  • Metric Filters — Extract metrics from log patterns
  • Logs Insights — SQL-like query language for log analysis
  • Export — Send to S3 for long-term storage
Explain the hierarchy: log groups contain log streams. A typical pattern is one log group per application, one log stream per instance. CloudWatch Logs Insights supports a purpose-built query language — you can filter, aggregate, and visualize log data interactively. Mention that metric filters let you turn log patterns (e.g., ERROR count) into CloudWatch metrics that you can alarm on.

CloudWatch Logs Insights

Query your logs with a purpose-built query language:

fields @timestamp, @message
| filter @message like /ERROR/
| stats count(*) as errorCount by bin(5m)
| sort @timestamp desc
| limit 20
  • Automatically discovers fields in JSON-formatted logs
  • Supports parse, filter, stats, sort, limit
  • Visualize results as time-series graphs or tables
  • Query multiple log groups simultaneously
Walk through the query syntax. This example counts ERROR messages in 5-minute bins. Logs Insights is significantly cheaper than running a full-time Elasticsearch cluster for log analysis. It is ideal for ad-hoc troubleshooting and investigation. Cost-Effective: You only pay for the data scanned by the query -- $0.005 per GB scanned.

⚡ CloudWatch Events & EventBridge

EventBridge is the evolution of CloudWatch Events. AWS recommends EventBridge for all new development. Existing CloudWatch Events rules continue to work.
  • Event Sources — AWS services, custom apps, SaaS partners
  • Rules — Match incoming events based on pattern or schedule
  • Targets — Lambda, SNS, SQS, Step Functions, ECS tasks, and 20+ more

Example: Automated Response

EC2 instance state changes to "stopped" ➞ EventBridge rule matches ➞ triggers Lambda function ➞ sends Slack notification and creates a Jira ticket.

EventBridge enables event-driven architectures. Give concrete examples: auto-tag resources on creation, trigger remediation when security groups change, schedule Lambda functions to run nightly. The schema registry feature lets you discover and version event schemas from your applications.

👁️ AWS CloudTrail

What is CloudTrail?

Records API activity across your AWS account — who did what, when, and from where.

  • Enabled by default (90-day event history)
  • Create a trail for long-term S3 storage
  • Integrates with CloudWatch Logs for real-time alerting

Event Types

  • Management Events — Control plane operations (CreateBucket, RunInstances)
  • Data Events — Data plane operations (GetObject, PutItem)
  • Insights Events — Unusual API activity detection
Governance & Compliance: CloudTrail is essential for auditing, regulatory compliance, and security investigations. It answers: "Who changed this resource and when?"
Distinguish between CloudWatch (operational monitoring — how are things performing?) and CloudTrail (audit logging — who did what?). CloudTrail is critical for security. You can set up a multi-region trail that captures activity across all regions into a single S3 bucket. Mention that CloudTrail logs should be protected with S3 bucket policies, encryption, and log file validation.

CloudWatch vs CloudTrail

Aspect CloudWatch CloudTrail
Purpose Performance monitoring & observability API activity auditing & governance
Data Type Metrics, logs, events API call records (who, what, when)
Question Answered "How is my application performing?" "Who changed this resource?"
Primary Users DevOps, SRE, developers Security, compliance, auditors
Actions Alarms, Auto Scaling, notifications Log analysis, forensics, alerting
Students often confuse these two services. The key distinction: CloudWatch is about performance and operational data; CloudTrail is about API-level audit logging. In practice, you want both enabled. CloudTrail events can be sent to CloudWatch Logs so you can create metric filters and alarms on specific API calls (e.g., alert when someone creates a new IAM user). They are complementary: Use CloudWatch to monitor operational health. Use CloudTrail to track who made changes. Use both together for full visibility.

✅ Monitoring Best Practices

  • Monitor at every layer — Infrastructure, application, and business metrics
  • Set meaningful alarms — Tune thresholds carefully and use composite alarms to reduce false positives
  • Centralize logs & enable CloudTrail — Aggregate logs into CloudWatch; use a multi-region trail with log file validation
  • Automate responses — Use EventBridge + Lambda for self-healing architectures
  • Establish baselines and review regularly — Know normal behavior before defining abnormal; tune dashboards and alarms periodically
Summarize the monitoring section. The goal is not just to collect data but to turn data into actionable insights. Emphasize the concept of alarm fatigue — too many alarms and teams start ignoring them. Recommend starting with a small set of critical alarms and expanding thoughtfully.

🚀 Optimization

High availability, load balancing, and auto scaling

Transition to optimization. Now that we know how to monitor, we can use that data to build architectures that respond dynamically to demand.

Designing for High Availability

Availability = Uptime / (Uptime + Downtime)

High availability means designing systems that continue operating even when individual components fail.

🎯 HA Design Principles

  • Deploy across multiple Availability Zones
  • Eliminate single points of failure
  • Use managed services where possible
  • Implement health checks at every tier

📈 Availability Targets

  • 99.9% = 8.77 hrs downtime/year
  • 99.95% = 4.38 hrs downtime/year
  • 99.99% = 52.6 min downtime/year
  • 99.999% = 5.26 min downtime/year
Emphasize that high availability is an architectural decision, not a single feature. Multi-AZ deployment is the foundational pattern. Each "nine" of availability requires significantly more engineering effort and cost. Help students match their actual business requirements to the right availability target.

⚖️ Elastic Load Balancing (ELB)

Automatically distributes incoming traffic across multiple targets in one or more Availability Zones.

  • Managed service — AWS handles scaling, availability, and maintenance
  • Health checks — Automatically routes traffic away from unhealthy targets
  • Integration — Works with Auto Scaling, ECS, EKS, Lambda
  • SSL/TLS termination — Offload encryption from your application
Three Load Balancer Types: Application Load Balancer (ALB), Network Load Balancer (NLB), and Classic Load Balancer (CLB, legacy).
ELB is a fully managed service — it automatically scales its own capacity to handle traffic fluctuations. Explain that ELB itself is deployed across multiple AZs for high availability. Classic Load Balancer is considered legacy; new applications should use ALB or NLB.

ALB vs NLB

🖧️ Application Load Balancer

  • Layer 7 (HTTP/HTTPS) with native HTTP/2 and WebSocket support
  • Content-based routing: path, host, header, and query string rules
  • Redirect, fixed-response actions, and Lambda targets
  • Best for web applications needing advanced routing

⚡ Network Load Balancer

  • Layer 4 (TCP/UDP/TLS) with ultra-low latency
  • Static IP addresses per AZ with Elastic IP support
  • Preserves source IP; handles millions of requests/sec
  • Ideal for non-HTTP workloads or when static IPs are required
ALB is the right choice for most web applications. NLB is for TCP-level workloads, extreme performance needs, or when you require static IPs. A common pattern is NLB in front of ALB when you need both static IPs and content-based routing. CLB supports both L4 and L7 but lacks the advanced routing features of ALB and NLB.

ELB Type Comparison

Feature ALB NLB CLB (Legacy)
OSI Layer Layer 7 Layer 4 Layer 4 & 7
Protocols HTTP, HTTPS, gRPC TCP, UDP, TLS HTTP, HTTPS, TCP, SSL
Content Routing Path, host, header, query No No
Static IP No (use NLB in front) Yes No
Performance High Ultra-high (millions rps) Moderate
Target Types Instance, IP, Lambda Instance, IP, ALB Instance only
Use this comparison table as a quick reference. For the exam and real-world scenarios, the decision usually comes down to: ALB for web apps with routing needs, NLB for raw TCP/UDP performance or static IPs. CLB should only be used for existing applications that have not been migrated.

💚 Health Checks and Routing

How ELB Health Checks Work

ELB periodically sends requests to registered targets to test their status. Unhealthy targets stop receiving traffic automatically.

  • Health check protocol — HTTP, HTTPS, TCP (configurable path for HTTP: e.g., /health)
  • Interval — Time between health checks (default: 30 seconds)
  • Healthy threshold — Consecutive successes to mark healthy (default: 5)
  • Unhealthy threshold — Consecutive failures to mark unhealthy (default: 2)
  • Timeout — Time to wait for a response (default: 5 seconds)
Health checks are critical — they are the mechanism that makes ELB "smart." A shallow health check (just returning 200) may report healthy even when the app cannot connect to its database. A deep health check verifies actual application readiness. Discuss the tradeoffs: deep health checks are more accurate but can cause cascading failures if a downstream dependency is slow. Best Practice: Create a dedicated /health endpoint that checks database connectivity, disk space, and downstream dependencies -- not just "200 OK."

📈 Amazon EC2 Auto Scaling

What is Auto Scaling?

Automatically adjusts the number of EC2 instances to match demand, maintaining performance while minimizing cost.

Key Benefits

  • Availability — Replace unhealthy instances automatically
  • Elasticity — Scale out for demand spikes, scale in when quiet
  • Cost — Only pay for instances you need right now

Two Components

  • Launch Template — What to launch (AMI, instance type, key pair, security groups)
  • Auto Scaling Group — Where and how many (VPC, subnets, capacity settings)
Auto Scaling is one of the most important services for cost optimization and availability. Distinguish between launch configurations (legacy, do not support versioning) and launch templates (recommended, support versioning and mixed instance types). The launch template defines the "recipe" for each instance; the ASG controls the fleet.

Auto Scaling Group Capacity Settings

Minimum Capacity

The floor — ASG will never scale below this number.

Example: 2 instances (one per AZ for HA)

Desired Capacity

The target — ASG launches or terminates to reach this number.

Example: 2 instances (normal load)

Maximum Capacity

The ceiling — ASG will never scale above this number.

Example: 6 instances (peak capacity)

Rule: Minimum ≤ Desired ≤ Maximum. Scaling policies adjust the desired capacity within the min/max boundaries.
Use a visual analogy: minimum is the safety net, desired is the current target, maximum is the budget constraint. When a scaling policy triggers, it changes the desired capacity, and ASG adds or removes instances to match. If an instance fails a health check, ASG terminates it and launches a replacement to maintain desired capacity.

Scaling Policies

Policy Type How It Works Best For
Target Tracking Maintain a metric at a target value (e.g., CPU at 50%) Most workloads — simple and effective
Step Scaling Scale by different amounts based on alarm severity Variable scaling responses (moderate vs severe spikes)
Simple Scaling Add/remove fixed number of instances on alarm Basic scenarios (legacy approach)
Scheduled Scaling Scale at specific times (cron-like) Predictable traffic patterns (business hours, campaigns)
Predictive Scaling ML-based forecasting of future traffic Cyclical workloads with recurring patterns
Recommendation: Start with target tracking. It is the easiest to configure and handles most use cases well. Combine with scheduled scaling for known patterns.
Target tracking is like a thermostat — you set the desired temperature (e.g., 50% CPU), and Auto Scaling adjusts capacity to maintain it. Step scaling gives more control: e.g., add 1 instance at 60% CPU, add 3 at 80% CPU. Simple scaling has a cooldown period that can delay responses. Predictive scaling uses ML on 14 days of historical data.

Scale-Out and Scale-In Behavior

📈 Scale-Out (Adding Instances)

  • Triggered when demand exceeds capacity
  • New instances launched from launch template
  • Distributed across AZs for balance
  • Registered with ELB target group
  • Receives traffic after passing health checks

📉 Scale-In (Removing Instances)

  • Triggered when demand decreases
  • Default termination policy: AZ with most instances ➞ oldest launch config ➞ closest to billing hour
  • Scale-in protection available for critical instances
  • Cooldown periods prevent rapid fluctuation
Discuss the default termination policy in detail — it first balances across AZs, then uses launch configuration age, then proximity to billing hour. You can customize this. Lifecycle hooks let you run scripts during launch (install software) or termination (drain connections, deregister from service discovery). Instance warm-up settings prevent newly launched instances from being considered for scale-in before they are fully operational. Watch Out: Scale-out is typically fast (minutes), but your application must handle the bootstrap time. Use AMI pre-baking or lifecycle hooks to ensure instances are ready before receiving traffic.

🏗️ ELB + Auto Scaling = HA Architecture

The Complete Picture

ELB distributes traffic. Auto Scaling manages capacity. Together, they create a self-healing, elastic architecture.

Architecture Flow:
  1. Users connect to the ALB DNS name
  2. ALB distributes requests across healthy instances in multiple AZs
  3. Auto Scaling monitors CloudWatch metrics (CPU, request count)
  4. When a threshold is breached, ASG adjusts desired capacity
  5. New instances register with the ALB target group
  6. Failed instances are detected, terminated, and replaced
Tie everything together. This is the canonical AWS HA pattern. Draw the architecture mentally: Route 53 DNS pointing to an ALB spanning 2+ AZs, each AZ with EC2 instances managed by an Auto Scaling Group, backed by a Multi-AZ RDS database. Each layer has redundancy. This is exactly what the upcoming lab implements. This is the architecture you will build in the lab.

💰 Cost Optimization Strategies

  • Right-sizing & Auto Scaling — Match instance types to actual workload requirements; scale in during low demand
  • Reserved Instances / Savings Plans — Commit to 1- or 3-year usage for up to 72% savings
  • Spot Instances — Up to 90% savings for fault-tolerant, flexible workloads
  • Storage optimization & serverless — S3 lifecycle policies for cheaper tiers; Lambda for pay-per-invocation compute
  • AWS Cost Explorer & Budgets — Visualize spending, set alerts, identify underutilized resources
Cost optimization is an ongoing process, not a one-time activity. Emphasize that monitoring data (from CloudWatch) directly informs cost optimization decisions. Right-sizing is often the biggest quick win — many organizations over-provision instances because they do not have visibility into actual utilization. Savings Plans are generally recommended over Reserved Instances for flexibility. First step: Enable Cost Explorer and review your spending by service. Use CloudWatch to identify underutilized resources.

Serverless on AWS

What Does "Serverless" Mean?

No infrastructure to provision or manage. Automatic scaling. Pay-per-use pricing. Built-in high availability and fault tolerance.

⚙️ Compute

  • AWS Lambda
  • AWS Fargate

🗂️ Storage & Data

  • Amazon S3
  • Amazon DynamoDB
  • Amazon Aurora Serverless

🔗 Integration

  • Amazon API Gateway
  • Amazon EventBridge
  • AWS Step Functions
  • Amazon SQS / SNS
Clarify that "serverless" does not mean "no servers" — it means you do not manage servers. AWS handles provisioning, patching, scaling, and availability. DynamoDB and S3 are already serverless services that students have been using. The new pieces are Lambda for compute and API Gateway for the API layer.

⚡ AWS Lambda

Run code without provisioning or managing servers. Pay only for compute time consumed.

How It Works

  • Upload your code (or container image)
  • Configure trigger (API Gateway, S3, EventBridge, etc.)
  • Lambda runs your code in response to events
  • Scales automatically from 0 to thousands of concurrent executions

Key Specs

  • Runtimes: Python, Node.js, Java, .NET, Go, Ruby, custom
  • Memory: 128 MB to 10,240 MB
  • Timeout: Up to 15 minutes
  • Pricing: $0.20 per 1M requests + duration
Lambda is the compute engine for serverless architectures. Emphasize that CPU is allocated proportionally to memory — more memory means more CPU. The 15-minute timeout makes Lambda suitable for request-response workloads but not for long-running processes. Lambda functions should be stateless; any state goes in DynamoDB, S3, or ElastiCache.

🔗 Amazon API Gateway

Fully Managed API Front Door

Create, publish, maintain, monitor, and secure APIs at any scale. Handles traffic management, authorization, throttling, and monitoring.

  • REST APIs — Full-featured, supports API keys, usage plans, caching
  • HTTP APIs — Lower latency, lower cost, simpler (ideal for Lambda proxies)
  • WebSocket APIs — Real-time two-way communication
  • Authentication — IAM, Cognito User Pools, Lambda authorizers
  • Stages — Deploy to dev, staging, production with different configurations
API Gateway is the "front door" for serverless APIs. It handles concerns that would otherwise be in your application code: rate limiting, authentication, request validation, response transformation. HTTP APIs are up to 71% cheaper and 60% lower latency than REST APIs for simple proxy use cases. For the employee directory, HTTP APIs with Lambda proxy integration would be the recommended approach. In our serverless architecture: API Gateway replaces the ALB as the entry point, routing HTTP requests to Lambda functions instead of EC2 instances.

🌐 S3 for Static Content Hosting

In a serverless architecture, static assets (HTML, CSS, JavaScript, images) are served directly from S3.

S3 Static Website Hosting

  • Enable static website hosting on an S3 bucket
  • Configure index and error documents
  • Highly durable (99.999999999%) and available
  • Scales automatically to any traffic level

Add CloudFront CDN

  • Global edge locations for low latency
  • HTTPS with custom domain names
  • Origin Access Control for security
  • Caching reduces S3 request costs
Pattern: S3 + CloudFront serves the frontend. API Gateway + Lambda serves the backend. DynamoDB stores the data. No servers to manage.
This pattern separates the frontend from the backend entirely. The static frontend (single-page application) makes API calls to API Gateway. This is the standard pattern for modern serverless web applications. CloudFront adds HTTPS support, custom domains, and global distribution with caching. The combination is extremely cost-effective for most web application traffic patterns.

🔄 AWS Step Functions

Visual workflow service for orchestrating multiple AWS services into serverless applications.

Key Features

  • State machines — Define workflows as a series of steps
  • Built-in error handling — Retry, catch, and fallback logic
  • Visual monitoring — See execution state in real time
  • 200+ AWS service integrations

Use Cases

  • Multi-step data processing pipelines
  • Order fulfillment workflows
  • Employee onboarding automation
  • ETL orchestration
Two Workflow Types: Standard (up to 1 year, exactly-once execution) and Express (up to 5 minutes, at-least-once, high volume/low cost).
Step Functions solves the orchestration problem in serverless architectures. Without it, you end up with complex Lambda-to-Lambda chains that are hard to debug and monitor. Step Functions provides a visual designer, built-in retries, and parallel execution. For the employee directory, a Step Function could orchestrate the onboarding workflow: create employee record, resize photo, send welcome email — all as discrete Lambda functions coordinated by Step Functions.

EC2-Based vs Serverless: Tradeoffs

💻 EC2-Based

  • ✅ Full control over runtime and persistent connections
  • ✅ Long-running processes supported
  • ❌ Server management overhead and patching
  • ❌ Pay for idle capacity; slower scaling

⚡ Serverless

  • ✅ Zero server management with automatic scaling
  • ✅ Pay only when code runs
  • ❌ Cold start latency and 15-min execution limit
  • ❌ Vendor-specific patterns and tooling
This is critical — avoid presenting serverless as always superior. EC2 is the right choice for long-running workloads, applications that need full OS access, or teams that already have strong server management practices. Serverless shines for event-driven workloads, variable traffic, and teams that want to focus on business logic rather than infrastructure. Many production architectures are hybrid — using both EC2 and serverless components. Neither is universally better. Choose based on workload characteristics, team skills, and business requirements.

Serverless Architecture Patterns

🖥️ Synchronous (API-Driven)

Client ➞ API Gateway ➞ Lambda ➞ DynamoDB ➞ Response

  • REST or HTTP APIs
  • Request-response model
  • Low latency requirements

📨 Asynchronous (Event-Driven)

S3 Upload ➞ EventBridge ➞ Lambda ➞ Process ➞ DynamoDB

  • Decoupled components
  • Queue-based processing (SQS)
  • Fan-out with SNS

🔄 Orchestration Pattern

Step Functions coordinates multiple Lambda functions, handles branching logic, retries, and parallel execution. Ideal for complex business workflows.

Most serverless applications combine these patterns. The employee directory API endpoints are synchronous. Photo processing (resize, thumbnail generation) is asynchronous — triggered by an S3 upload event. A new employee onboarding workflow could use Step Functions for orchestration. Understanding these patterns is key to designing effective serverless architectures.

🌟 Why Serverless?

🛠️ No Server Management

No patching, no capacity planning, no OS updates. Focus entirely on business logic.

📈 Automatic Scaling

Scales from zero to peak demand instantly. No launch templates, no ASG configuration.

💰 Pay-Per-Use

Pay only for actual compute time and requests. Zero cost when idle.

For the employee directory: A serverless architecture could reduce costs by 60-80% for a typical internal application with variable traffic patterns, while eliminating all server maintenance.
Summarize the serverless value proposition. For an internal employee directory with relatively low and variable traffic, serverless is often dramatically cheaper than running EC2 instances 24/7. The break-even point depends on traffic volume — at very high sustained request rates, EC2 may be more cost-effective. But for most internal applications, serverless wins on both cost and operational simplicity.
