🔎 Monitoring

Visibility into your AWS environment

Transition into the monitoring section. Monitoring is foundational — you cannot optimize what you cannot measure.

Why Monitoring Matters

Without Monitoring

  • Blind to resource utilization
  • Reactive incident response
  • Unpredictable costs
  • No capacity planning data

With Monitoring

  • Real-time operational insights
  • Proactive alerting and automation
  • Data-driven scaling decisions
  • Compliance and audit trails
Emphasize that cloud environments are inherently more dynamic than on-premises. Resources spin up and down constantly, so traditional monitoring approaches (checking a fixed set of servers) are insufficient. Key principle: in the cloud, every API call can be logged, measured, and acted upon.

☁️ CloudWatch: Metrics & Dashboards

📦 Metrics

  • Built-in: CPU, network, disk (auto-published)
  • Custom: Memory, app KPIs via agent/API
  • Resolution: 5 min (basic), 1 min (detailed)

📊 Dashboards

  • Cross-region widget visualizations
  • Line graphs, numbers, alarms, text
  • First 3 free, then $3/month each
Gotcha: EC2 does not report memory or disk usage natively — install the CloudWatch Agent for OS-level metrics.
CloudWatch is the backbone of AWS observability. Most AWS services automatically publish metrics at no cost. EC2 hypervisor-level metrics like CPU are free, but OS-level metrics like memory require the CloudWatch Agent. The agent can also collect logs simultaneously. Create separate dashboards for different audiences — operational for engineers, summary for leadership.

🚨 CloudWatch Alarms

Watch a metric and trigger actions when a threshold is breached.

✅ OK

Within threshold

🚨 ALARM

Threshold breached

❓ INSUFFICIENT_DATA

Not enough data yet

  • Configure: metric, statistic, period, threshold, evaluation periods
  • Actions: SNS notification, Auto Scaling, EC2 stop/terminate/reboot
  • Tip: Use "M out of N" evaluation to reduce false alarms
Walk through a concrete example: set a CPUUtilization alarm at 80% for 3 consecutive 5-minute periods. If breached, send an SNS notification and trigger an Auto Scaling policy. INSUFFICIENT_DATA is the initial state for new alarms. Composite alarms combine multiple alarm states with AND/OR logic. The "M out of N" approach (e.g., alarm if 3 out of 5 periods breach) prevents a single spike from triggering while catching sustained issues.

📜 CloudWatch Logs & Insights

Log Management

  • Log Groups — container per application
  • Log Streams — events from one source
  • Retention — 1 day to 10 years
  • Metric Filters — extract metrics from logs

Logs Insights Query

fields @timestamp, @message
| filter @message like /ERROR/
| stats count(*) by bin(5m)
| sort @timestamp desc
  • SQL-like query language
  • Query multiple log groups
  • $0.005/GB scanned
Explain the hierarchy: log groups contain log streams. A typical pattern is one log group per application, one log stream per instance. Metric filters let you turn log patterns (e.g., ERROR count) into CloudWatch metrics you can alarm on. Logs Insights is significantly cheaper than running a full-time OpenSearch cluster. It supports parse, filter, stats, sort, limit and auto-discovers fields in JSON-formatted logs.

⚡ Amazon EventBridge

EventBridge is the evolution of CloudWatch Events — recommended for all new development.
  • Sources: AWS services, custom apps, SaaS partners
  • Rules: Match events by pattern or schedule
  • Targets: Lambda, SNS, SQS, Step Functions, and 20+ more

Example: Automated Response

EC2 state changes to "stopped" ➞ EventBridge rule ➞ Lambda ➞ Slack notification + Jira ticket

EventBridge enables event-driven architectures. Give concrete examples: auto-tag resources on creation, trigger remediation when security groups change, schedule Lambda functions to run nightly. The schema registry feature lets you discover and version event schemas.

👁️ AWS CloudTrail

API Activity Auditing

Records who did what, when, and from where.

  • Enabled by default (90-day history)
  • Create a trail for long-term S3 storage
  • Integrates with CloudWatch Logs

Event Types

  • Management — control plane (CreateBucket)
  • Data — data plane (GetObject)
  • Insights — unusual API activity
CloudWatch vs CloudTrail: CloudWatch = "How is it performing?" CloudTrail = "Who changed it?"
Distinguish between CloudWatch (operational monitoring — how are things performing?) and CloudTrail (audit logging — who did what?). CloudTrail is critical for security, governance, and compliance. You can set up a multi-region trail that captures activity across all regions into a single S3 bucket. CloudTrail events can be sent to CloudWatch Logs so you can alarm on specific API calls (e.g., new IAM user created). They are complementary — use both together for full visibility.

✅ Monitoring Best Practices

  • Monitor every layer — infrastructure, application, business metrics
  • Tune alarms carefully — use composite alarms to reduce noise
  • Centralize logs — aggregate into CloudWatch, enable multi-region trails
  • Automate responses — EventBridge + Lambda for self-healing
  • Establish baselines — know normal before defining abnormal
The goal is not just to collect data but to turn data into actionable insights. Emphasize the concept of alarm fatigue — too many alarms and teams start ignoring them. Recommend starting with a small set of critical alarms and expanding thoughtfully.

🚀 Optimization

High availability, load balancing, and auto scaling

Transition to optimization. Now that we know how to monitor, we can use that data to build architectures that respond dynamically to demand.

Designing for High Availability

🎯 HA Design Principles

  • Deploy across multiple AZs
  • Eliminate single points of failure
  • Use managed services
  • Health checks at every tier

📈 Availability Targets

  • 99.9% = 8.77 hrs/year down
  • 99.95% = 4.38 hrs/year down
  • 99.99% = 52.6 min/year down
  • 99.999% = 5.26 min/year down
High availability is an architectural decision, not a single feature. Multi-AZ deployment is the foundational pattern. Each "nine" of availability requires significantly more engineering effort and cost. Help students match their actual business requirements to the right availability target.

⚖️ Elastic Load Balancing (ELB)

Distributes traffic across targets in multiple Availability Zones.

  • Managed service — AWS handles scaling and maintenance
  • Health checks — routes away from unhealthy targets
  • Integration — Auto Scaling, ECS, EKS, Lambda
  • SSL/TLS termination — offload encryption
ELB is a fully managed service — it automatically scales its own capacity to handle traffic fluctuations. ELB itself is deployed across multiple AZs for high availability. Classic Load Balancer is legacy; new applications should use ALB or NLB.

ALB vs NLB

🖧️ Application LB (Layer 7)

  • HTTP/HTTPS with path/host routing
  • Redirect and fixed-response actions
  • Lambda as target
  • Best for web applications

⚡ Network LB (Layer 4)

  • TCP/UDP with ultra-low latency
  • Static IP per AZ (Elastic IP)
  • Millions of requests/sec
  • Best for non-HTTP or static IP needs
Decision: ALB for web apps with routing needs. NLB for raw TCP/UDP performance or static IPs. CLB is legacy only.
ALB is the right choice for most web applications. NLB is for TCP-level workloads, extreme performance needs, or when you require static IPs. A common pattern is NLB in front of ALB when you need both static IPs and content-based routing. ALB supports Instance, IP, and Lambda targets. NLB supports Instance, IP, and ALB targets.

💚 ELB Health Checks

ELB periodically tests targets; unhealthy ones stop receiving traffic.

  • Protocol: HTTP, HTTPS, or TCP (configurable path)
  • Interval: time between checks (default 30s)
  • Healthy threshold: consecutive successes (default 5)
  • Unhealthy threshold: consecutive failures (default 2)
Best Practice: Use a dedicated /health endpoint that checks database connectivity and downstream dependencies.
Health checks are the mechanism that makes ELB smart. A shallow health check (just returning 200) may report healthy even when the app cannot connect to its database. A deep health check verifies actual application readiness. Discuss the tradeoffs: deep health checks are more accurate but can cause cascading failures if a downstream dependency is slow.

📈 EC2 Auto Scaling

Automatically adjusts instance count to match demand.

Launch Template ("What")

  • AMI, instance type, key pair
  • Security groups, IAM role
  • User data, storage config

Auto Scaling Group ("Where/How Many")

  • VPC and subnets (multi-AZ)
  • Min / Desired / Max capacity
  • ELB target group attachment
Rule: Minimum ≤ Desired ≤ Maximum. Scaling policies adjust desired capacity within boundaries.
Auto Scaling is one of the most important services for cost optimization and availability. The launch template defines the "recipe" for each instance; the ASG controls the fleet. Use launch templates (not launch configurations) — they support versioning and mixed instance types. When an instance fails a health check, ASG terminates it and launches a replacement to maintain desired capacity.

Scaling Policies

Policy Type How It Works Best For
Target Tracking Maintain metric at target (e.g., CPU 50%) Most workloads
Step Scaling Different amounts by alarm severity Variable response needs
Scheduled Scale at specific times (cron) Predictable patterns
Predictive ML-based traffic forecasting Cyclical workloads
Start with target tracking — easiest to configure, handles most use cases. Combine with scheduled scaling for known patterns.
Target tracking is like a thermostat — you set the desired temperature (e.g., 50% CPU), and Auto Scaling adjusts to maintain it. Step scaling gives more control: e.g., add 1 instance at 60% CPU, add 3 at 80% CPU. Simple scaling (legacy) has a cooldown that can delay responses. Predictive scaling uses ML on 14 days of historical data. Scale-out is fast (minutes) but your app must handle bootstrap time. Use AMI pre-baking or lifecycle hooks to ensure instances are ready. Default scale-in termination: balance AZs, then oldest launch config, then closest to billing hour.

🏗️ ELB + Auto Scaling = HA Architecture

ELB distributes traffic. Auto Scaling manages capacity. Together they create a self-healing, elastic architecture.

Architecture Flow:
  1. Users connect to ALB DNS name
  2. ALB distributes across healthy instances in multiple AZs
  3. CloudWatch metrics trigger ASG scaling policies
  4. New instances register with ALB target group
  5. Failed instances detected, terminated, and replaced
This is the canonical AWS HA pattern: Route 53 DNS pointing to an ALB spanning 2+ AZs, each AZ with EC2 instances managed by an Auto Scaling Group, backed by a Multi-AZ RDS database. Each layer has redundancy. This is exactly what the upcoming lab implements.

💰 Cost Optimization Strategies

  • Right-sizing — match instances to actual workload needs
  • Reserved / Savings Plans — 1- or 3-year commitment, up to 72% off
  • Spot Instances — up to 90% savings for fault-tolerant work
  • Serverless & lifecycle policies — pay-per-use compute, tiered storage
  • Cost Explorer & Budgets — visualize spending, set alerts
Cost optimization is an ongoing process. Monitoring data from CloudWatch directly informs decisions. Right-sizing is often the biggest quick win — many organizations over-provision. Savings Plans are generally recommended over Reserved Instances for flexibility. First step: Enable Cost Explorer and review spending by service.

⚡ Serverless Architecture

Rethinking the employee directory application

Transition to serverless. We have been building and optimizing an EC2-based architecture throughout the course. Now we explore how serverless can simplify or replace parts of that architecture.

Serverless on AWS

What Does "Serverless" Mean?

No infrastructure to manage. Automatic scaling. Pay-per-use pricing. Built-in high availability.

⚙️ Compute

  • AWS Lambda
  • AWS Fargate

🗂️ Storage & Data

  • Amazon S3
  • Amazon DynamoDB
  • Aurora Serverless

🔗 Integration

  • API Gateway
  • EventBridge
  • Step Functions
Clarify that "serverless" does not mean "no servers" — it means you do not manage servers. AWS handles provisioning, patching, scaling, and availability. DynamoDB and S3 are already serverless services that students have been using. The new pieces are Lambda for compute and API Gateway for the API layer.

⚡ AWS Lambda

Run code without managing servers. Pay only for compute time consumed.

How It Works

  • Upload code or container image
  • Configure trigger (API GW, S3, etc.)
  • Runs in response to events
  • Scales from 0 to thousands concurrently

Key Specs

  • Runtimes: Python, Node.js, Java, .NET, Go
  • Memory: 128 MB – 10,240 MB
  • Timeout: up to 15 minutes
  • Pricing: $0.20/1M requests + duration
Lambda is the compute engine for serverless architectures. CPU is allocated proportionally to memory — more memory means more CPU. The 15-minute timeout makes Lambda suitable for request-response workloads but not for long-running processes. Lambda functions should be stateless; any state goes in DynamoDB, S3, or ElastiCache.

🔗 Amazon API Gateway

Fully Managed API Front Door

Create, publish, and secure APIs at any scale. Handles throttling, auth, and monitoring.

  • REST APIs — full-featured with caching and usage plans
  • HTTP APIs — lower cost, lower latency (ideal for Lambda)
  • WebSocket APIs — real-time two-way communication
  • Auth: IAM, Cognito, Lambda authorizers
API Gateway is the "front door" for serverless APIs. It handles rate limiting, authentication, request validation, and response transformation. HTTP APIs are up to 71% cheaper and 60% lower latency than REST APIs for simple proxy use cases. In the serverless architecture, API Gateway replaces the ALB as the entry point, routing HTTP requests to Lambda functions instead of EC2 instances. Stages allow deploying to dev, staging, and production with different configurations.

🌐 S3 + CloudFront for Static Hosting

S3 Static Website

  • Enable static hosting on a bucket
  • Configure index/error documents
  • 11 nines durability, auto-scales

Add CloudFront CDN

  • Global edge locations, low latency
  • HTTPS with custom domains
  • Origin Access Control for security
Pattern: S3 + CloudFront (frontend) + API Gateway + Lambda (backend) + DynamoDB (data) = no servers.
This pattern separates the frontend from the backend entirely. The static frontend (single-page application) makes API calls to API Gateway. CloudFront adds HTTPS support, custom domains, and global distribution with caching. The combination is extremely cost-effective for most web application traffic patterns.

🔄 AWS Step Functions

Visual workflow service for orchestrating serverless applications.

Key Features

  • Define workflows as state machines
  • Built-in retry and error handling
  • Visual execution monitoring
  • 200+ AWS service integrations

Two Workflow Types

  • Standard: up to 1 year, exactly-once
  • Express: up to 5 min, high volume

Use cases: data pipelines, order fulfillment, ETL orchestration

Step Functions solves the orchestration problem in serverless architectures. Without it, you end up with complex Lambda-to-Lambda chains that are hard to debug and monitor. Step Functions provides a visual designer, built-in retries, and parallel execution. For the employee directory, a Step Function could orchestrate onboarding: create record, resize photo, send welcome email — all as discrete Lambda functions.

🏗️ Employee Directory: EC2 vs Serverless

💻 EC2-Based

  • EC2 + ALB + Auto Scaling
  • Full runtime control
  • Long-running process support
  • Server management overhead

⚡ Serverless

  • Lambda + API GW + S3/CloudFront
  • Zero server management
  • Pay only when code runs
  • Cold starts, 15-min limit
Both use: DynamoDB (data) + S3 (images) + IAM (access) + CloudWatch (monitoring). The compute and API layers differ.
DynamoDB and S3 remain the same — they are already serverless. The big changes are replacing EC2 + ALB + ASG with Lambda + API Gateway, and decoupling the frontend to S3/CloudFront. Neither is universally better. EC2 is right for long-running workloads or teams with strong server management. Serverless shines for event-driven, variable traffic, and teams that want to focus on business logic. Many production architectures are hybrid. For a typical internal app with variable traffic, serverless could reduce costs 60-80% while eliminating maintenance.
