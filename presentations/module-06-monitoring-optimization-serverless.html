🔎 Monitoring

Visibility into your AWS environment

Monitoring is foundational — you cannot optimize what you cannot measure. AWS provides CloudWatch for metrics and alarms, and CloudTrail for API auditing.

🔎 Why Monitoring Matters

  • Detect issues before users do
  • Right-size resources and reduce costs
  • Track performance trends over time
  • Meet compliance and audit requirements
Without monitoring, you're flying blind. Monitoring answers: Is my application healthy? Am I paying too much? Are my resources right-sized? Who changed what and when? Proactive monitoring with alarms catches problems before they become outages. Historical data helps with capacity planning and cost optimization. Compliance often requires audit trails of all changes to your environment.

📈 Amazon CloudWatch

Monitoring and observability for AWS resources and applications.

  • Collects metrics from all AWS services
  • Custom metrics from your applications
  • Dashboards for real-time visualization
  • Logs aggregation and search
CloudWatch is the central monitoring service in AWS. It automatically collects metrics from EC2 (CPU, network, disk), RDS (connections, read/write latency), ELB (request counts, error rates), and many more services. You can publish custom metrics from your applications (e.g., business KPIs, queue depths). CloudWatch Logs lets you centralize log data from EC2 instances, Lambda functions, and other sources. Dashboards provide at-a-glance views of your most important metrics. Basic monitoring (5-minute intervals) is free; detailed monitoring (1-minute) has a small cost.

🔔 CloudWatch Alarms

  • Trigger on metric thresholds
  • Actions: SNS notification, Auto Scaling, EC2 action
  • States: OK, ALARM, INSUFFICIENT_DATA
  • Example: CPU > 80% for 5 minutes
Key pattern: CloudWatch Alarm → SNS Topic → email / Lambda / Auto Scaling
Alarms watch a single metric and trigger actions when a threshold is breached. The most common pattern: alarm triggers an SNS topic, which sends an email notification or invokes a Lambda function. Alarms are also the trigger mechanism for Auto Scaling policies — e.g., "add 2 instances when average CPU exceeds 70% for 3 consecutive 5-minute periods." Billing alarms are also CloudWatch alarms that watch your estimated charges. Always set up billing alarms in a new account.

👁️ AWS CloudTrail

API audit trail — records every API call in your account.

  • Who did what, when, from where
  • Enabled by default (90-day history)
  • Send to S3 for long-term retention
  • Essential for security and compliance
CloudTrail records every API call made in your AWS account — Console actions, CLI commands, SDK calls, and service-to-service calls. The default event history shows 90 days of management events for free. For long-term retention and analysis, create a trail that delivers logs to an S3 bucket. CloudTrail answers: Who deleted that EC2 instance? When was the security group modified? Who created that IAM user? It's essential for security investigations, compliance audits, and operational troubleshooting. CloudTrail is about WHO did WHAT; CloudWatch is about HOW resources are PERFORMING.

⚖️ Load Balancing & Auto Scaling

High availability and elasticity

Now let's use monitoring data to build highly available, auto-scaling architectures. ELB distributes traffic, Auto Scaling adjusts capacity, and together they create resilient systems.

⚖️ Elastic Load Balancing

Distribute traffic across multiple targets for availability and performance.

  • Automatically scales with traffic
  • Health checks remove unhealthy targets
  • Distributes across multiple AZs
  • Integrates with Auto Scaling groups
ELB is a managed load balancer that distributes incoming traffic across multiple targets (EC2 instances, containers, Lambda functions, IPs). It performs health checks and only routes to healthy targets. ELB is itself highly available — deployed across multiple AZs automatically. It integrates directly with Auto Scaling groups, so new instances are automatically registered and unhealthy ones are deregistered.

🔧 ALB vs NLB

Application LB (ALB)

  • Layer 7 (HTTP/HTTPS)
  • Path and host-based routing
  • Web apps and microservices

Network LB (NLB)

  • Layer 4 (TCP/UDP)
  • Ultra-low latency
  • Gaming, IoT, high throughput
ALB operates at Layer 7 and understands HTTP. It can route based on URL path (/api to one target group, /images to another) or hostname (api.example.com vs www.example.com). ALB is the default choice for web applications. NLB operates at Layer 4 and handles millions of requests per second with ultra-low latency. Use NLB for non-HTTP protocols, extreme performance needs, or when you need a static IP. There's also a Gateway Load Balancer (GWLB) for third-party network appliances, but that's an advanced use case.

📈 EC2 Auto Scaling

Automatically adjust the number of EC2 instances based on demand.

  • Set minimum, desired, and maximum capacity
  • Scale out on high demand, scale in when idle
  • Uses launch templates for instance config
  • Spans multiple AZs for high availability
Auto Scaling groups (ASGs) maintain a fleet of EC2 instances. You define: minimum (never go below this), desired (target count), and maximum (never exceed this). Scaling policies determine when to add or remove instances — typically based on CloudWatch alarms (e.g., average CPU > 70%). Launch templates define the AMI, instance type, security groups, and user data for new instances. ASGs distribute instances across AZs and automatically replace unhealthy instances detected by ELB health checks.

🔄 Auto Scaling + ELB

The HA pattern: ELB distributes traffic → Auto Scaling adjusts capacity → CloudWatch triggers scaling
  • ELB health checks drive instance replacement
  • New instances auto-register with ELB
  • Multi-AZ for fault tolerance
  • Pay only for what you use
This is the standard high-availability pattern on AWS. The ELB sits in front of an Auto Scaling group that spans multiple AZs. CloudWatch alarms trigger scaling policies to add or remove instances. When demand rises, Auto Scaling launches new instances which automatically register with the ELB. When demand drops, instances are terminated to save cost. If an instance fails health checks, ELB stops sending traffic and Auto Scaling replaces it. This pattern gives you fault tolerance, elasticity, and cost optimization simultaneously.

⚡ Serverless Architecture

No servers to manage

Serverless takes the concept further — AWS manages not just the infrastructure but the entire runtime environment. You focus purely on code and configuration.

⚡ Serverless on AWS

No provisioning, no patching, no scaling decisions. Pay per use.

  • Compute: Lambda, Fargate
  • Storage: S3
  • Database: DynamoDB, Aurora Serverless
  • API: API Gateway
Serverless doesn't mean there are no servers — it means you don't manage them. AWS handles provisioning, scaling, patching, and high availability. You've already seen several serverless services in this course: S3 for storage, DynamoDB for database, Lambda for compute. API Gateway provides managed REST and WebSocket APIs. The serverless model shifts your focus entirely to business logic and data.

🔗 API Gateway + Lambda

Pattern: Client → API Gateway → Lambda → DynamoDB
  • API Gateway handles HTTP routing and auth
  • Lambda runs business logic per request
  • DynamoDB stores and retrieves data
  • All fully managed, scales to zero
This is the canonical serverless architecture. API Gateway receives HTTP requests, validates them, handles authentication, and routes to Lambda functions. Lambda executes your business logic and interacts with DynamoDB (or S3, or other services). When there's no traffic, you pay nothing — all three services scale to zero. When traffic spikes, they auto-scale without any intervention. This pattern eliminates EC2 instances, load balancers, and auto scaling groups entirely.

🔨 Serverless vs EC2-Based

EC2 + ELB + ASG

  • Full control over environment
  • Long-running processes
  • Predictable, sustained load

API GW + Lambda + DynamoDB

  • Zero server management
  • Pay-per-request pricing
  • Variable / spiky traffic
Both architectures can run the same application. The EC2 approach gives more control and is cost-effective for sustained high traffic. The serverless approach minimizes operations and is cost-effective for variable or low traffic. Many organizations start with EC2 and migrate to serverless as they mature. Others build new projects serverless from the start. The right choice depends on team skills, traffic patterns, and how much infrastructure management you want to take on.

📋 Module 6 Summary

  • CloudWatch monitors; CloudTrail audits
  • ELB + Auto Scaling = high availability
  • Serverless eliminates server management
  • Choose architecture based on workload needs
Recap: CloudWatch provides metrics, alarms, logs, and dashboards for monitoring. CloudTrail records every API call for auditing. ELB distributes traffic across healthy targets. Auto Scaling adjusts capacity based on demand. Together, they create highly available, elastic architectures. Serverless (API Gateway + Lambda + DynamoDB) offers an alternative with zero server management. The right architecture depends on your control needs, traffic patterns, and operational preferences.
