📚 Databases in AWS

Managed services for every data need

AWS offers managed database services that handle provisioning, patching, backups, and high availability so you can focus on your application and data.

🛠️ Managed vs Unmanaged

Unmanaged (DB on EC2)

  • You manage everything
  • OS patches, backups, HA
  • Full control, full burden

Managed (RDS, DynamoDB)

  • AWS handles operations
  • Auto backups, patching, HA
  • Focus on data and queries
You can always install a database on EC2 and manage it yourself — this gives full control but means you handle patching, backups, replication, failover, and scaling. Managed services like RDS and DynamoDB offload this operational burden. AWS handles provisioning, patching, automated backups, and high availability. For most workloads, managed is the right choice — use unmanaged only when you need a database engine or configuration that isn't supported by a managed service.

🗂 Amazon RDS

Relational Database Service — managed SQL databases in the cloud.

  • MySQL, PostgreSQL, MariaDB, Oracle, SQL Server
  • Automated backups (up to 35 days)
  • Push-button Multi-AZ failover
  • Read replicas for scaling reads
RDS supports six popular relational database engines. It handles provisioning, patching, backups, and recovery. Multi-AZ deployments maintain a synchronous standby in another AZ for automatic failover — typically completes in 60-120 seconds. Read replicas create asynchronous copies that can handle read queries, reducing load on the primary. RDS runs inside your VPC on dedicated instances that you size based on workload.

⚡ RDS Features & Aurora

  • Multi-AZ: automatic failover to standby
  • Read Replicas: scale read-heavy workloads
  • Encryption: at rest and in transit

Amazon Aurora

AWS-built, MySQL/PostgreSQL compatible. 5x MySQL throughput. Storage auto-scales to 128 TB. 6-way replication across 3 AZs.

Multi-AZ is for high availability — the standby takes over if the primary fails. Read replicas are for performance — offload read traffic from the primary. They can also be promoted to standalone instances for disaster recovery. Aurora is AWS's cloud-native relational database. It's compatible with MySQL and PostgreSQL but redesigned for the cloud with a distributed storage layer that replicates 6 copies across 3 AZs. Aurora Serverless v2 can auto-scale compute capacity based on demand, which is useful for variable workloads.

🎯 Purpose-Built Databases

AWS philosophy: use the right database for each workload.

  • DynamoDB: key-value / document (NoSQL)
  • ElastiCache: in-memory caching (Redis, Memcached)
  • Neptune: graph relationships
  • Redshift: data warehousing / analytics
AWS offers 15+ purpose-built database services. The idea is that no single database is optimal for every use case. Relational databases are great for structured data with complex queries. DynamoDB excels at key-value lookups at any scale. ElastiCache provides microsecond latency for caching. Neptune handles graph queries for social networks and fraud detection. Redshift is a columnar data warehouse for analytics. DocumentDB is MongoDB-compatible. Timestream is for time-series data. Choose based on data model, access patterns, and scale requirements.

⚡ Amazon DynamoDB

Fully managed NoSQL — key-value and document database at any scale.

  • Single-digit millisecond latency
  • Serverless: no instances to manage
  • Automatic scaling and replication
  • Built-in encryption and backups
DynamoDB is AWS's flagship NoSQL database. It's fully serverless — you create a table, define the key schema, and start writing data. No instances to provision, no OS to patch. It automatically replicates across 3 AZs within a Region. DynamoDB handles millions of requests per second with consistent single-digit millisecond latency. Use cases include gaming leaderboards, shopping carts, IoT data, session stores, and any workload that needs fast, scalable key-value access.

🔑 DynamoDB Key Concepts

  • Table: collection of items (like rows)
  • Partition Key: unique identifier for each item
  • Sort Key: optional, enables range queries
  • Capacity: on-demand or provisioned mode
DynamoDB tables store items, each identified by a primary key. The primary key is either a simple partition key or a composite key (partition + sort). The partition key determines which physical partition stores the item — good key design distributes data evenly. On-demand mode charges per request (great for unpredictable traffic). Provisioned mode lets you set read/write capacity units with auto-scaling. Items can have different attributes — DynamoDB is schema-flexible. Maximum item size is 400 KB.

🎯 Choosing a Database

Use RDS/Aurora when...

  • Structured data with relationships
  • Complex SQL queries needed
  • ACID transactions required

Use DynamoDB when...

  • Key-value access patterns
  • Massive scale, low latency
  • Schema flexibility needed
The choice between relational and NoSQL depends on your data model and access patterns. If you need joins, transactions across tables, and complex queries — use RDS or Aurora. If you need fast lookups by key, massive scale, and flexible schemas — use DynamoDB. Many architectures use both: RDS for transactional data and DynamoDB for session state or caching. Don't force a relational model onto data that's naturally key-value, and vice versa.

📋 Module 5 Summary

  • Managed databases reduce operational burden
  • RDS/Aurora for relational SQL workloads
  • DynamoDB for serverless NoSQL at scale
  • Purpose-built: right database for each job
Recap: Managed databases let you focus on data instead of infrastructure. RDS supports six SQL engines with automated backups, Multi-AZ failover, and read replicas. Aurora is AWS's high-performance cloud-native option. DynamoDB is fully serverless NoSQL with single-digit millisecond latency at any scale. AWS's purpose-built philosophy means choosing the right database for each workload rather than forcing everything into one engine.
