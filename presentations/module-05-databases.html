📚 Explore Databases in AWS

History, types, and the managed advantage

We begin by looking at database fundamentals and why managed database services are so valuable in the cloud.

Database History and Types

📊 Relational (SQL)

  • Structured data in tables with rows and columns
  • Fixed schema, ACID transactions
  • SQL for queries and manipulation
  • Examples: MySQL, PostgreSQL, Oracle

🔬 Non-Relational (NoSQL)

  • Flexible schemas, varied data models
  • Key-value, document, graph, wide-column
  • Optimized for specific access patterns
  • Examples: DynamoDB, MongoDB, Neo4j
Relational databases have been the backbone of enterprise IT for decades. NoSQL databases emerged to address needs around flexible schemas, horizontal scaling, and specific data model requirements.

Managed vs. Unmanaged Databases

⚙️ Self-Managed on EC2

  • You install and configure the database engine
  • You manage patching, backups, replication
  • Full control over OS and engine settings
  • Higher operational burden

☁️ AWS Managed Service

  • AWS handles provisioning, patching, backups
  • Built-in high availability and failover
  • Monitoring integrated with CloudWatch
  • Focus on application, not infrastructure
Running a database on EC2 gives maximum control but means you own every operational task. Managed services like RDS shift that responsibility to AWS, letting you focus on your application and data.

Benefits of Managed Database Services

Why let AWS manage your database?

  • Reduced operational overhead — no patching, no backup scripts
  • High availability — Multi-AZ, automatic failover built in
  • Scalability — scale compute and storage independently
  • Security — encryption at rest and in transit, VPC isolation, IAM integration
  • Cost optimization — pay-as-you-go, Reserved Instances, Serverless options
Managed services significantly reduce the undifferentiated heavy lifting associated with database administration. This lets teams iterate faster and spend time on features rather than maintenance.

🛠️ Amazon Relational Database Service

Managed relational databases in the cloud

Now we dive into Amazon RDS, the managed relational database service that supports six popular database engines.

What is Amazon RDS?

Amazon RDS is a managed service that makes it straightforward to set up, operate, and scale a relational database in the cloud. AWS handles routine database tasks such as provisioning, patching, backups, recovery, and failure detection.

Supported Engines

MySQL

PostgreSQL

MariaDB

Oracle

SQL Server

Amazon Aurora

RDS supports six engines. You choose the engine, instance type, and storage, and AWS handles the rest. Aurora is the AWS-native engine with enhanced performance and availability.

RDS Key Features

  • Automated backups — point-in-time recovery within retention window (up to 35 days)
  • DB snapshots — user-initiated, stored in S3, retained until deleted
  • Multi-AZ deployments — synchronous standby replica in a different AZ
  • Read replicas — asynchronous replicas for read-heavy workloads
  • Encryption — at rest (KMS) and in transit (SSL/TLS)
  • Monitoring — CloudWatch metrics, Enhanced Monitoring, Performance Insights
  • Automatic minor version upgrades during maintenance windows
These features are available across all RDS engines. Automated backups and snapshots together provide robust data protection. Multi-AZ and read replicas address availability and scaling.

Multi-AZ Deployments

How It Works

  • Primary DB instance in one AZ
  • Standby replica in a different AZ
  • Synchronous replication
  • Automatic failover (typically 60–120 seconds)
Key Points:
  • Standby is not used for read traffic
  • DNS endpoint stays the same after failover
  • Failover triggers: AZ outage, instance failure, manual reboot with failover
Multi-AZ = High Availability. It does not improve read performance. Use read replicas for read scaling.
Multi-AZ is about availability, not performance. The standby is invisible to your application. Failover is automatic and handled via a DNS CNAME flip. The application connects to the same endpoint before and after failover.

Read Replicas

Characteristics

  • Asynchronous replication from primary
  • Up to 15 read replicas (Aurora) or 5 (other engines)
  • Can be in same AZ, cross-AZ, or cross-Region
  • Each replica has its own DNS endpoint

Use Cases

  • Offload read-heavy queries (reports, analytics)
  • Reduce latency for geographically distributed users
  • Can be promoted to a standalone DB instance
Read Replicas = Read Scaling. Application must direct read traffic to replica endpoints explicitly.
Read replicas use asynchronous replication, which means there can be slight replication lag. They are ideal for read-heavy workloads. Cross-Region replicas can also serve as a disaster recovery strategy.

Multi-AZ vs. Read Replicas

Feature Multi-AZ Read Replica
Purpose High availability Read scaling
Replication Synchronous Asynchronous
Serves traffic? Standby does not Yes, read queries
Failover Automatic Manual promotion
Cross-Region No (same Region only) Yes
Pro tip: You can combine Multi-AZ and read replicas on the same RDS instance to get both high availability and read scaling.
This is a common exam and interview topic. Multi-AZ and read replicas solve different problems and can be used together. Multi-AZ provides automatic failover; read replicas provide horizontal read scaling.

⭐ Amazon Aurora

Amazon Aurora is a MySQL- and PostgreSQL-compatible relational database built for the cloud. It combines the performance and availability of commercial databases with the simplicity and cost-effectiveness of open-source databases.

Performance

  • Up to 5x throughput of MySQL
  • Up to 3x throughput of PostgreSQL
  • Distributed, SSD-backed storage layer

Storage & Availability

  • Auto-scales storage from 10 GB up to 128 TB
  • 6 copies of data across 3 AZs
  • Up to 15 low-latency read replicas
Aurora decouples compute from storage. The storage layer is a distributed, fault-tolerant, self-healing system that replicates data six ways across three AZs. This architecture delivers both high performance and durability.

Aurora Serverless v2

On-Demand, Auto-Scaling Database Capacity

  • Instantly scales compute capacity up or down based on application demand
  • Scales in fine-grained increments (as small as 0.5 ACU)
  • Supports Multi-AZ for high availability
  • Compatible with Aurora features: Global Database, read replicas, backtrack
Ideal for: variable or unpredictable workloads, dev/test environments, multi-tenant SaaS applications, and infrequently used applications.
Aurora Serverless v2 is a significant improvement over v1. It scales without disruption and supports the full Aurora feature set. You set minimum and maximum ACU (Aurora Capacity Units) and the service handles the rest.

RDS Pricing Considerations

Cost Components

  • Instance hours — based on instance class
  • Storage — provisioned (gp3, io1) or Aurora auto-scaling
  • I/O requests — Aurora charges per million I/Os
  • Backup storage — beyond free allocation
  • Data transfer — cross-AZ and outbound

Cost Optimization

  • Reserved Instances (1- or 3-year terms)
  • Aurora Serverless for variable workloads
  • Right-size instance types
  • Use gp3 storage for cost-efficient performance
RDS pricing is multi-dimensional. The largest cost drivers are typically instance hours and storage. Reserved Instances can save up to 60% compared to On-Demand pricing for steady-state workloads.

🎯 Purpose-Built Databases

The right tool for each workload

AWS offers a broad portfolio of purpose-built databases. Instead of forcing every workload into one engine, you choose the best fit for each data model and access pattern.

One-Size-Fits-All vs. Purpose-Built

⛔ Traditional Approach

  • Single relational database for everything
  • Complex schemas to accommodate all use cases
  • Scaling limitations under diverse workloads
  • Performance trade-offs inevitable

✅ Purpose-Built Approach

  • Match the database to the data model
  • Optimize for specific access patterns
  • Scale each workload independently
  • Better performance, lower cost per workload
Modern architectures often use multiple purpose-built databases in a single application. For example, a relational database for transactions, a cache for sessions, and a document database for a product catalog.

AWS Database Portfolio

Service Type Use Case
Amazon RDS / Aurora Relational Traditional apps, ERP, CRM, e-commerce
DynamoDB Key-value / Document High-scale apps, gaming, IoT
DocumentDB Document (MongoDB compatible) Content management, catalogs, profiles
ElastiCache In-memory (Redis, Memcached) Caching, session stores, leaderboards
Neptune Graph Social networks, fraud detection, knowledge graphs
QLDB Ledger Immutable audit logs, supply chain
Keyspaces Wide-column (Cassandra compatible) Equipment maintenance, fleet management
Timestream Time series IoT telemetry, DevOps metrics
This is not an exhaustive list, but it covers the major purpose-built databases in the AWS portfolio. Each is optimized for a specific data model and access pattern.

Purpose-Built Databases — Highlights

📄 DocumentDB

MongoDB-compatible document database. Supports JSON workloads with the scalability and durability of a managed service.

🔗 Neptune

Fully managed graph database. Supports Property Graph (Gremlin) and RDF (SPARQL) models for highly connected datasets.

⚡ ElastiCache

In-memory data store supporting Redis and Memcached. Microsecond latency for caching, session management, and real-time analytics.

📑 QLDB

Ledger database with an immutable, transparent, and cryptographically verifiable transaction log.

📋 Keyspaces

Managed Apache Cassandra-compatible service. Serverless, pay-per-use, with single-digit-ms latency.

⏰ Timestream

Purpose-built for time series data. Automatic tiered storage (memory → magnetic) and built-in analytics functions.

Each of these services is fully managed. The key takeaway is to match the service to the workload rather than forcing all data into a single relational database.

⚡ Introduction to Amazon DynamoDB

Fully managed NoSQL at any scale

DynamoDB is one of the most important services in the AWS portfolio. It provides single-digit millisecond performance at virtually any scale.

What is Amazon DynamoDB?

Amazon DynamoDB is a fully managed, serverless NoSQL database service that provides single-digit millisecond performance at any scale. It supports both key-value and document data models.
  • No servers to provision, patch, or manage
  • Automatic scaling of throughput and storage
  • Built-in security, backup, and recovery
  • Multi-Region, multi-active replication with global tables
  • Event-driven programming with DynamoDB Streams
DynamoDB is truly serverless. You create a table, define your keys, and start writing data. There are no instances to size and no storage to provision. It handles replication, patching, and scaling transparently.

DynamoDB Core Concepts

Tables, Items, and Attributes

  • Table — collection of items (similar to a SQL table)
  • Item — a single record (similar to a row)
  • Attribute — a data element (similar to a column)
  • Items can have different attributes (flexible schema)

Data Types

  • Scalar — String, Number, Binary, Boolean, Null
  • Document — List, Map
  • Set — String Set, Number Set, Binary Set
DynamoDB is schemaless beyond the primary key. Each item can have its own set of attributes. This flexibility is a core advantage for rapidly evolving applications.

DynamoDB Primary Keys

🔑 Partition Key (Simple)

  • Single attribute uniquely identifies each item
  • DynamoDB hashes the partition key to determine storage partition
  • Example: UserID

🔑 Composite Key (Partition + Sort)

  • Two attributes together form the primary key
  • Partition key determines partition; sort key orders items within it
  • Example: UserID (partition) + Timestamp (sort)
Key design is critical. A well-chosen partition key distributes data evenly across partitions, avoiding hot partitions and ensuring consistent performance.
Primary key design is the single most important decision in DynamoDB. A poor partition key leads to hot partitions and throttled requests. Choose a high-cardinality attribute that distributes access evenly.

Read/Write Capacity Modes

📋 Provisioned Mode

  • You specify Read Capacity Units (RCUs) and Write Capacity Units (WCUs)
  • 1 RCU = 1 strongly consistent read/sec (up to 4 KB)
  • 1 WCU = 1 write/sec (up to 1 KB)
  • Auto Scaling adjusts within set min/max
  • Best for predictable traffic patterns

⚡ On-Demand Mode

  • No capacity planning required
  • Pay per request (reads and writes)
  • Instantly accommodates traffic spikes
  • Best for unpredictable or new workloads
  • Can switch between modes once per 24 hours
Provisioned mode is more cost-effective for steady workloads. On-Demand is simpler and better for spiky or unknown traffic patterns. You can switch between modes, but only once every 24 hours.

DynamoDB Features

🔒 Security

  • Encryption at rest (AWS-owned, customer-managed KMS)
  • Fine-grained access control via IAM
  • VPC endpoints

🌐 Global Tables

  • Multi-Region, multi-active replication
  • 99.999% availability SLA
  • Local read/write in each Region

🔄 Streams

  • Ordered flow of item-level changes
  • Trigger Lambda functions
  • Cross-Region replication, analytics

⏱ TTL (Time to Live)

Automatically delete expired items at no extra cost. Ideal for session data, temporary records, and logs.

🚀 DAX (DynamoDB Accelerator)

In-memory cache for DynamoDB. Reduces read latency from milliseconds to microseconds. API-compatible — no application code changes.

These features make DynamoDB enterprise-ready. Global tables enable multi-region active-active architectures. Streams enable event-driven architectures. DAX provides microsecond read latency without code changes.

DynamoDB Use Cases

  • 🎮 Gaming — player state, leaderboards, session history
  • 📡 IoT — device telemetry, time series data ingestion
  • 📱 Mobile backends — user profiles, app preferences
  • 🔒 Session management — web session data with TTL
  • 🛒 E-commerce — shopping carts, product catalogs
  • 📈 Ad tech — real-time bidding, click-stream analytics
  • 💰 Financial — transaction processing, fraud detection
  • 👥 Social media — news feeds, activity timelines
Scale fact: DynamoDB handles more than 10 trillion requests per day and can support peaks of more than 20 million requests per second.
DynamoDB is used by some of the largest applications in the world, including Amazon.com itself. Its single-digit millisecond performance and limitless scaling make it ideal for high-throughput, low-latency workloads.

DynamoDB Pricing Model

Provisioned Mode

  • Pay per provisioned RCU and WCU per hour
  • Reserved Capacity available (1- or 3-year)
  • Free tier: 25 RCU + 25 WCU + 25 GB storage

On-Demand Mode

  • Pay per read and write request unit
  • No minimum capacity or commitment
  • Typically 5–7x more expensive per request than provisioned
Additional costs: storage ($0.25/GB-month), streams, global table replicated writes, backups, and data transfer.
Start with on-demand for new workloads to understand traffic patterns, then switch to provisioned with auto-scaling once patterns stabilize. This is a common cost-optimization strategy.

🤔 Choose the Right AWS Database

Decision criteria and comparisons

With so many database options, how do you choose? We will look at the key decision criteria and compare the most commonly used services.

Choosing the Right Database

Key Decision Criteria

  • Data model — relational, key-value, document, graph, time series?
  • Access patterns — complex queries (SQL) vs. simple key lookups?
  • Scalability — vertical (RDS) vs. horizontal (DynamoDB)?
  • Consistency — strong ACID transactions vs. eventual consistency?
  • Latency requirements — milliseconds vs. microseconds?
  • Operational model — serverless vs. instance-based?
No single database is best for every workload. Walk through these criteria for each component of your application and the right choice usually becomes clear.

Service Comparison

Criteria RDS / Aurora DynamoDB ElastiCache
Data model Relational (SQL) Key-value / Document Key-value (in-memory)
Scaling Vertical + read replicas Horizontal (automatic) Cluster mode sharding
Latency Low milliseconds Single-digit ms (µs with DAX) Sub-millisecond
Transactions Full ACID ACID (TransactWriteItems) Redis transactions
Serverless Aurora Serverless v2 Yes (fully) ElastiCache Serverless
Best for Complex queries, joins High throughput, simple queries Caching, real-time
This table helps frame the most common decision: RDS for complex relational queries, DynamoDB for high-throughput simple access patterns, and ElastiCache for microsecond caching. Many applications use a combination of these.

When to Use Relational vs. NoSQL

📊 Choose Relational (RDS/Aurora)

  • Complex queries with JOINs
  • Strict schema enforcement needed
  • Multi-table ACID transactions
  • Existing SQL-based applications
  • Reporting and business intelligence

⚡ Choose NoSQL (DynamoDB)

  • Simple key-based access patterns
  • Massive read/write throughput
  • Flexible or evolving schema
  • Horizontal scale with low latency
  • Serverless architecture fit
Remember: It is not relational or NoSQL. Modern architectures commonly use both — each for the workload it handles best.
The answer is rarely either/or. A single application might use Aurora for order management, DynamoDB for session state, and ElastiCache for query caching. Design for the access pattern, not the engine you know best.
