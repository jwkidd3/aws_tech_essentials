📚 Explore Databases in AWS

History, types, and the managed advantage

We begin by looking at database fundamentals and why managed database services are so valuable in the cloud.

Database History and Types

📊 Relational (SQL)

  • Structured tables with fixed schema
  • ACID transactions
  • SQL for queries and manipulation
  • MySQL, PostgreSQL, Oracle

🔬 Non-Relational (NoSQL)

  • Flexible schemas, varied data models
  • Key-value, document, graph, wide-column
  • Optimized for specific access patterns
  • DynamoDB, MongoDB, Neo4j
Relational databases have been the backbone of enterprise IT for decades. NoSQL databases emerged to address needs around flexible schemas, horizontal scaling, and specific data model requirements.

Managed vs. Self-Managed Databases

⚙️ Self-Managed on EC2

  • You install, patch, and back up
  • Full OS and engine control
  • Higher operational burden

☁️ AWS Managed Service

  • AWS handles patching and backups
  • Built-in HA and automatic failover
  • Encryption, VPC isolation, IAM
  • Scale compute and storage independently
Key benefit: Managed services let you focus on your application instead of database administration.
Running a database on EC2 gives maximum control but means you own every operational task. Managed services like RDS shift that responsibility to AWS. Benefits include reduced overhead, built-in HA, scalability, security with encryption at rest/in transit, and cost optimization through pay-as-you-go and Reserved Instances.

🛠️ Amazon Relational Database Service

Managed relational databases in the cloud

Now we dive into Amazon RDS, the managed relational database service that supports six popular database engines.

What is Amazon RDS?

Amazon RDS — managed service to set up, operate, and scale relational databases. AWS handles provisioning, patching, backups, and failure detection.

Supported Engines

MySQL

PostgreSQL

MariaDB

Oracle

SQL Server

Amazon Aurora

RDS supports six engines. You choose the engine, instance type, and storage, and AWS handles the rest. Aurora is the AWS-native engine with enhanced performance and availability.

RDS Key Features

  • Automated backups — point-in-time recovery up to 35 days
  • Multi-AZ — synchronous standby for automatic failover
  • Read replicas — asynchronous replicas for read scaling
  • Encryption — at rest (KMS) and in transit (TLS)
  • Monitoring — CloudWatch, Performance Insights
These features are available across all RDS engines. Multi-AZ provides high availability with a synchronous standby in a different AZ; failover is automatic via DNS CNAME flip (60-120 seconds). The standby does NOT serve read traffic. Read replicas use asynchronous replication and each has its own endpoint. Up to 15 replicas for Aurora, 5 for other engines. Can be cross-AZ or cross-Region. Replicas can be promoted to standalone instances.

Multi-AZ vs. Read Replicas

Feature Multi-AZ Read Replica
Purpose High availability Read scaling
Replication Synchronous Asynchronous
Serves traffic? Standby does not Yes, read queries
Failover Automatic Manual promotion
Cross-Region No Yes
Pro tip: Combine Multi-AZ and read replicas for both HA and read scaling.
This is a common exam and interview topic. Multi-AZ and read replicas solve different problems and can be used together. Multi-AZ provides automatic failover; read replicas provide horizontal read scaling.

⭐ Amazon Aurora

Aurora — MySQL- and PostgreSQL-compatible, built for the cloud with commercial-grade performance at open-source cost.

Performance

  • 5x MySQL throughput
  • 3x PostgreSQL throughput
  • Distributed, SSD-backed storage

Storage & Availability

  • Auto-scales 10 GB to 128 TB
  • 6 copies across 3 AZs
  • Up to 15 low-latency read replicas
Aurora Serverless v2: Auto-scales in 0.5 ACU increments for variable or unpredictable workloads.
Aurora decouples compute from storage. The storage layer is distributed, fault-tolerant, and self-healing across three AZs. Aurora Serverless v2 scales without disruption, supports the full Aurora feature set including Global Database and read replicas. You set min/max ACU and the service handles the rest. Ideal for dev/test, multi-tenant SaaS, and infrequently used applications.

RDS Pricing Considerations

Cost Components

  • Instance hours by instance class
  • Storage — gp3, io1, or Aurora auto
  • Backup storage beyond free tier
  • Data transfer — cross-AZ, outbound

Cost Optimization

  • Reserved Instances (1- or 3-year)
  • Aurora Serverless for variable loads
  • Right-size instance types
  • Use gp3 for cost-efficient performance
RDS pricing is multi-dimensional. The largest cost drivers are typically instance hours and storage. Reserved Instances can save up to 60% compared to On-Demand pricing for steady-state workloads. Aurora also charges per million I/Os.

🎯 Purpose-Built Databases

The right tool for each workload

AWS offers a broad portfolio of purpose-built databases. Instead of forcing every workload into one engine, you choose the best fit for each data model and access pattern.

AWS Purpose-Built Database Portfolio

Service Type Use Case
RDS / Aurora Relational ERP, CRM, e-commerce
DynamoDB Key-value / Document High-scale apps, gaming, IoT
ElastiCache In-memory (Redis, Memcached) Caching, session stores
Neptune Graph Social networks, fraud detection
Keyspaces Wide-column (Cassandra) Fleet management, maintenance
Timestream Time series IoT telemetry, DevOps metrics
Match the engine to the workload — optimize cost, performance, and scale independently.
Modern architectures often use multiple purpose-built databases. For example, a relational database for transactions, a cache for sessions, and a document database for a product catalog. Additional services include DocumentDB (MongoDB-compatible) and QLDB (immutable ledger). The key principle: instead of forcing all data into one relational engine, choose the best fit for each data model and access pattern.

⚡ Introduction to Amazon DynamoDB

Fully managed NoSQL at any scale

DynamoDB is one of the most important services in the AWS portfolio. It provides single-digit millisecond performance at virtually any scale.

What is Amazon DynamoDB?

Amazon DynamoDB — fully managed, serverless NoSQL database with single-digit millisecond performance at any scale. Supports key-value and document models.
  • No servers to provision or manage
  • Automatic throughput and storage scaling
  • Built-in security, backup, and recovery
  • Global tables for multi-Region replication
  • Event-driven programming via Streams
DynamoDB is truly serverless. You create a table, define your keys, and start writing data. There are no instances to size and no storage to provision. It handles replication, patching, and scaling transparently.

DynamoDB Core Concepts

Tables, Items, Attributes

  • Table — collection of items
  • Item — single record
  • Attribute — data element
  • Items can differ (flexible schema)

Data Types

  • Scalar — String, Number, Boolean
  • Document — List, Map
  • Set — String, Number, Binary
DynamoDB is schemaless beyond the primary key. Each item can have its own set of attributes. This flexibility is a core advantage for rapidly evolving applications. Scalar types also include Binary and Null.

DynamoDB Primary Keys

🔑 Partition Key (Simple)

  • Single attribute identifies each item
  • Hashed to determine storage partition
  • Example: UserID

🔑 Composite Key

  • Partition key + sort key together
  • Sort key orders items within partition
  • Example: UserID + Timestamp
Key design is critical. Choose high-cardinality partition keys to distribute data evenly and avoid hot partitions.
Primary key design is the single most important decision in DynamoDB. A poor partition key leads to hot partitions and throttled requests. Choose a high-cardinality attribute that distributes access evenly.

Read/Write Capacity Modes

📋 Provisioned Mode

  • Set RCUs and WCUs explicitly
  • 1 RCU = 1 read/sec (4 KB)
  • 1 WCU = 1 write/sec (1 KB)
  • Auto Scaling adjusts within limits

⚡ On-Demand Mode

  • No capacity planning needed
  • Pay per request
  • Instantly handles traffic spikes
  • Switch modes once per 24 hours
Provisioned mode is more cost-effective for steady workloads. On-Demand is simpler and better for spiky or unknown traffic patterns. Provisioned is best for predictable traffic; on-demand for new or unpredictable workloads.

DynamoDB Features

🔒 Security

  • Encryption at rest (KMS)
  • Fine-grained IAM access
  • VPC endpoints

🌐 Global Tables

  • Multi-Region, multi-active
  • 99.999% availability SLA
  • Local read/write per Region

🔄 Streams & More

  • Ordered item-level changes
  • Trigger Lambda functions
  • TTL auto-deletes expired items
DAX (DynamoDB Accelerator): In-memory cache — microsecond reads, API-compatible, no code changes.
These features make DynamoDB enterprise-ready. Global tables enable multi-region active-active architectures. Streams enable event-driven architectures. TTL is useful for automatically cleaning up session data and temporary records. DAX provides microsecond read latency without code changes.

DynamoDB Use Cases

  • 🎮 Gaming — leaderboards, player state
  • 📡 IoT — device telemetry ingestion
  • 📱 Mobile — user profiles, preferences
  • 🔒 Sessions — web session data with TTL
  • 🛒 E-commerce — carts, catalogs
  • 📈 Ad tech — real-time bidding
  • 💰 Financial — transaction processing
  • 👥 Social — feeds, activity timelines
DynamoDB is used by some of the largest applications in the world, including Amazon.com itself. Its single-digit millisecond performance and limitless scaling make it ideal for high-throughput, low-latency workloads. It handles more than 10 trillion requests per day.

DynamoDB Pricing Model

Provisioned Mode

  • Pay per RCU/WCU per hour
  • Reserved Capacity (1- or 3-year)
  • Free tier: 25 RCU + 25 WCU + 25 GB

On-Demand Mode

  • Pay per read/write request
  • No minimum commitment
  • 5–7x more per request vs. provisioned
Additional costs: storage ($0.25/GB-month), streams, global table writes, backups, data transfer.
Start with on-demand for new workloads to understand traffic patterns, then switch to provisioned with auto-scaling once patterns stabilize. This is a common cost-optimization strategy.

🤔 Choose the Right AWS Database

Decision criteria and comparisons

With so many database options, how do you choose? We will look at the key decision criteria and compare the most commonly used services.

Choosing the Right Database

📊 Choose Relational (RDS/Aurora)

  • Complex queries with JOINs
  • Strict schema enforcement
  • Multi-table ACID transactions
  • Reporting and BI workloads

⚡ Choose NoSQL (DynamoDB)

  • Simple key-based access patterns
  • Massive read/write throughput
  • Flexible or evolving schema
  • Serverless, horizontal scale
Consider: data model, access patterns, scalability needs, consistency, and latency requirements.
No single database is best for every workload. Walk through criteria for each component: data model (relational, key-value, graph, time series), access patterns (SQL joins vs key lookups), scalability (vertical vs horizontal), consistency (ACID vs eventual), latency, and operational model (serverless vs instance). Modern architectures commonly use both relational and NoSQL, each for the workload it handles best.

Service Comparison

Criteria RDS / Aurora DynamoDB ElastiCache
Data model Relational (SQL) Key-value / Document Key-value (in-memory)
Scaling Vertical + read replicas Horizontal (automatic) Cluster mode sharding
Latency Low milliseconds Single-digit ms (µs with DAX) Sub-millisecond
Transactions Full ACID ACID (TransactWriteItems) Redis transactions
Serverless Aurora Serverless v2 Yes (fully) ElastiCache Serverless
Best for Complex queries, joins High throughput, simple queries Caching, real-time
This table helps frame the most common decision: RDS for complex relational queries, DynamoDB for high-throughput simple access patterns, and ElastiCache for microsecond caching. Many applications use a combination of these.
