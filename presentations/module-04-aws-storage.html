🗃 AWS Storage Types

Block, File, and Object Storage

Before diving into specific services, let's establish the three fundamental storage paradigms and how they map to AWS offerings.

Three Storage Paradigms

🧱 Block Storage

  • Fixed-size blocks with unique addresses
  • Low latency, high IOPS
  • Use: databases, boot volumes

📁 File Storage

  • Hierarchical directory structure
  • Shared via NFS or SMB
  • Use: shared app data, CMS

📦 Object Storage

  • Flat namespace: data + metadata + key
  • Massive scalability
  • Use: backups, media, data lakes
Block storage is like a raw hard drive — the OS manages the filesystem. File storage provides a shared filesystem. Object storage treats each item as an immutable object with metadata. Each paradigm has distinct performance, cost, and access characteristics.

🖩 EC2 Instance Storage & Amazon EBS

Block Storage for EC2 Workloads

Let's start with the block storage options directly attached to or associated with EC2 instances.

EC2 Instance Store

⚡ Characteristics

  • Physically attached NVMe/SSD disks
  • Millions of IOPS possible
  • Included in instance price
  • Size depends on instance type

⚠️ Limitations

  • Ephemeral — data lost on stop/terminate
  • Cannot detach or reattach
  • No snapshot support
  • Not available on all instance types
Rule of thumb: Never store data on instance store that you cannot afford to lose. Use it for caches, scratch data, and temporary buffers. The i3 and i3en families offer up to 60 TB of NVMe instance storage. Always replicate critical data elsewhere.

Amazon Elastic Block Store (EBS)

💾 Persistent Block Storage for EC2

Network-attached block volumes within a single Availability Zone.

  • Replicated within its AZ for durability
  • Live resizing, type changes, IOPS adjustments
  • Point-in-time snapshots stored in S3
  • AES-256 encryption at rest and in transit
  • Billed per GB-month + IOPS (io1/io2)
EBS is the workhorse for persistent block storage. Think of it as a virtual hard drive. Unlike instance store, EBS data persists across stop/start and can be snapshotted. The tradeoff is slightly higher latency since it's network-attached.

gp3 is the recommended default. It provides a baseline of 3,000 IOPS and 125 MiB/s regardless of volume size. gp2 and io1 still exist but gp3 and io2 Block Express are the current-generation options. HDD volumes are optimized for large sequential I/O, not random reads. st1 is great for streaming workloads like Kafka or Hadoop HDFS. sc1 is the lowest cost option for rarely accessed data.

EBS Snapshots

📸 Point-in-Time Backups

Incremental backups stored in S3 — only changed blocks are saved after the first snapshot.

  • Cross-region copy for disaster recovery
  • Restore into new volumes in any AZ
  • Share with other accounts or make public
  • Fast Snapshot Restore eliminates first-read latency
Automate: Use Data Lifecycle Manager (DLM) for scheduled snapshots.
Snapshots are critical for backup and disaster recovery. Any single snapshot is self-sufficient — you can delete earlier snapshots without affecting later ones. FSR is useful for boot volumes or databases that need immediate full performance. Snapshots are billed per GB-month of data stored.

EBS Encryption & Multi-Attach

🔒 Encryption

  • AES-256 at rest and in transit
  • AWS KMS keys (managed or custom)
  • Encrypted snapshots automatic
  • Enable encryption by default per region

🔗 Multi-Attach

  • io1/io2 only
  • Up to 16 Nitro instances per volume
  • All instances in the same AZ
  • App must manage concurrent writes
Encryption is transparent to the application — no code changes needed. Multi-Attach enables shared-disk clusters but requires a cluster-aware filesystem (like GFS2 or OCFS2) to prevent data corruption from concurrent writes. Cost tip: Delete unattached EBS volumes and old snapshots to avoid unnecessary charges. Use Elastic Volumes to right-size without downtime.

📦 Amazon S3

Object Storage at Any Scale

Now we move to one of AWS's most important and widely used services — Amazon S3. It's not just storage; it's a foundational building block for data lakes, backups, content delivery, and more.

What is Amazon S3?

🌎 Object Storage, Built for the Cloud

Data stored as objects in buckets. Designed for 99.999999999% (11 nines) durability.

Core Concepts

  • Bucket: globally unique container
  • Object: file + metadata (up to 5 TB)
  • Key: unique ID within a bucket
  • Prefix: simulates folder structure

Key Properties

  • Unlimited storage capacity
  • Regional service
  • RESTful API access
  • Event notifications and triggers
S3 is one of the oldest AWS services (launched 2006) and is incredibly versatile. The flat namespace with keys means there are no real directories — the console simulates folders using the "/" delimiter in keys. Objects can range from 0 bytes to 5 TB; uploads over 5 GB must use multipart upload.

S3 Bucket Naming & Structure

Naming Rules

  • Globally unique across all accounts
  • 3–63 chars; lowercase, numbers, hyphens
  • Must start with letter or number
  • Cannot look like an IP address

Object Addressing

https://my-bucket.s3.amazonaws.com/photos/image.jpg

Bucket: my-bucket
Key:    photos/image.jpg
Prefix: photos/ acts as a virtual folder.
Bucket names form part of the DNS name, hence the strict rules. The global uniqueness requirement means you cannot create a bucket with a name someone else already uses in any region. Plan your naming convention early — include account ID or project prefix for uniqueness.

Standard-IA and One Zone-IA have a 30-day minimum charge and per-GB retrieval fee. Glacier classes have 90-day (Instant/Flexible) or 180-day (Deep Archive) minimums. Standard is ~$23/TB/mo in US East.

S3 Lifecycle Policies

Automate transitions between storage classes or schedule deletion.

🔄 Example Lifecycle Rule

{
  "Rules": [{
    "ID": "ArchiveOldLogs",
    "Filter": { "Prefix": "logs/" },
    "Transitions": [
      { "Days": 30, "StorageClass": "STANDARD_IA" },
      { "Days": 90, "StorageClass": "GLACIER" }
    ],
    "Expiration": { "Days": 2555 }
  }]
}
Fully automated — no Lambda needed. Transitions run at midnight UTC.
Lifecycle policies are one of the most effective ways to optimize S3 costs. You can apply rules to the entire bucket or filter by prefix and/or tags. Transitions only move in one direction (Standard -> IA -> Glacier). You can also expire (delete) objects or incomplete multipart uploads.

S3 Versioning

How It Works

  • Enabled at the bucket level
  • Each PUT creates a new Version ID
  • DELETE inserts a delete marker
  • Cannot disable — only suspend

Benefits

  • Protects against accidental deletes
  • Enables MFA Delete
  • Required for Cross-Region Replication
  • Combine with lifecycle to expire old versions
Cost impact: Every version is a separate object that incurs storage charges. Use lifecycle rules to expire noncurrent versions (e.g., keep only the last 3). MFA Delete requires multi-factor authentication to permanently delete a version or change versioning state.

S3 Access Control

📜 Bucket Policies

  • JSON resource policy on bucket
  • Cross-account access
  • Condition keys (IP, VPC, MFA)

👥 IAM Policies

  • Identity-based, centralized
  • Fine-grained per-action control
  • Recommended primary method

🚫 Block Public Access

  • Account and bucket-level guardrail
  • Overrides ACLs and policies
  • Enabled by default
ACLs are legacy. AWS recommends disabling ACLs (Bucket Owner Enforced) and using bucket policies + IAM policies for all access control. Block Public Access acts as a safety net. The final effective permission is the union of IAM policy and bucket policy, restricted by Block Public Access.

S3 Bucket Policy — Example

{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "AllowReadFromVPC",
    "Effect": "Allow",
    "Principal": "*",
    "Action": ["s3:GetObject"],
    "Resource": "arn:aws:s3:::my-app-data/*",
    "Condition": {
      "StringEquals": { "aws:SourceVpc": "vpc-0a1b2c3d4e5f" }
    }
  }]
}
This example restricts access to a specific VPC. Bucket policies support the full range of IAM condition keys. You can also add Deny statements to enforce encryption on uploads (e.g., deny PutObject unless SSE-KMS header is present).

S3 Encryption

MethodKey ManagementWhen to Use
SSE-S3 AWS manages keys Default; simplest option
SSE-KMS AWS KMS (CMK or managed) Audit trail, key rotation, compliance
SSE-C Customer provides key per request Full key control, HTTPS required
Client-side Customer encrypts before upload End-to-end encryption requirements
Since Jan 2023: All new S3 objects encrypted with SSE-S3 by default.
SSE-KMS adds CloudTrail logging of key usage and separation of duties between data and key administrators. SSE-C is for organizations that must retain full control of encryption keys. Client-side encryption is the most secure but requires application changes.

S3 Use Cases

💾 Backup & DR

  • Cross-region replication
  • EBS snapshot storage
  • Database backup target

🌐 Static Website Hosting

  • Host HTML/CSS/JS from S3
  • Pair with CloudFront CDN
  • Custom domain via Route 53

📊 Data Lake Foundation

  • Central raw data repository
  • Query with Athena or Redshift Spectrum
  • Decouple storage from compute

📶 Content Distribution

  • Media files, software updates
  • Pre-signed URLs for secure sharing
  • Transfer Acceleration for uploads
S3 is often called the "storage backbone of AWS." It underpins many other services — EBS snapshots are stored in S3 (though not visible in your bucket), CloudTrail logs go to S3, and many analytics services read directly from S3.

S3 Pricing Model

💰 Pay for What You Use

No minimum commitment. Three cost components:

Storage

  • Per GB-month
  • Varies by storage class

Requests

  • PUT/GET per 1,000 requests
  • LIST requests 5x more costly

Data Transfer

  • Data IN: free
  • S3 to CloudFront: free
Cost tip: Transferring data from S3 to CloudFront is free — use CloudFront to serve S3 content and save on data transfer. A common surprise is request pricing — millions of tiny GET requests can add up. For data transfer, the first 100 GB/month out to the internet is free (free tier). Cross-region replication incurs data transfer charges between regions. Lifecycle transitions also count as requests.

🧮 Choose the Right Storage Service

Decision Factors and Comparisons

Now that we've covered the details of EBS and S3, let's step back and talk about how to make the right choice for a given workload.

AWS Storage Comparison

FeatureInstance StoreEBSS3EFS
TypeBlockBlockObjectFile (NFS)
PersistenceEphemeralPersistentPersistentPersistent
ScopeInstance lifetimeSingle AZRegionalRegional (multi-AZ)
DurabilityNone99.999%11 nines11 nines
Best ForCaches, scratchDBs, boot volsBackups, data lakesShared app data
Most architectures combine services — EBS for the database, S3 for backups, EFS for shared config.
No single service fits all workloads. FSx is also available for SMB/Lustre/NetApp specialty filesystems. S3 and EFS both provide eleven nines durability. S3 is accessed via HTTPS/REST API while EFS provides a POSIX-compliant filesystem. EBS achieves durability through replication within the AZ; for cross-AZ durability, you need snapshots.

When to Use What

🖩 Use EBS When...

  • Running a self-managed database
  • EC2 boot volume needed
  • Sub-millisecond latency required

📦 Use S3 When...

  • Storing backups, logs, or media
  • Building a data lake
  • Hosting static websites

📁 Use EFS When...

  • Multiple instances need shared files
  • POSIX filesystem required
  • Unpredictable storage growth

⚡ Use Instance Store When...

  • Temporary scratch space or buffers
  • Regenerable caches
  • Maximum I/O performance needed
These are guidelines, not hard rules. Real architectures often combine multiple services. For example, a media processing pipeline might use S3 for source and destination files, EBS for the processing instance's OS and temp files, and EFS if multiple processing instances need to share intermediate results.
