🗃 AWS Storage Types

Block, File, and Object Storage

Before diving into specific services, let's establish the three fundamental storage paradigms and how they map to AWS offerings.

Three Storage Paradigms

🧱 Block Storage

  • Data split into fixed-size blocks
  • Each block has a unique address
  • Low latency, high IOPS
  • Use: databases, boot volumes, transactional workloads

📁 File Storage

  • Hierarchical directory structure
  • Accessed via NFS or SMB
  • Shared across multiple clients
  • Use: home directories, CMS, shared application data

📦 Object Storage

  • Flat namespace, no hierarchy
  • Data + metadata + unique key
  • Massive scalability
  • Use: backups, media, data lakes, static assets
Block storage is like a raw hard drive — the OS manages the filesystem. File storage provides a shared filesystem. Object storage treats each item as an immutable object with metadata. Each paradigm has distinct performance, cost, and access characteristics.

🖩 EC2 Instance Storage & Amazon EBS

Block Storage for EC2 Workloads

Let's start with the block storage options directly attached to or associated with EC2 instances.

EC2 Instance Store

⚡ Characteristics

  • Physically attached NVMe/SSD disks on the host
  • Extremely high I/O performance (millions of IOPS)
  • Included in instance price — no extra cost
  • Size and count depend on instance type

⚠️ Limitations

  • Ephemeral — data lost on stop, terminate, or hardware failure
  • Cannot be detached or reattached
  • No snapshot support
  • Not all instance types include instance store
Rule of thumb: Never store data on instance store that you cannot afford to lose. Use it for caches, scratch data, and temporary buffers. Instance store is ideal for temporary storage that requires very high throughput — like buffers, caches, or scratch space. The i3 and i3en families offer up to 60 TB of NVMe instance storage. Always replicate critical data elsewhere.

Amazon Elastic Block Store (EBS)

💾 Persistent Block Storage for EC2

EBS provides persistent, network-attached block storage volumes for EC2 instances within a single Availability Zone.

  • Automatically replicated within its AZ for durability
  • Supports live volume resizing, type changes, and IOPS adjustments
  • Point-in-time snapshots stored in Amazon S3
  • Encryption at rest and in transit (AES-256)
  • Billed per GB-month provisioned + IOPS (for io1/io2)
EBS is the workhorse for persistent block storage. Think of it as a virtual hard drive. Unlike instance store, EBS data persists across stop/start and can be snapshotted. The tradeoff is slightly higher latency since it's network-attached.

EBS Volume Types — HDD-Backed

📊 st1 — Throughput Optimized

  • Max throughput: 500 MiB/s
  • Size: 125 GiB – 16 TiB
  • Use case: big data, data warehouses, log processing
  • Cannot be a boot volume

🗃 sc1 — Cold HDD

  • Max throughput: 250 MiB/s
  • Size: 125 GiB – 16 TiB
  • Use case: infrequently accessed data, lowest cost
  • Cannot be a boot volume
HDD volumes are optimized for large, sequential I/O — not random reads. For random I/O workloads, always choose SSD-backed volumes. HDD volumes are significantly cheaper per GB. st1 is great for streaming workloads like Kafka, Hadoop HDFS, or ETL pipelines. sc1 is the lowest cost option for data that is rarely accessed but needs to be available when needed. Neither can serve as a boot volume.

EBS Snapshots

📸 Point-in-Time Backups

EBS snapshots capture the state of a volume at a specific moment. Snapshots are stored incrementally in Amazon S3 — only changed blocks since the last snapshot are saved.

  • Incremental: first snapshot is full; subsequent snapshots store only deltas
  • Cross-region copy: snapshots can be copied to other regions for DR
  • Create volumes: restore a snapshot into a new EBS volume in any AZ within the region
  • Share: snapshots can be shared with other AWS accounts or made public
  • Fast Snapshot Restore (FSR): eliminates latency on first read (at additional cost)
Best practice: Use Amazon Data Lifecycle Manager (DLM) to automate snapshot creation and retention schedules. Snapshots are critical for backup and disaster recovery. Even though only changed blocks are stored, any single snapshot is self-sufficient — you can delete earlier snapshots without affecting later ones. FSR is useful for boot volumes or databases that need immediate full performance.

EBS Encryption & Multi-Attach

🔒 Encryption

  • AES-256 at rest, encrypted in transit to EC2
  • Uses AWS KMS keys (AWS-managed or customer-managed)
  • Snapshots of encrypted volumes are encrypted
  • Enable encryption by default per region

🔗 Multi-Attach

  • Available on io1 and io2 volumes only
  • Attach a single volume to up to 16 Nitro-based instances
  • All instances must be in the same AZ
  • Application must manage concurrent write access
Encryption is transparent to the application — no code changes needed. Multi-Attach enables shared-disk clusters but requires a cluster-aware filesystem (like GFS2 or OCFS2) to prevent data corruption from concurrent writes.

EBS Pricing & Lifecycle Considerations

  • gp3/gp2: charged per GB-month provisioned
  • io1/io2: per GB-month + per provisioned IOPS-month
  • st1/sc1: per GB-month (lowest rates)
  • Snapshots: per GB-month of data stored (incremental)
  • Data transfer: no charge for EBS ↔ EC2 within the same AZ
⚠️ Cost optimization tips — see speaker notes
Cost optimization tips: Delete unattached EBS volumes — you pay even if they are not attached. Delete old snapshots you no longer need. Right-size volumes; use Elastic Volumes to resize without downtime. Migrate gp2 to gp3 for most workloads to save ~20%. A common source of unexpected AWS bills is orphaned EBS volumes left behind after terminating instances. AWS Trusted Advisor and Cost Explorer can help identify these. Elastic Volumes lets you change type, size, and IOPS on the fly.

📦 Amazon S3

Object Storage at Any Scale

Now we move to one of AWS's most important and widely used services — Amazon S3. It's not just storage; it's a foundational building block for data lakes, backups, content delivery, and more.

What is Amazon S3?

🌎 Object Storage, Built for the Cloud

Amazon Simple Storage Service (S3) stores data as objects within buckets. Each object consists of data, metadata, and a unique key. S3 is designed for 99.999999999% (11 nines) durability.

Core Concepts

  • Bucket: container for objects (globally unique name)
  • Object: file + metadata (up to 5 TB)
  • Key: unique identifier within a bucket
  • Prefix: simulates folder structure

Key Properties

  • Unlimited storage capacity
  • Regional service (data stays in selected region)
  • RESTful API access (GET, PUT, DELETE)
  • Event notifications and triggers
S3 is one of the oldest AWS services (launched 2006) and is incredibly versatile. The flat namespace with keys means there are no real directories — the console simulates folders using the "/" delimiter in keys. Objects can range from 0 bytes to 5 TB; uploads over 5 GB must use multipart upload.

S3 Bucket Naming & Structure

Naming Rules

  • Globally unique across all AWS accounts
  • 3–63 characters long
  • Lowercase letters, numbers, hyphens only
  • Must start with a letter or number
  • Cannot be formatted as an IP address

Object Addressing

https://my-bucket.s3.amazonaws.com/photos/image.jpg

Bucket: my-bucket
Key:    photos/image.jpg
Region: inferred or specified
Prefix: photos/ acts as a virtual folder. S3 has a flat structure internally.
Bucket names form part of the DNS name, hence the strict rules. The global uniqueness requirement means you cannot create a bucket with a name someone else already uses in any region. Plan your naming convention early — include account ID or project prefix for uniqueness.

S3 Storage Class — Cost Tradeoffs

Storage Cost (Low → High)

  1. Glacier Deep Archive (~$1/TB/mo)
  2. Glacier Flexible / Instant Retrieval
  3. One Zone-IA / Standard-IA
  4. Intelligent-Tiering
  5. S3 Standard (~$23/TB/mo)

Retrieval Speed (Fast → Slow)

  1. S3 Standard / IA classes (milliseconds)
  2. Glacier Instant Retrieval (milliseconds)
  3. Glacier Flexible (1 min–12 hrs)
  4. Glacier Deep Archive (12–48 hrs)
Standard-IA and One Zone-IA have a minimum storage duration charge of 30 days and a per-GB retrieval fee. Glacier classes have a 90-day (Instant/Flexible) or 180-day (Deep Archive) minimum. Intelligent-Tiering has a small monitoring fee per object but no retrieval fees when objects move between tiers.

S3 Lifecycle Policies

Automate the transition of objects between storage classes or schedule deletion.

🔄 Example Lifecycle Rule

{
  "Rules": [{
    "ID": "ArchiveOldLogs",
    "Filter": { "Prefix": "logs/" },
    "Status": "Enabled",
    "Transitions": [
      { "Days": 30, "StorageClass": "STANDARD_IA" },
      { "Days": 90, "StorageClass": "GLACIER" },
      { "Days": 365, "StorageClass": "DEEP_ARCHIVE" }
    ],
    "Expiration": { "Days": 2555 }
  }]
}
Key benefit: Lifecycle policies run automatically — no Lambda functions or scripts needed. Transitions happen at midnight UTC on the day the rule triggers.
Lifecycle policies are one of the most effective ways to optimize S3 costs. You can apply rules to the entire bucket or filter by prefix and/or tags. Transitions only move in one direction (Standard -> IA -> Glacier). You can also expire (delete) objects or incomplete multipart uploads.

S3 Versioning

How It Works

  • Enabled at the bucket level
  • Every PUT creates a new version with a unique Version ID
  • DELETE inserts a delete marker (soft delete)
  • Retrieve any previous version by Version ID
  • Once enabled, cannot be disabled — only suspended

Benefits

  • Protection against accidental overwrites and deletes
  • Enables MFA Delete for additional protection
  • Required for S3 Cross-Region Replication
  • Combine with lifecycle rules to expire old versions
Cost impact: Every version is a separate object that incurs storage charges. Use lifecycle rules to expire noncurrent versions (e.g., keep only the last 3). Versioning is essential for data protection but can significantly increase storage costs if not managed. A common pattern is to enable versioning with a lifecycle rule that expires noncurrent versions after 30 days. MFA Delete requires multi-factor authentication to permanently delete a version or change versioning state.

S3 Access Control

📜 Bucket Policies

  • JSON-based resource policy
  • Attached to the bucket
  • Grant cross-account access
  • Condition keys (IP, VPC, MFA)

👥 IAM Policies

  • Attached to IAM users, groups, or roles
  • Centralized identity-based control
  • Fine-grained per-action permissions
  • Recommended primary method

🚫 S3 Block Public Access

  • Account and bucket-level guardrail
  • Overrides ACLs and bucket policies
  • Enabled by default on new buckets
  • Four independent settings
ACLs are legacy. AWS recommends disabling ACLs (Bucket Owner Enforced) and using bucket policies + IAM policies for all access control. Access control in S3 has evolved significantly. The current best practice is: disable ACLs, use Block Public Access as a safety net, and manage permissions through IAM policies for same-account and bucket policies for cross-account access. The final effective permission is the union of IAM policy and bucket policy, restricted by Block Public Access.

S3 Bucket Policy — Example

{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "AllowReadFromVPC",
    "Effect": "Allow",
    "Principal": "*",
    "Action": ["s3:GetObject"],
    "Resource": "arn:aws:s3:::my-app-data/*",
    "Condition": {
      "StringEquals": { "aws:SourceVpc": "vpc-0a1b2c3d4e5f" }
    }
  }]
}
This example shows two common patterns: restricting access to a specific VPC, and denying uploads that are not encrypted with KMS. Bucket policies support the full range of IAM condition keys. The Deny statement ensures that every object uploaded must use SSE-KMS encryption.

Default encryption: Since January 2023, all new objects in S3 are encrypted with SSE-S3 by default — no configuration needed. SSE-KMS adds CloudTrail logging of key usage and separation of duties between data and key administrators. SSE-C is for organizations that must retain full control of encryption keys but don't want to handle encryption themselves. Client-side encryption is the most secure but requires application changes.

S3 Use Cases

💾 Backup & Disaster Recovery

  • Cross-region replication for DR
  • EBS snapshot storage
  • Database backup targets

🌐 Static Website Hosting

  • Host HTML/CSS/JS directly from S3
  • Combine with CloudFront for CDN
  • Custom domain with Route 53

📊 Data Lake Foundation

  • Central repository for raw data
  • Query in-place with Athena, Redshift Spectrum
  • Decouple storage from compute

📶 Content Distribution

  • Media files, software updates
  • Pre-signed URLs for secure sharing
  • Transfer Acceleration for global uploads
S3 is often called the "storage backbone of AWS." It underpins many other services — EBS snapshots are stored in S3 (though not visible in your bucket), CloudTrail logs go to S3, and many analytics services read directly from S3. S3 is also the most common origin for CloudFront distributions.

S3 Pricing Model

💰 Pay for What You Use

S3 pricing has three main components. There is no minimum commitment and no upfront cost.

Storage

  • Per GB-month
  • Varies by storage class
  • Standard: ~$0.023/GB (US East)

Requests

  • PUT/COPY/POST/LIST per 1,000 requests
  • GET/SELECT per 1,000 requests
  • Lifecycle transitions count as requests

Data Transfer

  • Data IN: free
  • Data OUT to internet: tiered pricing
  • S3 → CloudFront: free
Cost tip: Transferring data from S3 to CloudFront is free. Use CloudFront to serve S3 content and save on data transfer. A common surprise is request pricing — millions of tiny GET requests can add up. Also, listing requests (LIST) are 5x more expensive than GET. For data transfer, the first 100 GB/month out to the internet is free (free tier). Cross-region replication incurs data transfer charges between regions.

🧮 Choose the Right Storage Service

Decision Factors and Comparisons

Now that we've covered the details of EBS and S3, let's step back and talk about how to make the right choice for a given workload.

Storage Decision Factors

  • Access pattern: random vs. sequential, frequency of access
  • Performance: latency requirements, IOPS, throughput
  • Durability: how critical is the data?
  • Availability: single-AZ vs. multi-AZ vs. multi-region
  • Scalability: known size vs. unbounded growth
  • Sharing: single instance vs. shared access
  • Protocol: block, file (NFS/SMB), or HTTP/API
  • Cost: per-GB storage, per-IO, data transfer
No single service fits all workloads. Most architectures use a combination of storage services — for example, EBS for the database volume, S3 for backups, and EFS for shared configuration files.
Encourage participants to think about their own workloads. A web application might use EBS for the database, S3 for user uploads and static assets, and EFS for shared session data. The key is matching the service characteristics to the workload requirements.

When to Use What

🖩 Use EBS When...

  • Running a database (RDS custom, self-managed)
  • You need a boot volume for EC2
  • Sub-millisecond latency is required
  • Data must persist across instance stop/start

📦 Use S3 When...

  • Storing backups, logs, or media
  • Building a data lake
  • Hosting static websites
  • Objects are written once and read many times

📁 Use EFS When...

  • Multiple instances need shared file access
  • POSIX-compliant filesystem is required
  • Storage size is unpredictable (auto-scaling)

⚡ Use Instance Store When...

  • Temporary scratch space or buffers
  • Cache that can be regenerated
  • Maximum I/O performance needed
These are guidelines, not hard rules. Real architectures often combine multiple services. For example, a media processing pipeline might use S3 for source and destination files, EBS for the processing instance's OS and temp files, and EFS if multiple processing instances need to share intermediate results.

📋 Module 4 Summary

Key Takeaways

  • AWS offers block, file, and object storage paradigms
