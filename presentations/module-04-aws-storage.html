🗃 AWS Storage Types

Block, File, and Object

Before diving into specific services, let's establish the three fundamental storage paradigms and how they map to AWS offerings.

🗃 Three Storage Types

🧱 Block

  • Like a hard drive
  • Low latency, high IOPS
  • Databases, boot volumes

📁 File

  • Shared filesystem (NFS/SMB)
  • Multiple instances access
  • EFS, FSx

📦 Object

  • Flat namespace + metadata
  • Massive scalability
  • Backups, media, data lakes
Block storage is like a raw hard drive — the OS manages the filesystem. File storage provides a shared filesystem accessible by multiple servers. Object storage treats each item as a flat object with metadata and a unique key. Each has distinct performance, cost, and access characteristics. AWS maps these to EBS/Instance Store (block), EFS/FSx (file), and S3 (object).

🖩 EC2 Storage

Instance Store and Amazon EBS

Let's look at the block storage options directly attached to or associated with EC2 instances.

⚡ Instance Store vs EBS

Instance Store

  • Physically attached to host
  • Ephemeral (data lost on stop)
  • Highest possible IOPS
  • Use for caches, temp data

Amazon EBS

  • Network-attached volumes
  • Persistent across stop/start
  • Snapshots for backup
  • Use for databases, OS volumes
Instance store is included with certain EC2 instance types and offers the highest raw performance since it's physically attached. However, data is lost when the instance stops or terminates — never store data you can't afford to lose. EBS is the workhorse for persistent storage: network-attached volumes within a single AZ, replicated for durability, supporting snapshots and encryption. EBS volumes can be resized, re-typed, and moved between instances (within the same AZ).

💾 EBS Volume Types

SSD-Backed

  • gp3/gp2: general purpose (boot, apps)
  • io2/io1: high-performance databases

HDD-Backed

  • st1: throughput-optimized (big data)
  • sc1: cold storage (infrequent access)
Default choice: gp3 for most workloads (balanced cost and performance).
gp3 is the latest general purpose SSD — it offers a baseline of 3,000 IOPS and 125 MB/s, and you can independently provision more. gp2 uses a burst credit model. io2 Block Express is for the most demanding databases needing sub-millisecond latency and up to 256,000 IOPS. st1 is for sequential reads (data warehouses, log processing). sc1 is the lowest cost for infrequently accessed data. Most workloads should start with gp3 and upgrade only if monitoring shows they need more.

📦 Amazon S3

Scalable object storage

S3 is one of the most widely used AWS services. It stores virtually unlimited data as objects, accessed over HTTPS.

📦 What Is Amazon S3?

Simple Storage Service — store and retrieve any amount of data, anytime, from anywhere.

  • Objects stored in buckets (globally unique name)
  • 11 nines (99.999999999%) durability
  • Access via HTTPS / REST API
  • Up to 5 TB per object
S3 stores objects (files + metadata) in buckets. Bucket names must be globally unique across all AWS accounts. Each object is identified by a key (like a file path). S3 automatically replicates objects across at least 3 AZs within a Region for 11 nines of durability. Access is via HTTPS, AWS SDKs, CLI, or the Console. Common use cases: static website hosting, backup and archive, data lakes, media distribution, and application assets.

💰 S3 Storage Classes

  • S3 Standard: frequent access, lowest latency
  • S3 Intelligent-Tiering: auto-moves between tiers
  • S3 Standard-IA: infrequent access, lower cost
  • S3 Glacier / Deep Archive: archival, minutes to hours retrieval
Tip: Use lifecycle policies to automatically transition objects between classes.
S3 offers multiple storage classes to optimize cost based on access patterns. Standard is the default for frequently accessed data. Intelligent-Tiering automatically moves objects to the most cost-effective tier based on access patterns — great when you don't know the pattern. Standard-IA costs less per GB but charges per retrieval. One Zone-IA stores in a single AZ for even lower cost. Glacier Instant Retrieval, Flexible Retrieval, and Deep Archive offer progressively cheaper storage for archival with varying retrieval times from milliseconds to 12 hours. Lifecycle policies automate transitions.

🔒 S3 Features

  • Versioning: keep multiple versions of objects
  • Encryption: SSE-S3, SSE-KMS, or client-side
  • Access control: IAM policies, bucket policies, ACLs
  • Block Public Access: account-level safety net
Versioning protects against accidental deletion and overwrites — once enabled, every object change creates a new version. Encryption at rest is available with S3-managed keys (SSE-S3, free), KMS-managed keys (SSE-KMS, for audit trails), or client-side encryption. Access control uses IAM policies (identity-based), bucket policies (resource-based), and Block Public Access settings (account or bucket-level). Best practice: enable versioning, enable default encryption, and turn on Block Public Access unless you explicitly need public objects.

🎯 Choosing the Right Storage

Use EBS when...

  • Databases need low latency
  • Boot volumes for EC2
  • Single-instance block access

Use S3 when...

  • Storing files, images, backups
  • Data lakes and analytics
  • Static website hosting
The general rule: EBS for data that needs to be accessed as a mounted drive by a single instance (databases, application files). S3 for data accessed over HTTP by many consumers (images, documents, backups, logs). EFS when you need a shared filesystem across multiple instances. The decision comes down to access pattern, performance needs, and whether the data needs to look like a filesystem or can be accessed as objects via API.

📋 Module 4 Summary

  • Block, file, object: three storage paradigms
  • EBS: persistent block storage for EC2
  • S3: scalable object storage (11 nines durability)
  • Storage classes optimize cost by access pattern
Recap: AWS offers storage across three paradigms. Instance store is ephemeral but fast; EBS is persistent block storage within an AZ. S3 provides virtually unlimited object storage with 11 nines of durability and multiple storage classes. Versioning, encryption, and access control are built-in S3 features. Choose based on access patterns and performance requirements.
