☁️ Compute as a Service

From physical servers to cloud compute

We start by understanding what "compute as a service" means and how it differs from traditional on-premises server management.

🏢 Traditional vs Cloud Compute

Traditional Data Center

  • Upfront hardware purchase
  • Weeks-to-months lead time
  • Over-provision for peak
  • Maintain hardware, cooling, power

Cloud Compute

  • No upfront investment
  • Provision in seconds
  • Scale to actual demand
  • AWS manages infrastructure
The key shift is from capital expense to operational expense, and from fixed capacity planning to elastic scaling. Cloud compute removes the undifferentiated heavy lifting of managing physical servers. Traditional data centers also have fixed capacity that is difficult to scale.

💻 Amazon EC2

Elastic Compute Cloud — Virtual servers in the cloud

EC2 is the foundational compute service in AWS. Understanding EC2 is essential for almost every AWS workload.

What is Amazon EC2?

💡 Virtual Servers in the Cloud

Resizable virtual machines launched in minutes. Pay only while running.

  • Configurable — choose CPU, memory, storage, networking
  • Multi-OS — Amazon Linux, Ubuntu, Windows, RHEL, SUSE
  • VPC integration — network isolation and security controls
  • Root access — install any software
EC2 gives you virtual machines with root/admin access, allowing you to install and configure software just as you would on a physical server. The key difference is the speed of provisioning and the elasticity of scaling.

🗃 Amazon Machine Images (AMIs)

A template defining OS, software, and launch permissions for an instance.

AMI Sources

  • AWS-provided — Amazon Linux, Windows, Ubuntu
  • Marketplace — pre-configured vendor images
  • Community — shared by other users
  • Custom — your own golden images

AMI Contains

  • Root volume template (OS + apps)
  • Launch permissions
  • Block device mapping
  • Architecture (x86_64, arm64)
AMIs are region-specific but can be copied across regions. Custom AMIs are a best practice for creating golden images with your organization's software and security baselines pre-installed. This dramatically reduces launch time for new instances.

📚 EC2 Instance Type Naming

Example: c5n.xlarge

Component Value Meaning
Family c Compute Optimized
Generation 5 5th generation
Attributes n Network enhanced
Size xlarge CPU & memory allocation
Each step up in size typically doubles the vCPUs and memory. Additional attributes like 'n' (network optimized), 'a' (AMD processor), 'g' (Graviton/ARM), 'd' (local NVMe storage) provide further differentiation. Always choose the latest generation for best price-performance. Common sizes: nano, micro, small, medium, large, xlarge, 2xlarge, 4xlarge, up to metal.

📊 EC2 Instance Families

Family Prefixes Optimized For Use Cases
General Purpose T, M Balanced CPU/memory Web servers, small DBs
Compute Optimized C High-performance CPU Batch, HPC, ML inference
Memory Optimized R, X Large in-memory datasets Caches, real-time analytics
Storage Optimized D, I High sequential I/O Data warehousing
Accelerated P, G GPU / accelerators ML training, video
Tip: Start with General Purpose (T3/M6i). Optimize after profiling your workload.
T-series instances use a burstable CPU credit model, which is cost-effective for variable workloads. M-series provides fixed baseline performance. When selecting instance families, match the resource profile of your application: CPU-bound workloads go to C, memory-bound to R/X, and so on.

💰 EC2 Pricing Models

Model Commitment Discount Best For
On-Demand None 0% Short-term, unpredictable
Savings Plans 1 or 3 yr ($/hr) Up to 72% Steady-state, flexible
Reserved 1 or 3 yr Up to 72% Predictable, fixed type
Spot None Up to 90% Fault-tolerant batch
Dedicated Hosts Varies Varies Licensing, compliance
Strategy: Baseline on Savings Plans, burst with On-Demand, batch on Spot.
Savings Plans are generally recommended over Reserved Instances for new customers because they offer similar discounts with more flexibility. Spot Instances can be interrupted with a 2-minute warning, so design for fault tolerance. Dedicated Hosts give you a physical server and are required for some socket-based software licenses. A common cost optimization strategy is to cover your baseline with Savings Plans, handle variable demand with On-Demand, and use Spot for fault-tolerant batch workloads. This can reduce your overall compute bill by 50-70%. Payment options for Reserved: All Upfront, Partial, No Upfront. Convertible RIs can change instance type.

🔐 Key Pairs & Security Groups

🔑 Key Pairs

  • SSH (Linux) / RDP (Windows) access
  • Private key downloaded once
  • ED25519 or RSA supported
  • Protect with chmod 400

🛡 Security Groups

  • Instance-level virtual firewall
  • Stateful — return traffic auto-allowed
  • Allow rules only (no deny)
  • Default: all inbound denied
Key pairs are essential for secure access. Security groups act as instance-level firewalls. A common pattern is to create separate security groups for web, app, and database tiers, and reference them in rules rather than using IP addresses. This creates dynamic, self-updating firewall rules. Default outbound is all allowed. Can reference other security groups in rules.

🚀 User Data & Bootstrapping

Scripts that run once at first launch as root.

#!/bin/bash
yum update -y
yum install -y httpd
systemctl start httpd
systemctl enable httpd
echo "<h1>Hello from $(hostname)</h1>" > /var/www/html/index.html

Key Points

  • First boot only (by default)
  • Increases boot time — keep lean
  • Logged to /var/log/cloud-init-output.log
User data is frequently used to install agents, configure applications, pull code from repositories, or join the instance to a domain. For complex bootstrapping, consider using AWS Systems Manager, CloudFormation cfn-init, or pre-baked AMIs instead. Size limit: 16 KB raw or 64 KB base64-encoded.

🔄 EC2 Instance Lifecycle

States, transitions, and storage implications

Understanding the lifecycle is critical for cost management and operational procedures. Each state has billing and data implications.

🔄 EC2 Instance States

State Description Billing
pending Being launched Not charged
running Active and operational Charged
stopped Shut down, EBS persists EBS costs only
terminated Permanently deleted Not charged
Key: Stopped instances keep EBS volumes and private IP. Terminated instances are gone.
Additional transitional states: stopping and shutting-down. A stopped instance retains its instance ID, EBS volumes, Elastic IP associations, and private IP. However, data on instance store volumes is lost when stopping. A terminated instance is gone permanently, though EBS volumes can be configured to persist via the DeleteOnTermination attribute.

🔧 Lifecycle Actions

Start / Stop / Reboot

  • Stop — EBS data saved; may change host
  • Reboot — stays on same host, IPs kept
  • Public IPv4 changes on stop/start (unless Elastic IP)

Terminate / Hibernate

  • Terminate — permanently deleted; root EBS gone by default
  • Hibernate — saves RAM to EBS; faster resume
  • Enable termination protection for production
Start/stop only works for EBS-backed instances. Termination protection is a simple flag that prevents accidental termination via the console or API. It does not prevent termination from within the OS (shutdown command) or from Auto Scaling events. Hibernate must be enabled at launch time. Instance store data is always lost on stop or terminate, but preserved on reboot.

💾 EBS-Backed vs Instance Store

EBS-Backed (Default)

  • Data persists when stopped
  • Supports stop, start, reboot
  • Snapshots for backup
  • Resizable volumes

Instance Store

  • Ephemeral local disk
  • Data lost on stop/terminate
  • Very high I/O performance
  • Best for caches, temp data
Key takeaway: Use EBS-backed instances for most workloads. Use instance store for ephemeral, high-performance scratch data only. Most modern instances are EBS-backed. Instance store volumes provide very high IOPS because they are physically attached to the host. Some instance types like i3 and d2 are specifically designed to leverage local NVMe storage.

📦 AWS Container Services

ECS, EKS, and Fargate

Containers are a key technology for modern application architectures. AWS offers managed services that remove the operational overhead of running container orchestration platforms.

📦 Containers vs Virtual Machines

Virtual Machines

  • Full OS per instance
  • Hypervisor-based isolation
  • Minutes to boot
  • Gigabytes in size

Containers

  • Share host OS kernel
  • Process-level isolation
  • Seconds to start
  • Megabytes in size
Containers package application code, runtime, libraries, and dependencies into a single portable unit that runs consistently across environments. Containers are not a replacement for VMs — they solve different problems. VMs provide hardware-level isolation; containers provide application-level isolation. Both provide strong security boundaries. Containers enable consistent dev-to-prod environments. In AWS, containers often run ON EC2 instances (or Fargate), so both technologies are used together.

📦 Amazon ECS

AWS-native container orchestration service.

Tasks

  • Defined in JSON
  • Image, CPU, memory, ports
  • One+ containers per task

Services

  • Maintain desired task count
  • ALB/NLB integration
  • Auto-restart on failure

Clusters

  • Logical task grouping
  • EC2 or Fargate launch
  • Regional resource
ECS is AWS-native and tightly integrated with other AWS services like IAM, CloudWatch, ALB, and VPC. It's simpler to operate than Kubernetes and is a great choice if you don't already have Kubernetes expertise or requirements. Simpler learning curve than EKS; best for AWS-focused teams.

☸️ Amazon EKS

Managed Kubernetes control plane on AWS.

Key Features

  • Upstream K8s — fully conformant
  • AWS manages control plane
  • Worker nodes or Fargate
  • Full K8s ecosystem (Helm, operators)

Choose EKS When

  • Existing Kubernetes expertise
  • Multi-cloud / hybrid strategy
  • K8s-specific features needed
EKS is ideal when your team already knows Kubernetes or when portability across cloud providers is important. EKS Anywhere extends this to on-premises. The control plane runs across multiple AZs for high availability, and AWS handles patches and upgrades. Integrates with IAM, VPC, ALB, CloudWatch. Choose ECS for simplicity; choose EKS for K8s portability and ecosystem. Both support Fargate for serverless containers.

⚡ AWS Fargate

Serverless Compute for Containers

No EC2 instances to manage. Define CPU and memory per task; Fargate handles the rest.

  • Works with ECS and EKS
  • Per-task isolation — dedicated kernel per task
  • Pay per vCPU + memory per second
  • No cluster capacity planning
Fargate is often described as "serverless containers." The trade-off is slightly higher per-unit cost compared to self-managed EC2, but you eliminate the operational overhead of patching, scaling, and managing the underlying compute fleet. For many teams, this is a worthwhile trade-off. Benefits also include enhanced security isolation per task and the ability to scale without over-provisioning.

⚡ Serverless Computing

No servers to manage — focus on your code

Serverless is one of the fastest-growing cloud paradigms. It shifts even more responsibility to the cloud provider, letting developers focus purely on business logic.

⚡ What Serverless Means

Serverless ≠ No Servers

Servers exist — but you don't provision, patch, or scale them.

Core Benefits

  • No provisioning needed
  • Auto-scaling from zero to thousands
  • Pay-per-use pricing
  • Built-in HA across AZs

AWS Serverless Services

  • Lambda — compute
  • API Gateway — REST APIs
  • DynamoDB — database
  • S3 — storage
Other serverless services include AWS Fargate (containers) and Amazon EventBridge (events). Serverless is a spectrum, not a binary. Some services like Lambda are fully serverless while others like Fargate are "serverless containers." The key criteria: no server management, automatic scaling, pay-per-use pricing, and built-in fault tolerance.

λ How AWS Lambda Works

Event-Driven Execution

Upload code → Configure trigger → Lambda runs on events → Pay only for execution time.

Common Triggers

  • S3 — object uploads
  • API Gateway — HTTP requests
  • CloudWatch — scheduled (cron)
  • DynamoDB Streams — table changes

Execution Flow

  1. Event triggers Lambda
  2. Execution environment provisioned
  3. Handler processes event
  4. Response returned
Other common event sources include SQS/SNS (messages) and Kinesis (stream processing). Lambda automatically scales by creating new execution environments for each concurrent invocation. Cold starts occur when a new environment is provisioned; warm environments are reused for subsequent invocations. Provisioned Concurrency can eliminate cold starts for latency-sensitive applications.

🔧 Lambda Function Anatomy

Handler

def handler(event, context):
    name = event.get('name', 'World')
    return {
        'statusCode': 200,
        'body': f'Hello, {name}!'
    }

Configuration

  • Runtime: Python, Node.js, Java, .NET, Go, Ruby
  • Memory: 128 MB – 10,240 MB
  • Timeout: up to 15 minutes
  • Execution role: IAM permissions
The handler function signature varies by runtime. The event object contains the trigger data (e.g., S3 bucket name and key, API Gateway request body). The context object provides metadata like the function name, remaining execution time, and request ID. vCPU allocation scales linearly with memory — at 1,769 MB you get 1 full vCPU. Environment variables can be used for configuration. Resource policies control who can invoke the function.

💰 Lambda Pricing & Limits

Pricing

  • Requests: $0.20 per 1M
  • Duration: per GB-second (1ms granularity)
  • Free tier: 1M requests + 400K GB-sec/mo
  • Zero cost when idle

Key Limits

  • Timeout: 15 minutes max
  • Package: 50 MB zip, 250 MB unzipped
  • Concurrency: 1,000 (adjustable)
  • /tmp storage: up to 10 GB
Common use cases: REST API backends with API Gateway, real-time file processing via S3 triggers, ETL data transformation, and scheduled cron jobs. The generous free tier means many lightweight workloads run at no cost. Lambda is not suited for long-running processes (over 15 minutes) or workloads requiring persistent connections. Cost example: a function with 256 MB memory, 200ms average duration, invoked 10M times/month costs about $10.33. For sustained high-throughput workloads, EC2 or Fargate may be more cost-effective.

🤔 Choosing the Right Compute

Decision framework for AWS compute services

There is no one-size-fits-all answer. The right choice depends on your application requirements, team skills, and operational priorities.

🤔 Compute Decision Matrix

Factor EC2 ECS / EKS Lambda
Control Full OS Container-level Code-level
Scaling Auto Scaling Groups Service auto-scaling Automatic per request
Ops Overhead High Medium Low
Cost Model Per-second/hour Per-hour or per-task Per-request + duration
Startup Minutes Seconds Milliseconds
Best practice: Start with the least overhead that meets your requirements. Most architectures combine multiple services.
Decision guide questions: Need full OS access? EC2. Containerized or microservices? ECS/EKS + Fargate. Event-driven and short-lived? Lambda. Need Kubernetes portability? EKS. Want minimal ops? Lambda or Fargate. The decision is rarely binary. Many production environments use multiple compute services together. For example, a web application might use EC2 for the primary app server, ECS/Fargate for microservices, and Lambda for event-driven tasks like image processing.

📋 Module 2 Summary

  • EC2 — virtual servers with full OS control and multiple pricing models
  • Lifecycle states determine billing and data persistence
  • ECS / EKS — managed container orchestration; Fargate for serverless containers
  • Lambda — event-driven, pay-per-use serverless compute
  • No single best option — choose by control, cost, and ops needs
  • Modern apps often combine multiple services
Reinforce key takeaways: EC2 for full control, containers for microservices and portability, Lambda for event-driven workloads. The right answer is almost always "it depends" — and now students have the framework to make that decision.
