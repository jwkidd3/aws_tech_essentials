💻 Amazon EC2

Virtual servers in the cloud

EC2 is the foundational compute service in AWS. Understanding EC2 is essential because most AWS architectures include it in some form.

💻 What Is Amazon EC2?

Resizable virtual machines launched in minutes. Pay only while running.

  • Full OS-level control (Linux, Windows)
  • Choose CPU, memory, storage, network
  • Runs inside your VPC
  • Install any software you need
EC2 gives you virtual machines with root/admin access, allowing you to install and configure software just as you would on a physical server. The key difference is the speed of provisioning and the elasticity of scaling. You launch instances from Amazon Machine Images (AMIs), which are templates containing the OS and pre-installed software. You can use AWS-provided AMIs, marketplace AMIs, or create your own custom golden images.

🛠 EC2 Instance Type Families

  • General Purpose (M, T) — balanced workloads
  • Compute Optimized (C) — CPU-intensive tasks
  • Memory Optimized (R, X) — in-memory databases
  • Storage Optimized (I, D) — high disk throughput
Naming: e.g. m5.xlarge = family (m) + generation (5) + size (xlarge)
There are also accelerated computing instances (P, G families) with GPUs for machine learning and graphics. The T family includes burstable instances that are cost-effective for variable workloads. Instance naming convention: family letter, generation number, then size from nano to metal. Start with general purpose (T3 or M5) and right-size based on monitoring data. You don't need to memorize specs — the key is knowing which family fits which workload pattern.

💰 EC2 Pricing Models

On-Demand

  • Pay by the second/hour
  • No commitment required

Reserved / Savings Plans

  • 1 or 3-year commitment
  • Up to 72% discount

Spot Instances

  • Up to 90% discount
  • Can be interrupted by AWS

Dedicated Hosts

  • Physical server for you
  • Compliance / licensing needs
On-Demand is the default and most flexible — great for unpredictable workloads and development. Reserved Instances and Savings Plans reward commitment with significant discounts — ideal for steady-state production workloads. Spot Instances offer the deepest discounts but can be reclaimed with 2 minutes notice — perfect for batch processing, CI/CD, and fault-tolerant workloads. Dedicated Hosts are for regulatory or licensing requirements (e.g., per-socket Oracle licenses).

🔄 EC2 Instance Lifecycle

Launch → Running → Stop / Terminate

  • Running: billed, fully operational
  • Stopped: no compute charges, EBS persists
  • Terminated: instance deleted permanently
  • Rebooted: keeps public IP and data
Understanding the lifecycle is important for cost management. Stopped instances don't incur compute charges but you still pay for attached EBS volumes and Elastic IPs. Termination is permanent — the instance and its root volume are deleted (unless configured to preserve the volume). Hibernate is a special stop state that saves RAM contents to EBS, enabling faster restart. A stopped instance loses its public IP unless you use an Elastic IP.

📦 Containers & Serverless

Beyond virtual machines

Now let's look at lighter-weight compute options. Containers and serverless reduce the operational overhead compared to managing full EC2 instances.

📦 Containers on AWS

Amazon ECS

  • AWS-native orchestrator
  • Tight AWS integration
  • Simpler to start

Amazon EKS

  • Managed Kubernetes
  • Open-source ecosystem
  • Portable across clouds

AWS Fargate

  • Serverless containers
  • No EC2 to manage
  • Works with ECS or EKS
Containers package application code with dependencies for consistent deployment. ECS is AWS's own orchestrator — simpler if you're all-in on AWS. EKS runs Kubernetes for teams that want portability or already use K8s. Fargate removes the need to manage the underlying EC2 instances for either ECS or EKS — you just define CPU and memory per container. Use ECS for AWS-native simplicity, EKS for Kubernetes ecosystem, Fargate when you want serverless container hosting.

⚡ AWS Lambda

Run code without provisioning servers. Pay only per invocation.

  • Event-driven (S3, API Gateway, etc.)
  • Auto-scales to demand
  • Supports Python, Node.js, Java, Go, .NET
  • Max 15 minutes per execution
Lambda is AWS's Function-as-a-Service offering. You upload code, define a trigger, and AWS handles everything else: provisioning, scaling, patching, and high availability. Billing is per invocation (first 1M requests/month free) plus compute time in GB-seconds. Lambda is ideal for event-driven workloads: processing S3 uploads, handling API requests, running scheduled tasks, and responding to database changes. The 15-minute timeout means Lambda isn't suited for long-running batch jobs.

🎯 Choosing the Right Compute

💻 EC2 When...

  • Full OS control needed
  • Long-running processes
  • Legacy app migration

⚡ Lambda When...

  • Event-driven, short tasks
  • Variable / spiky traffic
  • Zero admin overhead desired
Containers (ECS/EKS) fit between: microservices needing portability without full VM overhead.
There's no one-size-fits-all answer. EC2 gives maximum control, Lambda gives minimum management. Containers sit in between. Many production architectures use a mix: EC2 for the main application, Lambda for event processing, containers for microservices. Start with the question: "How much infrastructure do I want to manage?" and work from there. Cost depends heavily on workload patterns — Lambda can be cheaper for sporadic traffic, EC2 for steady high utilization.

📋 Module 2 Summary

  • EC2: full-control virtual machines
  • Instance families match workload needs
  • Containers: ECS, EKS, Fargate
  • Lambda: serverless, event-driven compute
Recap: EC2 provides full VM control with multiple pricing options. Instance types are grouped into families optimized for different workloads. Containers offer portability and efficiency for microservices. Lambda removes all server management for event-driven functions. The right choice depends on your control needs, workload patterns, and operational preferences.
