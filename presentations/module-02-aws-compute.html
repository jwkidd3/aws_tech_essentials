☁️ Compute as a Service

From physical servers to cloud compute

We start by understanding what "compute as a service" means and how it differs from traditional on-premises server management.

🏢 Traditional Servers vs Cloud Compute

Traditional Data Center

  • Purchase physical hardware upfront
  • Weeks-to-months provisioning lead time
  • Over-provision for peak capacity
  • Maintain hardware, cooling, power
  • Fixed capacity — difficult to scale

Cloud Compute

  • No upfront hardware investment
  • Provision in seconds or minutes
  • Scale to actual demand
  • AWS manages infrastructure
  • Elastic — scale up and down freely
The key shift is from capital expense to operational expense, and from fixed capacity planning to elastic scaling. Cloud compute removes the undifferentiated heavy lifting of managing physical servers.

💻 Amazon EC2

Elastic Compute Cloud — Virtual servers in the cloud

EC2 is the foundational compute service in AWS. Understanding EC2 is essential for almost every AWS workload.

What is Amazon EC2?

💡 Virtual Servers in the Cloud

Amazon EC2 provides resizable virtual machines (instances) that you can launch in minutes, configure to your needs, and pay for only while running.

  • Secure & resizable — choose CPU, memory, storage, and networking
  • Multiple OS support — Amazon Linux, Ubuntu, Windows Server, Red Hat, SUSE, and more
  • Integrates with VPC — full network isolation and security controls
  • Root access — install any software, configure any way you need
EC2 gives you virtual machines with root/admin access, allowing you to install and configure software just as you would on a physical server. The key difference is the speed of provisioning and the elasticity of scaling.

🗃 Amazon Machine Images (AMIs)

An AMI is a template that defines the OS, application software, and launch permissions for an EC2 instance.

AMI Sources

  • AWS-provided — Amazon Linux 2023, Windows Server, Ubuntu
  • AWS Marketplace — third-party vendor images (pre-configured software)
  • Community AMIs — shared by other AWS users
  • Custom AMIs — create your own from running instances

AMI Contains

  • Root volume template (OS + apps)
  • Launch permissions
  • Block device mapping (EBS volumes)
  • Architecture (x86_64, arm64)
AMIs are region-specific but can be copied across regions. Custom AMIs are a best practice for creating golden images with your organization's software and security baselines pre-installed. This dramatically reduces launch time for new instances.

📚 EC2 Instance Type Naming

Naming Convention: c5n.xlarge

Component Value Meaning
Family c Compute Optimized
Generation 5 5th generation
Attributes n Network enhanced
Size xlarge CPU & memory allocation

Common sizes: nano, micro, small, medium, large, xlarge, 2xlarge, 4xlarge ... up to metal

Each step up in size typically doubles the vCPUs and memory. Additional attributes like 'n' (network optimized), 'a' (AMD processor), 'g' (Graviton/ARM), 'd' (local NVMe storage) provide further differentiation. Always choose the latest generation for best price-performance.

📊 EC2 Instance Families

Family Prefixes Optimized For Use Cases
General Purpose T, M Balanced CPU/memory/network Web servers, code repos, small DBs
Compute Optimized C High-performance processors Batch, HPC, ML inference, gaming
Memory Optimized R, X Large in-memory datasets In-memory caches, real-time analytics
Storage Optimized D, I High sequential read/write Data warehousing, distributed FS
Accelerated Computing P, G GPU / hardware accelerators ML training, video encoding, 3D viz
Tip: Start with General Purpose (T3/M6i) for most workloads. Optimize after you understand your application's resource profile.
T-series instances use a burstable CPU credit model, which is cost-effective for variable workloads. M-series provides fixed baseline performance. When selecting instance families, match the resource profile of your application: CPU-bound workloads go to C, memory-bound to R/X, and so on.

💰 EC2 Pricing Models

Model Commitment Discount Best For
On-Demand None 0% Short-term, unpredictable workloads
Reserved Instances 1 or 3 years Up to 72% Steady-state, predictable usage
Savings Plans 1 or 3 years ($/hr) Up to 72% Flexible across instance families/regions
Spot Instances None Up to 90% Fault-tolerant, flexible workloads
Dedicated Hosts On-Demand or Reserved Varies Licensing, compliance, regulatory
Savings Plans are generally recommended over Reserved Instances for new customers because they offer similar discounts with more flexibility. Spot Instances can be interrupted with a 2-minute warning, so design for fault tolerance. Dedicated Hosts give you a physical server and are required for some software licenses that are socket-based.

💰 Pricing: On-Demand vs Spot vs Reserved

On-Demand

  • Pay by the second (Linux) or hour (Windows)
  • No long-term commitment
  • Best for spiky or experimental workloads

Spot Instances

  • Use spare EC2 capacity at steep discounts
  • Can be reclaimed with 2-minute notice
  • Ideal for batch processing, CI/CD, big data

Reserved / Savings Plans

  • Commit to 1 or 3 years of usage
  • Payment options: All Upfront, Partial, No Upfront
  • Savings Plans offer flexibility across families
  • Convertible RIs can change instance type
A common cost optimization strategy is to cover your baseline with Reserved Instances or Savings Plans, handle variable demand with On-Demand, and use Spot for fault-tolerant batch workloads. This can reduce your overall compute bill by 50-70%.

🔐 Key Pairs & Security Groups

🔑 Key Pairs

  • Public/private key pair for SSH (Linux) or RDP password decryption (Windows)
  • AWS stores the public key; you download the private key once
  • ED25519 or RSA (2048-bit+) supported
  • Protect your private key — chmod 400

🛡 Security Groups

  • Virtual firewall at the instance level
  • Stateful — return traffic automatically allowed
  • Allow rules only (no deny rules)
  • Default: all inbound denied, all outbound allowed
  • Can reference other security groups
Key pairs are essential for secure access. Security groups act as instance-level firewalls. A common pattern is to create separate security groups for web, app, and database tiers, and reference them in rules rather than using IP addresses. This creates dynamic, self-updating firewall rules.

🚀 User Data & Bootstrapping

User data scripts run once at first launch as root, automating instance configuration.

#!/bin/bash
yum update -y
yum install -y httpd
systemctl start httpd
systemctl enable httpd
echo "<h1>Hello from $(hostname)</h1>" > /var/www/html/index.html

Important Considerations

  • Runs only on first boot by default (can be changed)
  • Increases boot time — keep scripts lean
  • Output logged to /var/log/cloud-init-output.log
  • Size limit: 16 KB (raw) or 64 KB (base64-encoded)
User data is frequently used to install agents, configure applications, pull code from repositories, or join the instance to a domain. For complex bootstrapping, consider using AWS Systems Manager, CloudFormation cfn-init, or pre-baked AMIs instead.

📄 Instance Metadata Service (IMDS)

Accessible from within the instance at 169.254.169.254

Provides information about the running instance: instance ID, type, region, security credentials, and more.

# IMDSv2 (recommended - token-based)
TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")

curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/instance-id

curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/public-ipv4
Security best practice: Use IMDSv2 (token-based) and disable IMDSv1 to mitigate SSRF attacks.
The instance metadata service is a link-local address available only from within the instance. IMDSv2 requires a session token obtained via a PUT request, which prevents most SSRF attack vectors. Common uses include discovering the instance's own IP, region, or retrieving IAM role credentials for API calls.

🔄 EC2 Instance Lifecycle

States, transitions, and storage implications

Understanding the lifecycle is critical for cost management and operational procedures. Each state has billing and data implications.

🔄 EC2 Instance States

State Description Billing
pending Instance is being launched Not charged
running Instance is active and operational Charged
stopping Instance is preparing to stop Not charged (EBS-backed)
stopped Instance is shut down (EBS volumes persist) Not charged (but EBS costs remain)
shutting-down Instance is preparing to terminate Not charged
terminated Instance is permanently deleted Not charged
A stopped instance retains its instance ID, EBS volumes, Elastic IP associations, and private IP. However, data on instance store volumes is lost when stopping. A terminated instance is gone permanently, though EBS volumes can be configured to persist via the DeleteOnTermination attribute.

🔧 Instance Lifecycle Actions

Start / Stop

  • Only for EBS-backed instances
  • Stopping saves EBS data
  • May launch on different host hardware
  • Public IPv4 changes on restart (unless Elastic IP)

Reboot

  • Equivalent to OS reboot
  • Instance stays on same host
  • Public/private IPs preserved
  • Instance store data preserved

Terminate

  • Instance is permanently deleted
  • Root EBS volume deleted by default
  • Enable termination protection for production
  • Instance store data always lost

Hibernate

  • Saves RAM contents to EBS root volume
  • Faster startup than cold boot
  • Must be enabled at launch
Termination protection is a simple flag that prevents accidental termination via the console or API. It does not prevent termination from within the OS (shutdown command) or from Auto Scaling events. Hibernate is useful for long-running processes that take a long time to initialize.

💾 EBS-Backed vs Instance Store-Backed

EBS-Backed (Recommended)

  • Root volume on Amazon EBS
  • Data persists when stopped
  • Supports stop, start, reboot, terminate
  • Snapshots for backup & AMI creation
  • Can resize volume and change type

Instance Store-Backed

  • Root volume on ephemeral local disk
  • Data lost on stop or terminate
  • Supports reboot and terminate only
  • Very high I/O performance
  • Best for temporary data: caches, buffers
Key takeaway: Use EBS-backed instances for most workloads. Use instance store for ephemeral, high-performance scratch data only.
Most modern instances are EBS-backed. Instance store volumes provide very high IOPS because they are physically attached to the host. Some instance types like i3 and d2 are specifically designed to leverage local NVMe storage for workloads like distributed databases.

📦 AWS Container Services

ECS, EKS, and Fargate

Containers are a key technology for modern application architectures. AWS offers managed services that remove the operational overhead of running container orchestration platforms.

📦 Containers vs Virtual Machines

Virtual Machines

  • Full OS per instance
  • Hypervisor-based isolation
  • Minutes to boot
  • Gigabytes in size
  • Strong security boundaries

Containers

  • Share host OS kernel
  • Process-level isolation (namespaces, cgroups)
  • Seconds to start
  • Megabytes in size
  • Consistent dev-to-prod environments

Containers package application code, runtime, libraries, and dependencies into a single portable unit that runs consistently across environments.

Containers are not a replacement for VMs — they solve different problems. VMs provide hardware-level isolation; containers provide application-level isolation. In AWS, containers often run ON EC2 instances (or Fargate), so both technologies are used together.

📦 Amazon ECS (Elastic Container Service)

Fully managed container orchestration service built by AWS.

Tasks

  • Defined by a task definition (JSON)
  • Specifies container image, CPU, memory, ports, env vars
  • One or more containers per task

Services

  • Maintain desired count of tasks
  • Integrate with ALB/NLB
  • Auto-restart failed tasks

Clusters

  • Logical grouping of tasks/services
  • Run on EC2 or Fargate
  • Regional resource
ECS is AWS-native and tightly integrated with other AWS services like IAM, CloudWatch, ALB, and VPC. It's simpler to operate than Kubernetes and is a great choice if you don't already have Kubernetes expertise or requirements.

☸️ Amazon EKS (Elastic Kubernetes Service)

Managed Kubernetes control plane on AWS.

Key Features

  • Runs upstream Kubernetes — fully conformant
  • AWS manages the control plane (API server, etcd)
  • You manage worker nodes (or use Fargate)
  • Integrates with IAM, VPC, ALB, CloudWatch
  • Supports Kubernetes ecosystem: Helm, operators, service mesh

When to Choose EKS

  • Existing Kubernetes expertise
  • Multi-cloud or hybrid strategy
  • Need K8s-specific features (CRDs, operators)
  • Rich open-source ecosystem required
EKS is ideal when your team already knows Kubernetes or when portability across cloud providers is important. EKS Anywhere extends this to on-premises. The control plane runs across multiple AZs for high availability, and AWS handles patches and upgrades.

⚡ AWS Fargate

Serverless Compute for Containers

Fargate removes the need to provision, configure, or scale EC2 instances for running containers. You define CPU and memory per task, and Fargate handles the rest.

How It Works

  • Works with both ECS and EKS
  • No EC2 instances to manage
  • Each task runs in its own isolation boundary
  • Pay per vCPU and memory per second

Benefits

  • No cluster capacity management
  • Enhanced security isolation per task
  • Scale without over-provisioning
  • Focus on application, not infrastructure
Fargate is often described as "serverless containers." The trade-off is slightly higher per-unit cost compared to self-managed EC2, but you eliminate the operational overhead of patching, scaling, and managing the underlying compute fleet. For many teams, this is a worthwhile trade-off.

🤔 When to Use ECS vs EKS

Factor Amazon ECS Amazon EKS
Complexity Simpler, AWS-native More complex, standard K8s
Portability AWS only Multi-cloud, hybrid, on-prem
Ecosystem AWS integrations Full K8s ecosystem (Helm, Istio, etc.)
Learning Curve Lower Higher (Kubernetes knowledge required)
Team Skills AWS-focused teams Teams with K8s experience
Fargate Support Yes Yes
Rule of thumb: If you have no strong preference, start with ECS + Fargate for simplicity. Use EKS when Kubernetes portability or ecosystem features are required.
Both services are production-grade and used by thousands of customers. The choice is primarily about team expertise and organizational requirements, not about technical capability. Both support Fargate for serverless container execution.

⚡ What is Serverless?

No servers to manage — focus on your code

Serverless is one of the fastest-growing cloud paradigms. It shifts even more responsibility to the cloud provider, letting developers focus purely on business logic.

⚡ What Serverless Means

Serverless ≠ No Servers

Servers still exist — but you don't provision, manage, patch, or scale them. The cloud provider handles all infrastructure operations.

Core Benefits

  • No provisioning — no capacity planning
  • Auto-scaling — from zero to thousands of concurrent executions
  • Pay-per-use — charged only for actual consumption
  • Built-in HA — runs across multiple AZs automatically

AWS Serverless Services

  • AWS Lambda — compute
  • Amazon API Gateway — REST/WebSocket APIs
  • Amazon DynamoDB — database
  • Amazon S3 — storage
  • AWS Fargate — containers
  • Amazon EventBridge — events
Serverless is a spectrum, not a binary. Some services like Lambda are fully serverless while others like Fargate are "serverless containers." The key criteria: no server management, automatic scaling, pay-per-use pricing, and built-in fault tolerance.

λ AWS Lambda

Event-driven serverless compute

Lambda is the core compute service in the AWS serverless portfolio. It fundamentally changes how you think about building applications.

λ How AWS Lambda Works

Event-Driven Execution

Upload your code → Configure a trigger → Lambda runs your function in response to events → You pay only for execution time.

Common Event Sources

  • API Gateway — HTTP requests
  • S3 — object uploads/deletes
  • DynamoDB Streams — table changes
  • SQS / SNS — messages
  • CloudWatch Events — scheduled (cron)
  • Kinesis — stream processing

Execution Flow

  1. Event triggers Lambda
  2. Lambda provisions execution environment
  3. Handler function processes the event
  4. Response returned, environment may be reused
Lambda automatically scales by creating new execution environments for each concurrent invocation. Cold starts occur when a new environment is provisioned; warm environments are reused for subsequent invocations. Provisioned Concurrency can eliminate cold starts for latency-sensitive applications.

🔧 Lambda Function Components

Handler

The entry point method that Lambda calls. Receives the event object and context.

def handler(event, context):
    name = event.get('name', 'World')
    return {
        'statusCode': 200,
        'body': f'Hello, {name}!'
    }

Configuration

  • Runtime: Python, Node.js, Java, .NET, Go, Ruby, or custom
  • Memory: 128 MB to 10,240 MB
  • Timeout: up to 15 minutes
  • Environment variables

Permissions

  • Execution role — IAM role defining what Lambda can access
  • Resource policy — who can invoke the function
The handler function signature varies by runtime. The event object contains the trigger data (e.g., S3 bucket name and key, API Gateway request body). The context object provides metadata like the function name, remaining execution time, and request ID. vCPU allocation scales linearly with memory — at 1,769 MB you get 1 full vCPU.

💰 Lambda Pricing Model

You Pay For

  • Requests: $0.20 per 1 million requests
  • Duration: $0.0000166667 per GB-second
  • Measured in 1ms increments
  • Free tier: 1M requests + 400,000 GB-seconds/month

Cost Example

A function with 256 MB memory, 200ms average duration, invoked 10M times/month:

  • Requests: 10M × $0.20/M = $2.00
  • Duration: 10M × 0.2s × 0.25GB × $0.0000166667 = $8.33
  • Total: ~$10.33/month
Zero cost when idle: Unlike EC2, Lambda has no charges when your function is not executing.
The generous free tier means many lightweight workloads can run at no cost. However, for sustained high-throughput workloads, EC2 or Fargate may be more cost-effective. Always model your expected invocation patterns to compare costs across compute options.

🎯 Lambda Use Cases & Limits

Common Use Cases

  • REST API backends (with API Gateway)
  • Real-time file processing (S3 triggers)
  • Data transformation in ETL pipelines
  • Scheduled tasks (cron jobs)
  • IoT event processing
  • Chatbot and notification workflows

Key Limits

Execution timeout15 minutes
Memory128 MB – 10,240 MB
Deployment package50 MB (zipped), 250 MB (unzipped)
Concurrent executions1,000 (soft limit, adjustable)
Ephemeral storage (/tmp)512 MB – 10,240 MB
Lambda is not suited for long-running processes (over 15 minutes) or workloads requiring persistent connections. For container-based functions that exceed deployment size limits, use Lambda container images (up to 10 GB). The concurrency limit is per-region and can be increased via a support request.

⚖️ Lambda vs EC2 Comparison

Dimension AWS Lambda Amazon EC2
Management Fully managed, no patching You manage OS, patching, security
Scaling Automatic, per-request Manual or via Auto Scaling
Pricing Per-request + duration Per-second/hour while running
Max Runtime 15 minutes Unlimited
State Stateless Stateful (persistent disk)
Cold Start Yes (ms to seconds) No (always running)
Customization Limited (runtime, memory) Full OS control
The key trade-off is control vs. operational simplicity. Lambda eliminates server management but restricts runtime, state, and customization. EC2 gives full control but requires you to manage everything. Many architectures use both — Lambda for event-driven glue logic and EC2 for core application workloads.

🤔 Choosing the Right Compute

Decision framework for AWS compute services

There is no one-size-fits-all answer. The right choice depends on your application requirements, team skills, and operational priorities.

🤔 Compute Decision Matrix

Factor EC2 ECS / EKS Lambda
Control Full OS control Container-level Code-level only
Flexibility Any language/framework Containerized apps Supported runtimes
Scaling Auto Scaling Groups Service auto-scaling Automatic per request
Operational Overhead High (patching, monitoring) Medium Low
Cost Model Per-hour/second Per-hour or per-task Per-request + duration
Startup Time Minutes Seconds Milliseconds
The decision is rarely binary. Many production environments use multiple compute services together. For example, a web application might use EC2 for the primary app server, ECS/Fargate for microservices, and Lambda for event-driven tasks like image processing or notifications.

🛠 Choosing Compute: Decision Guide

Ask These Questions

  1. Do you need full OS access? → EC2
  2. Is your app containerized or microservices-based? → ECS/EKS (+ Fargate for serverless)
  3. Is your workload event-driven and short-lived? → Lambda
  4. Do you need Kubernetes portability? → EKS
  5. Do you want minimal operational overhead? → Lambda or Fargate
Best practice: Start with the least operational overhead that meets your requirements. Move to more control only when necessary.
This is a simplification — real-world decisions involve cost modeling, team capabilities, compliance requirements, and existing tooling. The trend in the industry is toward serverless and containers, away from managing individual EC2 instances, but EC2 remains essential for many workloads.

🏗 Real-World Example: Multi-Tier Application

Employee Directory App

  • Web tier: EC2 behind Application Load Balancer
  • API tier: Lambda + API Gateway
  • Image processing: Lambda triggered by S3 uploads
  • Background jobs: Fargate tasks processing SQS queue
  • Database: Amazon DynamoDB (serverless)

Why Mix Compute Services?

  • Each tier has different requirements
  • Optimize cost for each workload pattern
  • Reduce operational burden where possible
  • Scale each tier independently

📋 Module 2 Summary

  • EC2 provides virtual servers with full OS control, multiple pricing models, and extensive instance type selection
  • EC2 lifecycle states determine billing and data persistence
  • ECS and EKS provide managed container orchestration with optional serverless execution via Fargate
  • Lambda enables event-driven, serverless compute with pay-per-use pricing
  • No single "best" compute — choose based on control, flexibility, cost, and operational requirements
  • Modern architectures often combine multiple compute services
Reinforce key takeaways: EC2 for full control, containers for microservices and portability, Lambda for event-driven workloads. The right answer is almost always "it depends" — and now students have the framework to make that decision.
