🌐 Networking in AWS

Foundational concepts for cloud networking

Before diving into VPC specifics, let's refresh core networking concepts and understand how they translate to the cloud.

Networking Basics Refresher

📅 IP Addresses

  • IPv4: 32-bit, e.g. 10.0.1.5
  • IPv6: 128-bit, e.g. 2001:db8::1
  • Public: internet-routable
  • Private: internal only (RFC 1918)

📚 Private Ranges (RFC 1918)

  • 10.0.0.0/8 — 16.7M addresses
  • 172.16.0.0/12 — 1M addresses
  • 192.168.0.0/16 — 65K addresses
Quick review of IP addressing. In AWS, every EC2 instance gets a private IP. Public IPs are optional and assigned from AWS-owned pools. Private ranges from RFC 1918 are commonly used for VPC CIDR blocks.

CIDR Notation

CIDR BlockSubnet MaskIPsUse Case
10.0.0.0/16255.255.0.065,536Typical VPC
10.0.1.0/24255.255.255.0256Typical subnet
10.0.1.0/28255.255.255.24016Small subnet
0.0.0.0/00.0.0.0AllDefault route
⚠️ AWS reserves 5 IPs per subnet: network, router (.1), DNS (.2), future (.3), broadcast
CIDR notation is essential for VPC design. The /16 gives you 65,536 addresses for the entire VPC. Each subnet carves out a portion. Remember AWS reserves 5 IPs in every subnet, so a /24 gives you 251 usable addresses, not 256.

How Cloud Networking Differs

🏢 Traditional Data Center

  • Physical routers and switches
  • Hardware firewalls
  • Manual VLAN configuration
  • Weeks to provision

☁️ AWS Cloud

  • Software-defined networking
  • Virtual firewalls (SGs, NACLs)
  • API-driven configuration
  • Minutes to provision
AWS networking is entirely software-defined. No physical hardware to manage. Everything is API-driven and can be automated with CloudFormation or Terraform. This is a paradigm shift from traditional networking. Fixed capacity vs elastic scalability is another major differentiator.

🏗️ Amazon Virtual Private Cloud

Your isolated network in the AWS Cloud

Now we get into the core of this module: Amazon VPC. This is the foundational networking service upon which almost everything in AWS is built.

What Is Amazon VPC?

💡 Definition

A logically isolated virtual network where you launch AWS resources with full control over IP ranges, subnets, routing, and gateways.

  • Spans all AZs within a Region
  • Isolated from other VPCs by default
  • Primary CIDR: /16 to /28 (plan for /16)
  • Optional secondary CIDRs and IPv6
Plan carefully: Primary CIDR cannot change after creation. Avoid overlaps for peering/VPN.
Think of a VPC as your own private data center inside AWS. You define the network topology, control access, and decide what connects to the internet. A VPC is Region-scoped but spans all AZs. Common CIDR choices: 10.0.0.0/16, 10.1.0.0/16, 172.16.0.0/16. You can add up to 4 secondary CIDR blocks. AWS can assign a /56 IPv6 block from Amazon's pool.

Default VPC vs. Custom VPC

📦 Default VPC

  • Auto-created per Region
  • CIDR: 172.31.0.0/16
  • Public subnet per AZ, IGW attached
  • Instances get public IPs

🔧 Custom VPC

  • You create and configure
  • Choose your own CIDR block
  • Public and private subnets
  • No public IPs by default
Best practice: Use custom VPCs for production workloads. The default VPC is convenient for learning and testing but lacks the isolation and control needed for production. Every AWS account gets a default VPC in each Region. For real workloads, always create a custom VPC with intentional CIDR ranges, proper subnet design, and controlled internet access.

Subnets: Public vs. Private

🌎 Public Subnet

  • Route to Internet Gateway
  • Resources can have public IPs
  • For: ALBs, bastion hosts, NAT GWs

🔒 Private Subnet

  • No route to Internet Gateway
  • Private IPs only
  • For: App servers, databases
Multi-AZ Layout (10.0.0.0/16):
us-east-1a: Public 10.0.1.0/24 | Private 10.0.10.0/24
us-east-1b: Public 10.0.2.0/24 | Private 10.0.20.0/24
A subnet is public or private based on its route table, not a setting on the subnet itself. Each subnet resides in one AZ and cannot span AZs. Deploy across at least 2 AZs for high availability. Use a consistent IP scheme for readability (1/2/3 for public, 10/20/30 for private).

Internet Gateway & NAT Gateway

🌎 Internet Gateway (IGW)

  • VPC-to-internet connectivity
  • One per VPC, highly available
  • No bandwidth constraints
  • Route: 0.0.0.0/0 → igw-xxx

🔒 NAT Gateway

  • Outbound-only for private subnets
  • Lives in public subnet + Elastic IP
  • AZ-specific; one per AZ for HA
  • Route: 0.0.0.0/0 → nat-xxx
⚠️ NAT GW cost: ~$0.045/hr + ~$0.045/GB processed. Consider VPC endpoints for AWS service traffic.
The IGW is fully managed by AWS and inherently highly available. Simply attaching it isn't enough; you must also update the route table and ensure the instance has a public IP. NAT Gateway is the recommended way to give private instances outbound internet access (e.g., software updates). It sits in the public subnet and forwards traffic to the IGW.

VPC Architecture: Putting It Together

Region: us-east-1 — VPC: 10.0.0.0/16

┌───── AZ-a ─────┐  ┌───── AZ-b ─────┐
│ Public 10.0.1.0/24 │  │ Public 10.0.2.0/24 │
│ [NAT-GW] [ALB]     │  │ [NAT-GW] [ALB]     │
│────────────────────│  │────────────────────│
│ Private 10.0.10.0/24│  │ Private 10.0.20.0/24│
│ [EC2] [EC2]         │  │ [EC2] [EC2]         │
└────────────────────┘  └────────────────────┘
          ┌───[Internet Gateway]───┐
                  ▼ Internet ▼
This is the canonical VPC architecture. Public subnets hold load balancers and NAT Gateways. Private subnets hold application servers and databases. The Internet Gateway sits at the VPC boundary. Each AZ has its own NAT Gateway for resilience.

VPC Design Best Practices

  • Multi-AZ: Subnets in 2+ AZs for fault tolerance
  • Public/private separation: Expose only what is necessary
  • CIDR planning: Use /16, avoid overlaps for peering
  • VPC endpoints: Access S3, DynamoDB without internet
  • Flow Logs: Capture traffic for troubleshooting and audit
These best practices come directly from the AWS Well-Architected Framework. CIDR planning is especially important in enterprise environments where dozens or hundreds of VPCs may need to communicate. VPC Flow Logs are invaluable for security investigations.

🛣️ Amazon VPC Routing

Controlling traffic flow within and beyond your VPC

Route tables are the mechanism that controls where network traffic is directed. Every subnet must be associated with a route table. Understanding routing is essential for VPC troubleshooting.

Route Tables

  • Main route table: auto-created, default for all subnets
  • Custom route tables: explicitly associated with subnets
  • Most specific route wins (longest prefix match)
DestinationTargetPurpose
10.0.0.0/16localVPC internal (cannot remove)
0.0.0.0/0igw-xxxInternet via IGW (public)
0.0.0.0/0nat-xxxInternet via NAT GW (private)
10.1.0.0/16pcx-xxxVPC peering
Each route has a destination CIDR and a target. The local route is automatically present and cannot be deleted. Best practice: Do not add internet routes to the main route table; create custom route tables for public subnets. This ensures new subnets default to private. One subnet can only be associated with one route table, but many subnets can share the same route table. VPC endpoints (e.g., for S3) can also be added as route targets.

Traffic Flow & Route Associations

🌎 Public Subnet Flow

Instance → Router → IGW → Internet

10.0.0.0/16 → local
0.0.0.0/0   → igw-abc

🔒 Private Subnet Flow

Instance → Router → NAT GW → IGW

10.0.0.0/16 → local
0.0.0.0/0   → nat-def

Pattern: One route table per tier. Public subnets share IGW route table; private subnets share NAT GW route table.
Traffic between subnets within the same VPC always uses the local route. The VPC router is implicit at the .1 address of every subnet. Intra-VPC traffic never hits the IGW. If you have NAT Gateways in multiple AZs, you'd have one private route table per AZ.

🛡️ Amazon VPC Security

Defense in depth with security groups and NACLs

VPC security is a layered approach. Security groups protect at the instance level, Network ACLs protect at the subnet level. Together, they form a defense-in-depth strategy.

Security Groups

🔒 Instance-Level Firewall

Stateful virtual firewalls controlling traffic at the network interface level.

  • Stateful: return traffic automatically allowed
  • Default: all inbound denied, all outbound allowed
  • Allow rules only — no deny rules
  • All rules evaluated as a collective set
  • Up to 5 SGs per instance
Stateful is the key characteristic. If you allow inbound HTTP on port 80, the response traffic is automatically allowed out, even if the outbound rules don't explicitly permit it. This simplifies rule management compared to NACLs.

Security Group Rules: Example

Web Server Security Group:

DirectionProtocolPortSource/DestDescription
InboundTCP800.0.0.0/0HTTP
InboundTCP4430.0.0.0/0HTTPS
InboundTCP2210.0.0.0/16SSH from VPC
OutboundAllAll0.0.0.0/0All outbound
Use descriptive names and add descriptions to every rule. This is essential for auditing and troubleshooting. Note that SSH is restricted to the VPC CIDR, not open to the internet. In production, you might further restrict SSH to a bastion host's security group.

Security Group Chaining

🔗 Reference SGs Instead of IPs

Use another security group as source/destination for dynamic, self-maintaining rules.

App SG: Inbound TCP 8080 from sg-alb
DB SG: Inbound TCP 3306 from sg-app
When you add or remove instances from a security group, the referencing rules automatically apply to the new set of instances. No IP address management needed. If you scale out by adding more app servers to sg-app, the database SG automatically allows them. This is the recommended approach for multi-tier architectures.

Network ACLs (NACLs)

🛡️ Subnet-Level Firewall

Stateless firewalls controlling traffic entering and leaving a subnet.

  • Stateless: must allow both inbound and return traffic
  • Numbered rules: evaluated lowest first, first match wins
  • Supports allow and deny rules
  • Default NACL allows all; custom NACL denies all
The key difference from security groups: NACLs are stateless. If you allow inbound port 80, you must also allow the ephemeral port range outbound for the response. Rules are evaluated in numeric order and the first matching rule applies. The implicit deny at rule * catches everything else.

NACL Rules: Example

Inbound:

RulePortSourceAction
100800.0.0.0/0ALLOW
1104430.0.0.0/0ALLOW
1201024-655350.0.0.0/0ALLOW
*All0.0.0.0/0DENY

Outbound:

RulePortDestAction
100800.0.0.0/0ALLOW
1104430.0.0.0/0ALLOW
1201024-655350.0.0.0/0ALLOW
*All0.0.0.0/0DENY
Ephemeral ports (1024-65535) are required for return traffic on both sides because NACLs are stateless. Use increments of 10 or 100 for rule numbers to allow inserting rules later. The asterisk rule is the implicit deny that catches anything not matched.

Security Groups vs. Network ACLs

FeatureSecurity GroupNetwork ACL
ScopeInstance (ENI) levelSubnet level
StateStateful (auto return)Stateless (explicit return)
RulesAllow onlyAllow and Deny
EvaluationAll rules as a setNumbered order, first match
DefaultDeny in, allow outAllow all (default NACL)
This comparison is a very common exam and interview topic. The stateful vs stateless distinction is the most important differentiator. In practice, security groups handle most traffic filtering. NACLs provide an additional layer, often used to block known malicious IPs.

Defense in Depth & Best Practices

Internet → [NACL: Subnet boundary] → [Security Group: Instance boundary] → EC2
  • Least privilege: only allow required traffic
  • Use SG references instead of IPs where possible
  • Avoid 0.0.0.0/0 inbound unless genuinely needed
  • Enable VPC Flow Logs for traffic auditing
  • Audit with AWS Config to detect permissive rules
Use both security groups and NACLs together. Traffic first passes through the NACL at the subnet boundary, then through the security group at the instance level. Even if one layer is misconfigured, the other provides protection. Common mistake: Opening SSH (port 22) to 0.0.0.0/0. Use Session Manager or a bastion host instead. Security group sprawl is a real problem; AWS Config can flag overly permissive groups.
