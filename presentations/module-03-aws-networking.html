🌐 Networking in AWS

Software-defined networking fundamentals

AWS networking is entirely software-defined. No physical hardware to manage. Everything is API-driven and can be automated. This is a paradigm shift from traditional networking.

📅 Networking Basics

  • IP addresses: unique identifiers for devices
  • Public IPs: internet-routable
  • Private IPs: internal only (10.x, 172.16.x, 192.168.x)
  • CIDR notation: defines IP ranges (e.g. 10.0.0.0/16)
Quick review of IP addressing. In AWS, every EC2 instance gets a private IP automatically. Public IPs are optional and assigned from AWS-owned pools. CIDR notation defines address ranges: /16 gives 65,536 addresses (typical VPC), /24 gives 256 addresses (typical subnet). AWS reserves 5 IPs per subnet for internal use. You don't need to be a networking expert, but understanding these basics helps when designing VPC architectures.

🔒 Amazon VPC

Your isolated network in the cloud

VPC is the networking foundation for nearly everything in AWS. Let's understand what it is and how its components work together.

🌐 What Is Amazon VPC?

Virtual Private Cloud — your own isolated section of the AWS cloud.

  • Logically isolated from other customers
  • You define the IP address range (CIDR)
  • Spans all AZs in a Region
  • Full control over routing and security
Think of a VPC as your own private data center in the cloud. You define the network layout: IP ranges, subnets, route tables, and security rules. A VPC lives in one Region but spans all AZs in that Region. Each AWS account gets a default VPC per Region, but production workloads should use custom VPCs designed for your architecture. VPC CIDR blocks range from /16 (65,536 IPs) to /28 (16 IPs).

🗃 Subnets: Public vs Private

Public Subnet

  • Route to Internet Gateway
  • Resources get public IPs
  • Web servers, load balancers

Private Subnet

  • No direct internet route
  • Private IPs only
  • Databases, app servers
Subnets partition your VPC into segments, each in a single AZ. A subnet is "public" if its route table has a route to an Internet Gateway. A subnet is "private" if it doesn't. Best practice is to place internet-facing resources (load balancers, bastion hosts) in public subnets and everything else (app servers, databases) in private subnets. Always deploy subnets across at least 2 AZs for high availability.

🖧 Gateways

Internet Gateway (IGW)

  • Connects VPC to internet
  • Enables inbound + outbound
  • Horizontally scaled, redundant

NAT Gateway

  • Outbound-only internet access
  • For private subnet resources
  • Managed, scales automatically
The Internet Gateway is the front door to the internet for your VPC. It's free, highly available, and you attach one per VPC. The NAT Gateway lets private subnet resources reach the internet (for updates, API calls) without being directly reachable from the internet. NAT Gateways are managed by AWS and charged per hour plus per GB of data processed. Deploy NAT Gateways in public subnets, one per AZ for high availability.

🛣️ Route Tables

Rules that determine where network traffic is directed.

  • Every subnet has exactly one route table
  • Local route for VPC traffic (automatic)
  • Add routes to IGW, NAT, peering, etc.
  • Most specific route wins
Route tables are the traffic directors in your VPC. Every VPC gets a main route table with a local route that allows all subnets within the VPC to communicate. You create custom route tables to control traffic: public subnets get a 0.0.0.0/0 route to the Internet Gateway, private subnets get a 0.0.0.0/0 route to a NAT Gateway. The most specific matching route wins — so a /24 route takes precedence over a /16 route for matching traffic.

🔒 Network Security

Layered defense for your VPC

AWS provides two layers of network security: security groups at the instance level and NACLs at the subnet level. Let's look at each.

🛡️ Security Groups

Virtual firewall at the instance level.

  • Stateful: return traffic auto-allowed
  • Allow rules only (no deny rules)
  • Default: all outbound allowed, no inbound
  • Can reference other security groups
Security groups are the primary firewall in AWS. They're stateful — if you allow inbound HTTP on port 80, the response traffic is automatically allowed out. You can only create allow rules; anything not explicitly allowed is denied by default. A powerful feature is referencing other security groups as sources, e.g., "allow traffic from the web-tier security group" without specifying IPs. Each instance can have up to 5 security groups attached.

📛 Network ACLs

Firewall at the subnet level.

  • Stateless: must allow both directions
  • Allow and deny rules supported
  • Rules evaluated by number (lowest first)
  • Default NACL: allows all traffic
NACLs are evaluated before security groups for inbound traffic. They're stateless — you must create rules for both inbound and outbound traffic explicitly. Rules are numbered and evaluated in order — the first matching rule wins. The default NACL allows all traffic; custom NACLs deny all traffic by default. NACLs are useful for blocking specific IP ranges or adding a subnet-level defense layer. Most architectures rely primarily on security groups with NACLs as an additional safeguard.

🔒 SGs vs NACLs

Security Groups

  • Instance level
  • Stateful
  • Allow rules only
  • Primary defense layer

Network ACLs

  • Subnet level
  • Stateless
  • Allow + deny rules
  • Additional defense layer
This is one of the most common exam and interview questions. Security groups are your primary defense — stateful, instance-level, allow-only. NACLs are your secondary defense — stateless, subnet-level, support both allow and deny. Think of it as defense in depth: NACLs are the outer wall (subnet boundary), security groups are the inner wall (instance boundary). In practice, most teams configure security groups carefully and leave NACLs at defaults unless they need to explicitly block specific IPs or ranges.

📋 Module 3 Summary

  • VPC: your isolated cloud network
  • Subnets: public (internet) vs private (internal)
  • Route tables direct traffic flow
  • Security Groups + NACLs = layered defense
Recap: A VPC is your isolated network in AWS. Subnets partition the VPC across AZs. Internet Gateways and NAT Gateways control internet connectivity. Route tables direct traffic. Security groups (stateful, instance-level) and NACLs (stateless, subnet-level) provide layered security. These networking fundamentals underpin every AWS architecture.
