🌐 Networking in AWS

Foundational concepts for cloud networking

Before diving into VPC specifics, let's refresh core networking concepts and understand how they translate to the cloud.

Networking Basics Refresher

📅 IP Addresses

  • IPv4: 32-bit, e.g. 10.0.1.5
  • IPv6: 128-bit, e.g. 2001:db8::1
  • Public: routable on the internet
  • Private: internal use only (RFC 1918)

📚 Private IP Ranges (RFC 1918)

  • 10.0.0.0/8 — 16.7M addresses
  • 172.16.0.0/12 — 1M addresses
  • 192.168.0.0/16 — 65K addresses
Quick review of IP addressing. In AWS, every EC2 instance gets a private IP. Public IPs are optional and assigned from AWS-owned pools. Private ranges from RFC 1918 are commonly used for VPC CIDR blocks.

CIDR Notation

Classless Inter-Domain Routing defines IP address ranges using a prefix length.

CIDR BlockSubnet MaskAvailable IPsUse Case
10.0.0.0/16255.255.0.065,536Typical VPC
10.0.1.0/24255.255.255.0256Typical subnet
10.0.1.0/28255.255.255.24016Small subnet
0.0.0.0/00.0.0.0AllDefault route
⚠️ AWS reserves 5 IPs per subnet: network address, VPC router (.1), DNS server (.2), future use (.3), and broadcast address.
CIDR notation is essential for VPC design. The /16 gives you 65,536 addresses for the entire VPC. Each subnet carves out a portion. Remember AWS reserves 5 IPs in every subnet, so a /24 gives you 251 usable addresses, not 256.

How Cloud Networking Differs

🏢 Traditional Data Center

  • Physical routers and switches
  • Hardware firewalls
  • Manual VLAN configuration
  • Weeks to provision
  • Fixed capacity

☁️ AWS Cloud

  • Software-defined networking (SDN)
  • Virtual firewalls (SGs, NACLs)
  • API-driven configuration
  • Minutes to provision
  • Elastic and scalable
AWS networking is entirely software-defined. No physical hardware to manage. Everything is API-driven and can be automated with CloudFormation or Terraform. This is a paradigm shift from traditional networking.

🏗️ Amazon Virtual Private Cloud

Your isolated network in the AWS Cloud

Now we get into the core of this module: Amazon VPC. This is the foundational networking service upon which almost everything in AWS is built.

What Is Amazon VPC?

💡 Definition

A Virtual Private Cloud (VPC) is a logically isolated section of the AWS Cloud where you launch AWS resources in a virtual network that you define.

  • You have complete control over your virtual networking environment
  • Select your own IP address range, create subnets, configure route tables and gateways
  • A VPC spans all Availability Zones within a Region
  • Resources in a VPC are isolated from other VPCs by default
Think of a VPC as your own private data center inside AWS. You define the network topology, control access, and decide what connects to the internet. A VPC is Region-scoped but spans all AZs in that Region.

Default VPC vs. Custom VPC

📦 Default VPC

  • Created automatically in each Region
  • CIDR: 172.31.0.0/16
  • One public subnet per AZ
  • Internet Gateway attached
  • Instances get public IPs by default

🔧 Custom VPC

  • You create and configure it
  • Choose your own CIDR block
  • Public and private subnets
  • Gateways added as needed
  • No public IPs by default
Best practice: Use custom VPCs for production workloads. The default VPC is convenient for learning and testing but lacks the isolation and control needed for production. Every AWS account gets a default VPC in each Region. It's designed for easy onboarding. For real workloads, always create a custom VPC with intentional CIDR ranges, proper subnet design, and controlled internet access.

VPC CIDR Blocks

  • Primary CIDR: Required when creating a VPC (/16 to /28)
  • Secondary CIDRs: Up to 4 additional blocks can be associated
  • IPv6: Optional — AWS assigns a /56 block from Amazon's pool
Recommended CIDR range: Use a /16 block (65,536 addresses) for flexibility. Common choices: 10.0.0.0/16, 10.1.0.0/16, 172.16.0.0/16
Plan carefully: You cannot change the primary CIDR block after VPC creation. Avoid overlapping CIDRs if you plan to use VPC peering or VPN connections. CIDR planning is one of the most important early decisions. Overlapping CIDRs between VPCs or with on-premises networks will prevent peering and VPN connectivity. A /16 gives you room for many subnets. Think ahead.

Subnets: Public vs. Private

🌎 Public Subnet

  • Route table has a route to an Internet Gateway
  • Resources can have public IPs
  • Directly reachable from the internet
  • Use for: Load balancers, bastion hosts, NAT Gateways

🔒 Private Subnet

  • No route to an Internet Gateway
  • Resources only have private IPs
  • Not directly reachable from the internet
  • Use for: Application servers, databases, internal services
Key insight: A subnet is public or private based on its route table, not a setting on the subnet itself.
This distinction is critically important. A subnet becomes public only because its route table has a 0.0.0.0/0 route pointing to an Internet Gateway. There is no "public" checkbox on the subnet. It's all about routing.

Subnets and Availability Zones

Example: VPC 10.0.0.0/16 in us-east-1

us-east-1a: Public 10.0.1.0/24 | Private 10.0.10.0/24
us-east-1b: Public 10.0.2.0/24 | Private 10.0.20.0/24
us-east-1c: Public 10.0.3.0/24 | Private 10.0.30.0/24
Best practice: Deploy subnets across at least 2 AZs for high availability. Use a consistent IP addressing scheme for readability.
Key rule: Each subnet resides entirely within one Availability Zone and cannot span AZs. A VPC spans all AZs in the Region. This is the standard multi-AZ pattern. By placing subnets in multiple AZs, your application can survive the failure of an entire AZ. The naming convention shown here (1/2/3 for public, 10/20/30 for private) makes it easy to identify subnet purpose at a glance.

Internet Gateway (IGW)

A highly available VPC component that enables communication between your VPC and the internet.

  • One IGW per VPC — attached at the VPC level
  • Performs NAT for instances with public IPv4 addresses
  • No bandwidth constraints or availability risks
  • Must be referenced in a route table to be useful
Two steps to enable internet access: (1) Attach an IGW to the VPC. (2) Add a route in the subnet's route table: 0.0.0.0/0 → igw-xxxxxxxx
The IGW is fully managed by AWS. It doesn't impose bandwidth limits and is inherently highly available. But simply attaching it isn't enough. You must also update the route table and ensure the instance has a public IP or Elastic IP.

NAT Gateway

Enables private subnet instances to reach the internet while blocking unsolicited inbound connections.

  • Deployed in a public subnet with an Elastic IP
  • Private subnet route table points 0.0.0.0/0 → nat-xxxxxxxx
  • Managed service — scales automatically up to 100 Gbps
  • AZ-specific — deploy one per AZ for high availability
Cost note: NAT Gateway charges per hour (~$0.045/hr) plus per GB processed (~$0.045/GB). For cost-sensitive workloads, consider NAT instances or VPC endpoints.
NAT Gateway is the recommended way to give private instances outbound internet access (e.g., for software updates). It sits in the public subnet and forwards traffic from the private subnet to the IGW. Remember: it's per-AZ, so for HA you need one in each AZ.

VPC Architecture: Putting It Together

Region: us-east-1 — VPC: 10.0.0.0/16

┌───── AZ-a ─────┐  ┌───── AZ-b ─────┐
│ Public 10.0.1.0/24 │  │ Public 10.0.2.0/24 │
│ [NAT-GW] [ALB]     │  │ [NAT-GW] [ALB]     │
│────────────────────│  │────────────────────│
│ Private 10.0.10.0/24│  │ Private 10.0.20.0/24│
│ [EC2] [EC2]         │  │ [EC2] [EC2]         │
└────────────────────┘  └────────────────────┘
          ┌───[Internet Gateway]───┐
                  ▼ Internet ▼
This is the canonical VPC architecture. Public subnets hold load balancers and NAT Gateways. Private subnets hold application servers and databases. The Internet Gateway sits at the VPC boundary. Each AZ has its own NAT Gateway for resilience.

VPC Design Best Practices

  • Multi-AZ deployment: Place subnets in at least 2 AZs for fault tolerance
  • Public/private separation: Keep workloads in private subnets; expose only what is necessary
  • CIDR planning: Use a /16 VPC CIDR, allocate subnets systematically, and avoid overlaps for peering
  • Use VPC endpoints: Access AWS services (S3, DynamoDB) without traversing the internet
  • Enable VPC Flow Logs: Capture IP traffic information for troubleshooting and auditing
These best practices come directly from the AWS Well-Architected Framework. CIDR planning is especially important in enterprise environments where dozens or hundreds of VPCs may need to communicate. VPC Flow Logs are invaluable for security investigations.

🛣️ Amazon VPC Routing

Controlling traffic flow within and beyond your VPC

Route tables are the mechanism that controls where network traffic is directed. Every subnet must be associated with a route table. Understanding routing is essential for VPC troubleshooting.

Route Tables

💡 Definition

A route table contains a set of rules (routes) that determine where network traffic from your subnets or gateway is directed.

  • Main route table: Automatically created with the VPC; used by default for all subnets
  • Custom route tables: You create and explicitly associate them with subnets
  • A subnet can only be associated with one route table at a time
  • Multiple subnets can share the same route table
Best practice: Do not add internet routes to the main route table. Create custom route tables for public subnets instead. This ensures new subnets default to private. The main route table is a safety net. If you don't explicitly associate a subnet with a custom route table, it uses the main route table. By keeping the main route table private, any new subnet you create is private by default.

Route Table Entries

Each route specifies a destination (CIDR block) and a target (where to send the traffic).

DestinationTargetPurpose
10.0.0.0/16localVPC internal traffic (cannot be removed)
0.0.0.0/0igw-xxxxxxxxInternet access via IGW (public subnet)
0.0.0.0/0nat-xxxxxxxxInternet access via NAT GW (private subnet)
10.1.0.0/16pcx-xxxxxxxxVPC peering connection
pl-xxxxxxxxvpce-xxxxxxxxVPC endpoint (e.g., S3)
The local route is automatically present and cannot be deleted. It ensures all traffic within the VPC CIDR can communicate. The most specific route wins when there are overlapping destinations. For example, a /24 route takes precedence over /16.

How Traffic Flows Through a VPC

🔍 Route Evaluation

When an instance sends traffic, the VPC router evaluates the route table associated with the instance's subnet. The most specific route (longest prefix match) wins.

🌎 Public Subnet Traffic Flow

Instance → VPC Router → Route Table → IGW → Internet

🔒 Private Subnet Traffic Flow

Instance → VPC Router → Route Table → NAT GW → IGW → Internet

Remember: Traffic between subnets within the same VPC always uses the local route and does not leave the VPC. The VPC router is an implicit component — you don't see or manage it, but it exists at the .1 address of every subnet. All traffic decisions are made by consulting the route table. Intra-VPC traffic never hits the IGW, even between public subnets.

Subnet-Route Table Associations

Public Route Table (rtb-pub)
10.0.0.0/16 → local
0.0.0.0/0   → igw-abc123
↳ Associated: subnet-pub-a, subnet-pub-b

Private Route Table (rtb-priv)
10.0.0.0/16 → local
0.0.0.0/0   → nat-def456
↳ Associated: subnet-priv-a, subnet-priv-b
Pattern: One route table per subnet tier. Public subnets share one route table (with IGW route). Private subnets share another (with NAT GW route).
This is the standard pattern. You typically have two route tables: one for public subnets pointing to the IGW, and one for private subnets pointing to a NAT Gateway. If you have NAT Gateways in multiple AZs, you'd have one private route table per AZ.

🛡️ Amazon VPC Security

Implementing defense in depth with security groups and NACLs

VPC security is a layered approach. Security groups protect at the instance level, Network ACLs protect at the subnet level. Together, they form a defense-in-depth strategy.

Security Groups

🔒 Instance-Level Firewall

Security groups act as stateful virtual firewalls that control inbound and outbound traffic at the network interface level.

  • Stateful: Return traffic is automatically allowed regardless of outbound rules
  • Default behavior: All inbound denied, all outbound allowed
  • Allow rules only — you cannot create deny rules
  • Evaluated as a collective set (all rules are considered)
  • An instance can have multiple security groups (up to 5)
Stateful is the key characteristic. If you allow inbound HTTP on port 80, the response traffic is automatically allowed out, even if the outbound rules don't explicitly permit it. This simplifies rule management compared to NACLs.

Security Group Rules: Example

Web Server Security Group:

DirectionProtocolPortSource / DestinationDescription
InboundTCP800.0.0.0/0HTTP from anywhere
InboundTCP4430.0.0.0/0HTTPS from anywhere
InboundTCP2210.0.0.0/16SSH from VPC only
OutboundAllAll0.0.0.0/0Allow all outbound
Tip: Use descriptive names and add descriptions to every rule. This is essential for auditing and troubleshooting in environments with many security groups. This is a typical web server security group. Note that SSH is restricted to the VPC CIDR, not open to the internet. In production, you might further restrict SSH to a bastion host's security group. The default outbound rule allows all traffic out.

Security Group Chaining

🔗 Referencing Other Security Groups

Instead of specifying IP addresses, you can reference another security group as the source or destination. This creates a dynamic, self-maintaining rule set.

App Server SG (sg-app):
Inbound: TCP 8080 from sg-alb (ALB security group)

Database SG (sg-db):
Inbound: TCP 3306 from sg-app (App server security group)
Advantage: When you add or remove instances from a security group, the referencing rules automatically apply to the new set of instances. No IP address management needed. Security group chaining is a powerful pattern. The ALB can only talk to the app tier on port 8080, and the app tier can only talk to the database on port 3306. If you scale out by adding more app servers to sg-app, the database SG automatically allows them. This is the recommended approach for multi-tier architectures.

Network ACLs (NACLs)

🛡️ Subnet-Level Firewall

Network ACLs are stateless firewalls that control traffic entering and leaving a subnet.

  • Stateless: You must explicitly allow both inbound and outbound traffic (including return traffic)
  • Numbered rules: Evaluated in order, lowest number first
  • Supports allow and deny rules
  • Default NACL: allows all inbound and outbound traffic
  • Custom NACL: denies all traffic by default until rules are added
The key difference from security groups: NACLs are stateless. If you allow inbound port 80, you must also allow the ephemeral port range outbound for the response. Rules are evaluated in numeric order and the first matching rule applies. The implicit deny at rule * catches everything else.

NACL Inbound Rules

Public Subnet NACL — Inbound:

Rule #ProtocolPort RangeSourceAction
100TCP800.0.0.0/0ALLOW
110TCP4430.0.0.0/0ALLOW
120TCP1024-655350.0.0.0/0ALLOW
*AllAll0.0.0.0/0DENY
Rule 120 allows ephemeral ports (1024-65535), which are required for return traffic. Without this rule, responses to allowed inbound requests will be blocked. Because NACLs are stateless, you must explicitly allow return traffic. The asterisk rule is the implicit deny that catches anything not matched by a numbered rule. Use increments of 10 or 100 for rule numbers to allow inserting rules later.

NACL Outbound Rules

Public Subnet NACL — Outbound:

Rule #ProtocolPort RangeDestinationAction
100TCP800.0.0.0/0ALLOW
110TCP4430.0.0.0/0ALLOW
120TCP1024-655350.0.0.0/0ALLOW
*AllAll0.0.0.0/0DENY
Ephemeral ports (1024-65535) are required on the outbound side as well for return traffic from inbound connections. The outbound rules mirror the inbound rules in this example, allowing HTTP, HTTPS, and ephemeral port responses.

Security Groups vs. Network ACLs

FeatureSecurity GroupNetwork ACL
Scope & DefaultInstance (ENI) level; deny in, allow outSubnet level; allow all (default NACL)
StateStateful (return traffic automatic)Stateless (return traffic must be allowed)
RulesAllow onlyAllow and Deny
EvaluationAll rules evaluated as a setRules in numeric order, first match wins
AssignmentAttached to ENIs / instancesAssociated with subnets
This comparison is a very common exam and interview topic. The stateful vs stateless distinction is the most important differentiator. In practice, security groups handle most traffic filtering. NACLs provide an additional layer, often used to block known malicious IPs or add subnet-level deny rules that security groups cannot provide.

Defense in Depth

Internet → [NACL: Subnet boundary] → [Security Group: Instance boundary] → EC2 Instance

🔒 Security Groups For

  • Fine-grained instance-level control
  • Application-tier segmentation
  • SG chaining between tiers

🛡️ NACLs For

  • Broad subnet-level policies
  • Blocking known malicious IPs
  • Explicit deny rules
Layered security model: Use both security groups and NACLs together to create multiple layers of network security. Defense in depth means no single point of failure in your security model. Traffic first passes through the NACL at the subnet boundary, then through the security group at the instance level. Even if one layer is misconfigured, the other provides protection. AWS recommends using both, though security groups typically carry the majority of the filtering logic.

VPC Security Best Practices

  • Principle of least privilege: Only allow traffic that is explicitly required
  • Use security group references instead of IP addresses where possible
  • Avoid 0.0.0.0/0 on inbound rules unless genuinely needed (e.g., public-facing ALB)
  • Enable VPC Flow Logs to CloudWatch Logs or S3 for traffic auditing
  • Audit regularly: Use AWS Config rules to detect overly permissive or stale security group entries
Common mistake: Opening SSH (port 22) or RDP (port 3389) to 0.0.0.0/0. Use Session Manager, a bastion host, or restrict to known IP ranges. Security group sprawl is a real problem in large environments. Over time, rules accumulate and become difficult to audit. AWS Config can flag security groups with 0.0.0.0/0 on sensitive ports. Session Manager is the modern alternative to SSH bastion hosts, as it requires no inbound ports at all.
