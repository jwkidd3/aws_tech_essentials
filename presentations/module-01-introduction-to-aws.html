☁️ Introduction to AWS Cloud

What is cloud computing and why does it matter?

Let's start with the fundamentals. Even if participants have experience with AWS, it's worth aligning on definitions and understanding the value proposition clearly.

🖥️ What Is Cloud Computing?

On-demand delivery of IT resources over the internet with pay-as-you-go pricing.

  • No physical data centers to manage
  • Provision resources in minutes
  • Scale up or down on demand
  • Pay only for what you consume
Stress the shift from CapEx to OpEx. In traditional environments, you had to guess capacity needs 3-6 months in advance. Cloud computing flips this: you provision what you need, when you need it, and pay only for what you consume. This is a fundamental shift in how IT operates. The NIST definition includes five characteristics: on-demand self-service, broad network access, resource pooling, rapid elasticity, and measured service.

🎯 Six Advantages of Cloud

  1. Trade CapEx for variable expense
  2. Massive economies of scale
  3. Stop guessing capacity
  1. Increase speed and agility
  2. Stop running data centers
  3. Go global in minutes
Walk through each advantage. 1) CapEx to OpEx shift is fundamental. 2) AWS has reduced pricing 130+ times since 2006 thanks to aggregated usage. 3) Eliminates procurement lead times and idle hardware. 4) Prototype in an afternoon vs. weeks of hardware setup. 5) AWS handles racking, stacking, power, cooling, physical security. 6) Deploy worldwide for lower latency and better user experience.

🌎 AWS Global Infrastructure

🌏 Regions

  • 33+ geographic locations
  • Physically isolated
  • Data stays in-Region

🏧 Availability Zones

  • 105+ AZs globally
  • One or more data centers
  • Low-latency interconnects

⚡ Edge Locations

  • 600+ Points of Presence
  • CloudFront CDN caching
  • Route 53 DNS hosting
Draw the hierarchy: Region contains multiple AZs, each AZ contains one or more data centers. Regions are completely independent. AZs within a Region are connected by dedicated high-bandwidth, low-latency networking. Edge Locations are separate from Regions and serve cached content to reduce latency for end users. Data sovereignty is Region-level: your data stays in the Region you put it in unless you explicitly replicate it elsewhere. When choosing a Region, consider compliance, latency, service availability, and pricing.

🛠️ Interacting with AWS

💻 Console

  • Web-based GUI
  • Great for exploration

⌨️ CLI

  • Command-line scripting
  • Automate tasks

🔌 SDKs

  • Language-specific APIs
  • Integrate into apps
Everything in AWS is an API call. The Console is a web interface that makes API calls on your behalf. The CLI lets you script and automate those same calls from the command line. SDKs provide language bindings for Python (boto3), JavaScript, Java, Go, .NET, and more. Infrastructure as Code tools like CloudFormation and Terraform also use these APIs under the hood. All three methods use the same underlying APIs and IAM permissions.

🔒 Security in the AWS Cloud

Shared responsibility and identity management

Security is the top priority at AWS. This section covers the shared responsibility model and IAM, which together form the foundation of cloud security.

🔒 Shared Responsibility Model

AWS Responsibility

"Security OF the cloud"

  • Physical data centers
  • Hardware and networking
  • Hypervisor and host OS

Your Responsibility

"Security IN the cloud"

  • Data and encryption
  • IAM and access control
  • OS patches and firewalls
The shared responsibility model is crucial. AWS secures the underlying infrastructure (physical security, hypervisor, global network). Customers are responsible for what they put IN the cloud: data, identity management, application code, OS patching, and firewall rules. The responsibility boundary shifts depending on the service type — with Lambda (serverless), AWS manages more; with EC2, the customer manages more.

👤 AWS Identity and Access Management

IAM controls who can access what in your AWS account.

  • Global service (not Region-specific)
  • Free to use, no additional charges
  • Principle of least privilege
  • Supports MFA for extra security
IAM is the first service you should understand deeply. It is global — IAM users, roles, and policies work across all Regions. The principle of least privilege means granting only the permissions needed to perform a task, nothing more. MFA adds a second factor to authentication, strongly recommended for all human users, especially the root account. IAM is free — you're never charged for creating users, roles, or policies.

👥 Users, Roles, and Policies

👤 Users

  • Represent a person or app
  • Long-term credentials
  • Console or programmatic

🎭 Roles

  • Temporary credentials
  • Assumed by services or users
  • Preferred for EC2 apps

📜 Policies

  • JSON permission documents
  • Attach to users or roles
  • Allow or deny actions
Users are for people or applications that need long-term credentials. Roles provide temporary credentials and are preferred — especially for EC2 instances, Lambda functions, and cross-account access. Policies are JSON documents that define permissions (which actions on which resources). Best practice: use groups to assign policies to users, and prefer roles over long-term access keys. An explicit deny in any policy always overrides an allow.
