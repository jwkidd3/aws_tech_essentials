☁️ Introduction to AWS Cloud

What is cloud computing and why does it matter?

Let's start with the fundamentals. Even if participants have experience with AWS, it's worth aligning on definitions and understanding the value proposition clearly.

🖥️ What Is Cloud Computing?

NIST Definition

The on-demand delivery of IT resources over the internet with pay-as-you-go pricing. Instead of buying, owning, and maintaining physical data centers and servers, you access technology services on an as-needed basis from a cloud provider.

Traditional Model

  • Capital expenditure upfront
  • Capacity planning months ahead
  • Hardware procurement cycles
  • Underutilized or over-provisioned

Cloud Model

  • Operational expenditure (OpEx)
  • Provision in minutes
  • API-driven resource creation
  • Scale up or down on demand
Stress the shift from CapEx to OpEx. In traditional environments, you had to guess capacity needs 3-6 months in advance. Cloud computing flips this: you provision what you need, when you need it, and pay only for what you consume. This is a fundamental shift in how IT operates.

🎯 Six Advantages of Cloud Computing (1/2)

1. Trade Capital Expense for Variable Expense

Instead of investing in data centers and servers before you know how you'll use them, pay only when you consume computing resources — and only for how much you consume.

2. Benefit from Massive Economies of Scale

AWS aggregates usage from hundreds of thousands of customers, achieving higher economies of scale. This translates into lower pay-as-you-go prices — AWS has reduced pricing over 130 times since 2006.

3. Stop Guessing Capacity

Eliminate guessing on infrastructure capacity needs. Scale up in minutes when demand spikes, scale down when it recedes. Auto Scaling policies can handle this automatically.

These first three advantages address the financial and operational pain points. Emphasize that AWS passes savings to customers through frequent price reductions. The capacity point resonates especially with anyone who has dealt with procurement lead times or unused hardware sitting in racks.

🎯 Six Advantages of Cloud Computing (2/2)

4. Increase Speed and Agility

New IT resources are a click or API call away. Reduce time to make resources available from weeks to minutes. This dramatically increases agility — the cost and time to experiment is significantly lower.

5. Stop Spending Money Running Data Centers

Focus on projects that differentiate your business, not the infrastructure. Racking, stacking, power, cooling, and physical security are AWS's responsibility — not yours.

6. Go Global in Minutes

Deploy your application in multiple AWS Regions around the world with a few clicks. Provide lower latency and a better experience for customers at minimal cost.

Speed and agility is the biggest differentiator for many engineering teams. The ability to spin up a prototype in an afternoon rather than waiting weeks for hardware changes the culture of innovation. Going global matters for any organization serving international users.

🌎 AWS Global Infrastructure

Key Concept: AWS infrastructure is organized into Regions, Availability Zones (AZs), and Edge Locations. Understanding this hierarchy is essential for designing resilient architectures.

🌏 Regions

  • 33+ geographic regions worldwide
  • Physically isolated from each other
  • Data does not replicate across Regions unless you explicitly configure it

🏧 Availability Zones

  • 105+ AZs globally
  • One or more discrete data centers per AZ
  • Connected via low-latency, high-throughput networking

⚡ Edge Locations

  • 600+ Points of Presence
  • Used by CloudFront (CDN) and Route 53 (DNS)
  • Cache content closer to end users
Draw the hierarchy: Region contains multiple AZs, each AZ contains one or more data centers. Regions are completely independent. AZs within a Region are connected by dedicated high-bandwidth, low-latency networking. Edge Locations are separate from Regions and serve cached content to reduce latency for end users. Emphasize that data sovereignty is Region-level: your data stays in the Region you put it in.

🏧 Regions & Availability Zones

Region Structure

Each Region consists of a minimum of 3 isolated Availability Zones. AZs are physically separated by a meaningful distance (many kilometers) but within 100 km of each other.

Design Principle: Always deploy across multiple AZs for high availability. A single AZ failure should not bring down your application.
Region NameCodeAZs
US East (N. Virginia)us-east-16
US West (Oregon)us-west-24
EU (Ireland)eu-west-13

us-east-1 is the largest and oldest Region with 6 AZs. It's also where many global AWS services have their control planes. When you see 'global' services like IAM, their API endpoints typically route through us-east-1. Each AZ has independent power, cooling, and physical security. The fiber connections between AZs provide single-digit millisecond latency.

🧮 Choosing an AWS Region

Four key factors to evaluate when selecting a Region:

📜 1. Compliance

Data governance and legal requirements may dictate where data must reside. GDPR, HIPAA, and local data residency laws can be the overriding factor.

⏱️ 2. Latency

Choose Regions close to your user base. A 200ms round-trip vs 20ms has a measurable impact on user experience and application performance.

🔧 3. Service Availability

Not all AWS services are available in every Region. New services typically launch in us-east-1 first. Verify your required services are available in your target Region.

💰 4. Cost

Pricing varies by Region. US Regions are typically the least expensive. São Paulo (sa-east-1) can be 40-50% more expensive than us-east-1 for the same resources.

Walk through the decision process in order. Compliance is typically non-negotiable and comes first. If you have no compliance constraints, proximity to users is usually the next priority. Then check that the services you need are available. Finally, compare costs. Remind participants they can use multiple Regions for different workloads.

💻 Interacting with AWS

Three primary ways to interact with AWS services — all of them use the AWS API under the hood:

🌐 AWS Management Console

  • Web-based GUI
  • Great for exploration and learning
  • Point-and-click interface
  • Not ideal for automation

🖱️ AWS CLI

  • Command-line interface
  • Scriptable and automatable
  • Available for Windows, macOS, Linux
  • Ideal for repeatable tasks

🛠️ AWS SDKs

  • Language-specific APIs
  • Python (Boto3), JavaScript, Java, .NET, Go, etc.
  • Embed AWS calls in your applications
  • Used for application integration
Key Insight: Everything in AWS is an API call. The Console, CLI, and SDKs are all just different interfaces to the same underlying REST APIs. This means anything you can do in the Console can be automated via CLI or SDK.
Stress that the API-first nature of AWS is fundamental. When you click a button in the Console, it's making an API call. The CLI wraps those same API calls in shell commands. SDKs wrap them in language-native code. This is what makes Infrastructure as Code possible. In practice, teams use the Console for exploration and debugging, CLI for scripting and quick operations, and SDKs for application-level integration.

🖱️ AWS CLI in Action

Common CLI patterns you'll use throughout this course:

# List S3 buckets
aws s3 ls

# Describe EC2 instances
aws ec2 describe-instances \
  --region us-east-1

# Create a DynamoDB table
aws dynamodb create-table \
  --table-name Employees \
  --attribute-definitions \
    AttributeName=id,AttributeType=S \

    AttributeName=id,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST

CLI Structure:
aws <service> <action> [options]
Pro Tip: Use --output table for human-readable output, --output json for scripting, and --query for JMESPath filtering.
Authentication: The CLI uses credentials configured via aws configure or environment variables (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY).
Walk through the CLI syntax. The pattern is always: aws, then the service name, then the action, then options. The sts get-caller-identity command is extremely useful for debugging authentication issues. It tells you exactly who AWS thinks you are. Mention that in the labs, they'll be using both the Console and CLI.

🔒 Security in the AWS Cloud

Shared Responsibility, IAM, and Access Control

Security is job zero at AWS, and it should be for your organization too. This section covers the shared responsibility model and IAM, which is arguably the most important service in all of AWS. You cannot do anything in AWS without IAM being involved.

🛡️ AWS Shared Responsibility Model

Security OF the Cloud vs. Security IN the Cloud

AWS and the customer share security responsibilities. Understanding the boundary is critical for compliance and proper security posture.

✅ AWS Responsibility (Security OF the Cloud)

  • Physical security of data centers
  • Hardware and infrastructure
  • Managed services' underlying OS/patching
  • Network infrastructure
  • Hypervisor layer (Nitro)
  • Global infrastructure (Regions, AZs, Edge)

⚠️ Customer Responsibility (Security IN the Cloud)

  • Data encryption (at rest and in transit)
  • IAM — identity and access management
  • Operating system patching (EC2)
  • Network & firewall configuration
  • Application-level security
  • Client-side data protection
This model is tested heavily on AWS certifications and is essential for real-world operations. The key insight: the responsibility line shifts depending on the service. With EC2, you're responsible for the OS and everything above it. With Lambda, AWS manages much more, and your responsibility shrinks to code and IAM. Use the phrase: 'If you can touch it in the Console, you're probably responsible for configuring it securely.'

🔄 Responsibility Shifts by Service Type

LayerEC2 (IaaS)RDS (Managed)Lambda (Serverless)
Application CodeCustomerCustomerCustomer
Data EncryptionCustomerCustomerCustomer
IAM / Access ControlCustomerCustomerCustomer
OS PatchingCustomerAWSAWS
Database EngineCustomerAWSAWS
Hardware / InfrastructureAWSAWSAWS
Takeaway: The more managed the service, the less operational responsibility you carry — but you always own IAM policies, data classification, and encryption decisions.
Walk through the table column by column. With EC2, you own almost everything above the hypervisor. With RDS, AWS handles the database engine, patching, backups (if configured), and OS. With Lambda, AWS handles nearly everything except your code and IAM configuration. Notice that IAM and data encryption are ALWAYS the customer's responsibility regardless of service type. This is a critical point.

👤 AWS Identity and Access Management (IAM)

What Is IAM?

IAM is a global service (not Region-specific) that controls authentication (who can sign in) and authorization (what permissions they have) for AWS resources.

Core IAM Components

  • Users — Individual identities with long-term credentials
  • Groups — Collections of users with shared permissions
  • Roles — Temporary credentials for services or federated users
  • Policies — JSON documents defining permissions
IAM is free. There is no charge for IAM users, groups, roles, or policies. You only pay for the AWS resources those identities access.
Global Service: IAM users, groups, and roles are available across all Regions. You create them once and they work everywhere.
IAM is the first service every AWS practitioner must understand. It underpins every single API call. When we say 'global service,' that means an IAM user created in any context can access resources in any Region (if their policies allow it). Distinguish between authentication (proving identity) and authorization (what you're allowed to do). IAM handles both.

👥 IAM Users & Groups

IAM Users

  • Represent a person or service
  • Console access: username + password
  • Programmatic access: access key ID + secret access key
  • Can belong to multiple groups
  • Can have policies attached directly (inline) or inherited from groups

IAM Groups

  • Logical collection of IAM users
  • Attach policies to the group, not individual users
  • Users inherit all policies from their group(s)
  • Groups cannot be nested (no groups within groups)
  • A user can belong to up to 10 groups
Best Practice: Always assign permissions to groups, then add users to those groups. Avoid attaching policies directly to individual users — this does not scale and becomes unmanageable.
Emphasize the group-based approach. In a real organization, you might have groups like 'Developers', 'DBAdmins', 'ReadOnlyAuditors'. When a new developer joins, you add them to the Developers group and they immediately get the right permissions. When they leave, you remove them. This is much cleaner than managing per-user policies. Note the 10-group limit per user, which is rarely hit in practice.

🎭 IAM Roles

Roles Provide Temporary Security Credentials

Unlike IAM users, roles do not have long-term credentials. Instead, when a role is assumed, AWS STS (Security Token Service) provides temporary credentials that expire automatically.

Common Role Use Cases

  • EC2 Instance Role: Grant your application on EC2 access to S3 or DynamoDB without embedding credentials
  • Cross-Account Access: Allow users from Account A to access resources in Account B
  • Federated Access: Allow corporate directory users to access AWS via SAML/OIDC
  • Service Roles: Allow Lambda, ECS, or other services to call AWS APIs on your behalf
Best Practice: Use IAM roles instead of access keys whenever possible. Roles provide temporary credentials that rotate automatically.
Trust Policy: Every role has a trust policy that defines who can assume the role (the principal).
Roles are the preferred mechanism for granting permissions in modern AWS architectures. The key benefit is that temporary credentials expire, reducing the blast radius of a compromise. For the employee directory application we'll build, we'll attach an IAM role to our EC2 instance so the application code can access DynamoDB and S3 without hardcoded credentials. This is a critical best practice that many organizations get wrong early on.

📜 IAM Policy Structure

IAM policies are JSON documents that define permissions. Every policy contains one or more statements:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3ReadAccess",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ]
    }
  ]
}

Statement Elements

ElementPurpose
EffectAllow or Deny
ActionWhich API operations (e.g., s3:GetObject)
ResourceWhich AWS resource(s) via ARN
ConditionOptional constraints (IP, time, MFA, etc.)
PrincipalWho (used in resource-based policies)
Default Deny: If a policy doesn't explicitly allow an action, it is denied. An explicit Deny always overrides any Allow.
Walk through the JSON policy line by line. Version is always 2012-10-17 (this is the policy language version, not a date you set). Sid is optional but recommended for readability. Effect is either Allow or Deny. Action uses the format service:APIOperation. Resource uses ARNs to specify exactly which resources the policy applies to. Note the two resource ARNs in the example: one for the bucket itself (for ListBucket) and one with /* for objects inside the bucket (for GetObject). This is a common pattern. Stress that explicit Deny always wins.

📄 Types of IAM Policies

AWS Managed Policies

  • Created and maintained by AWS
  • Cover common use cases
  • Examples: AmazonS3ReadOnlyAccess, AdministratorAccess
  • Good starting point but often too broad

Customer Managed Policies

  • Created by you in your account
  • Tailored to your specific needs
  • Reusable across multiple identities
  • Versioned (up to 5 versions)

Inline Policies

  • Embedded directly in a single user, group, or role
  • Not reusable
  • Deleted when the identity is deleted
  • Use sparingly — for strict 1:1 relationships
Recommendation: Start with AWS managed policies for learning and prototyping. Transition to customer managed policies for production workloads with least-privilege scoping.
In practice, most organizations use a combination. AWS managed policies are convenient but tend to be overly permissive. For example, AmazonS3FullAccess grants access to ALL S3 buckets in the account. Customer managed policies let you scope down to specific buckets, prefixes, and operations. Inline policies are generally discouraged except for edge cases where a policy must be tightly coupled to a specific identity and should never be reused.

🔐 Root User vs. IAM Users

⚠️ Root User

  • Created when the AWS account is first set up
  • Has unrestricted access to all resources
  • Cannot be limited by IAM policies
  • Required for: closing the account, changing support plan, restoring IAM permissions

✅ IAM Users

  • Created by administrators
  • Permissions defined by attached policies
  • Can be restricted, monitored, and audited
  • Use for all day-to-day operations
Root User Best Practices:
  • Enable MFA on the root user immediately
  • Do NOT create access keys for the root user
  • Do NOT use the root user for day-to-day tasks
  • Create an IAM admin user for administrative work
  • Store root credentials in a secure vault
The root user is the most powerful identity in an AWS account and the most dangerous if compromised. There are very few tasks that require root access: closing the account, changing the support plan, enabling certain S3 configurations, and restoring IAM permissions if you've locked yourself out. Everything else should be done with IAM users or roles. This is not optional guidance — it's a fundamental security requirement.

🔐 Multi-Factor Authentication (MFA)

Add a Second Layer of Security

MFA requires users to provide a one-time code from an authentication device in addition to their username and password. Even if credentials are compromised, the attacker cannot access the account without the MFA device.

Virtual MFA

Software-based authenticator app (Google Authenticator, Authy, Microsoft Authenticator). Free and most commonly used.

Hardware MFA

Physical key fob or display card (e.g., Gemalto token). Provides a dedicated device independent of a smartphone.

FIDO2 Security Key

USB-based hardware key (e.g., YubiKey). Phishing-resistant and supports multiple accounts on one device.

Requirement: At minimum, enable MFA on the root user and all IAM users with console access. You can also enforce MFA in IAM policies using the aws:MultiFactorAuthPresent condition key.
MFA is the single most impactful security control you can enable. AWS recommends FIDO2 security keys as the strongest option because they're phishing-resistant. Virtual MFA is the most common in practice due to convenience. You can also require MFA for specific API operations through IAM policy conditions. For example, you might allow users to list resources freely but require MFA to delete anything.

🔒 Principle of Least Privilege

Grant Only the Permissions Required to Perform a Task

Start with zero permissions and add only what is needed. It's easier to grant additional permissions later than to revoke excessive permissions that are already in use.

❌ Anti-Pattern

{
  "Effect": "Allow",
  "Action": "*",
  "Resource": "*"
}
This grants full admin access to every service and resource. Never use this in production.

✅ Least Privilege

{
  "Effect": "Allow",
  "Action": [
    "dynamodb:GetItem",
    "dynamodb:PutItem",
    "dynamodb:Query"
  ],
  "Resource":
    "arn:aws:dynamodb:us-east-1:
     123456789012:table/Employees"
}
Scoped to specific actions on a specific table.
This is one of the most important security principles. Compare the two policies side by side. The wildcard policy is a ticking time bomb. If those credentials leak, the attacker has full access to everything. The scoped policy limits damage to three operations on a single DynamoDB table. AWS provides tools to help: IAM Access Analyzer can generate least-privilege policies based on observed usage. CloudTrail logs show which API calls an identity actually makes, helping you trim excess permissions.

🔧 Implementing Least Privilege

AWS provides several tools to help you right-size permissions:

🔎 IAM Access Analyzer

  • Analyzes resource policies for external access
  • Generates policies based on CloudTrail activity
  • Validates policies against best practices
  • Identifies unused permissions

📊 Credential Reports & Access Advisors

  • Credential Report: Account-level CSV of all users and credential status
  • Access Advisor: Shows service-level last-accessed data per identity
  • Identify and remove unused permissions
  • Audit access key age and rotation status
Workflow: Start broad (AWS managed policy) → Monitor with CloudTrail → Analyze with Access Advisor → Refine to customer managed policy with only the permissions actually used.
This is the practical workflow for achieving least privilege without blocking developers. You start permissive during development, observe actual usage patterns via CloudTrail, then use Access Analyzer to generate a tight policy. The credential report is useful for auditing: you can see which users have never logged in, which access keys haven't been rotated, and which users don't have MFA enabled.

💫 IAM Best Practices Summary

Identity

  • Lock away root user credentials
  • Enable MFA everywhere
  • Create individual IAM users (no sharing)
  • Use groups to assign permissions

Access

  • Grant least privilege
  • Use roles for applications & services
  • Rotate credentials regularly
  • Remove unnecessary credentials

Monitoring

  • Enable CloudTrail for API logging
  • Review Access Advisor data
  • Use IAM Access Analyzer
  • Set up billing alerts
Remember: IAM configuration errors are the #1 cause of AWS security incidents. Invest time in getting IAM right from the start.
This slide is a quick reference card. Emphasize that these aren't aspirational goals but baseline requirements. Organizations that skip these steps regularly end up in security incidents. The most common mistakes: using the root user for daily tasks, not enabling MFA, using overly permissive policies, and sharing IAM user credentials between team members. Each of these creates audit and security risks.

🤔 Why This Architecture?

Architecture Decisions

Cost Efficiency: With DynamoDB on-demand pricing and EC2 Auto Scaling, you pay proportional to actual usage.

  3. Security is a shared responsibility. AWS secures the cloud infrastructure; you secure your workloads, data, and access controls.
  4. IAM is foundational. Use groups for permissions, roles for services, MFA for protection, and least privilege for everything.
  5. Never use the root user for daily operations. Create IAM admin users and lock root credentials away with MFA enabled.
These five points are the non-negotiable takeaways from this module. If participants remember nothing else, these five principles will serve them well. The IAM foundation we've laid here is critical — every lab and module going forward depends on understanding how permissions work.
