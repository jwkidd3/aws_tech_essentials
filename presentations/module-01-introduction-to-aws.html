☁️ Introduction to AWS Cloud

What is cloud computing and why does it matter?

Let's start with the fundamentals. Even if participants have experience with AWS, it's worth aligning on definitions and understanding the value proposition clearly.

🖥️ What Is Cloud Computing?

NIST Definition

On-demand delivery of IT resources over the internet with pay-as-you-go pricing — no physical data centers required.

Traditional Model

  • Capital expenditure upfront
  • Capacity planning months ahead
  • Hardware procurement cycles
  • Underutilized or over-provisioned

Cloud Model

  • Operational expenditure (OpEx)
  • Provision in minutes
  • API-driven resource creation
  • Scale up or down on demand
Stress the shift from CapEx to OpEx. In traditional environments, you had to guess capacity needs 3-6 months in advance. Cloud computing flips this: you provision what you need, when you need it, and pay only for what you consume. This is a fundamental shift in how IT operates.

🎯 Six Advantages of Cloud Computing

  1. Trade CapEx for variable expense — pay only for what you consume
  2. Massive economies of scale — lower prices from aggregated usage
  3. Stop guessing capacity — scale up or down in minutes
  1. Increase speed and agility — provision resources in minutes, not weeks
  2. Stop running data centers — focus on your business, not infrastructure
  3. Go global in minutes — deploy across Regions with a few clicks
Walk through each advantage. 1) CapEx to OpEx shift is fundamental. 2) AWS has reduced pricing 130+ times since 2006. 3) Eliminates procurement lead times and idle hardware. 4) Prototype in an afternoon vs. weeks of hardware setup. 5) AWS handles racking, stacking, power, cooling, physical security. 6) Deploy worldwide for lower latency and better user experience.

🌎 AWS Global Infrastructure

Key Concept: AWS infrastructure = Regions > Availability Zones (AZs) > Edge Locations

🌏 Regions

  • 33+ geographic regions worldwide
  • Physically isolated from each other
  • Data stays in-Region unless you replicate

🏧 Availability Zones

  • 105+ AZs globally
  • One or more data centers per AZ
  • Low-latency links between AZs

⚡ Edge Locations

  • 600+ Points of Presence
  • Used by CloudFront (CDN) and Route 53 (DNS)
  • Cache content closer to end users
Draw the hierarchy: Region contains multiple AZs, each AZ contains one or more data centers. Regions are completely independent. AZs within a Region are connected by dedicated high-bandwidth, low-latency networking. Edge Locations are separate from Regions and serve cached content to reduce latency for end users. Emphasize that data sovereignty is Region-level: your data stays in the Region you put it in.

🧮 Choosing an AWS Region

📜 1. Compliance

Data residency laws (GDPR, HIPAA) may dictate Region choice

⏱️ 2. Latency

Deploy close to users — 200ms vs 20ms matters

🔧 3. Service Availability

Not all services in every Region; new services launch in us-east-1 first

💰 4. Cost

Pricing varies by Region; US Regions typically least expensive

Walk through the decision process in order. Compliance is typically non-negotiable and comes first. If you have no compliance constraints, proximity to users is usually the next priority. Then check that the services you need are available. Finally, compare costs. Remind participants they can use multiple Regions for different workloads.

💻 Interacting with AWS

All three interfaces call the same underlying AWS APIs:

🌐 Management Console

  • Web-based GUI
  • Great for learning
  • Used in our labs

🖱️ AWS CLI

  • Command-line interface
  • Scriptable, automatable
  • Ideal for repeatable tasks

🛠️ AWS SDKs

  • Language-specific APIs
  • Python, JS, Java, .NET, Go
  • Application integration
Key Insight: Everything in AWS is an API call. Console clicks, CLI commands, and SDK methods all invoke the same REST APIs.
Stress the API-first nature of AWS. When you click a button in the Console, it makes an API call. The CLI wraps those same calls in shell commands. SDKs wrap them in language-native code. 🔒 Security in the AWS Cloud

Shared Responsibility, IAM, and Access Control

Security is job zero at AWS, and it should be for your organization too. This section covers the shared responsibility model and IAM, which is arguably the most important service in all of AWS. You cannot do anything in AWS without IAM being involved.

🛡️ AWS Shared Responsibility Model

Security OF the Cloud (AWS) vs. Security IN the Cloud (Customer)

✅ AWS Responsibility

  • Physical security and hardware
  • Managed services OS/patching
  • Hypervisor and network layer
  • Global infrastructure

⚠️ Customer Responsibility

  • Data encryption (at rest + in transit)
  • IAM and access management
  • OS patching, firewall config
  • Application-level security
This model is tested heavily on AWS certifications and is essential for real-world operations. The key insight: the responsibility line shifts depending on the service. With EC2, you're responsible for the OS and everything above it. With Lambda, AWS manages much more, and your responsibility shrinks to code and IAM. Use the phrase: 'If you can touch it in the Console, you're probably responsible for configuring it securely.'

🔄 Responsibility Shifts by Service Type

LayerEC2 (IaaS)RDS (Managed)Lambda (Serverless)
Application CodeCustomerCustomerCustomer
Data EncryptionCustomerCustomerCustomer
IAM / Access ControlCustomerCustomerCustomer
OS PatchingCustomerAWSAWS
Database EngineCustomerAWSAWS
Hardware / InfrastructureAWSAWSAWS
Takeaway: More managed = less ops responsibility, but you always own IAM, data, and encryption.
Walk through the table column by column. With EC2, you own almost everything above the hypervisor. With RDS, AWS handles the database engine, patching, backups (if configured), and OS. With Lambda, AWS handles nearly everything except your code and IAM configuration. Notice that IAM and data encryption are ALWAYS the customer's responsibility regardless of service type. This is a critical point.

👤 AWS Identity and Access Management (IAM)

Global service controlling authentication (who signs in) and authorization (what they can do). Free to use — create once, works in all Regions.

Core Components

  • Users — individual identities
  • Groups — collections with shared permissions
  • Roles — temporary credentials for services
  • Policies — JSON permission documents
Key Facts:
  • IAM is free — pay only for resources accessed
  • Global — not Region-specific
  • Underpins every AWS API call
IAM is the first service every AWS practitioner must understand. It underpins every single API call. When we say 'global service,' that means an IAM user created in any context can access resources in any Region (if their policies allow it). Distinguish between authentication (proving identity) and authorization (what you're allowed to do). IAM handles both.

👥 IAM Users & Groups

IAM Users

  • Represent a person or service
  • Console: username + password
  • Programmatic: access key + secret
  • Can belong to multiple groups

IAM Groups

  • Logical collection of users
  • Attach policies to groups, not users
  • Users inherit group policies
  • Cannot be nested
Best Practice: Always assign permissions to groups, then add users to those groups. Avoid attaching policies directly to individual users — this does not scale and becomes unmanageable. Emphasize the group-based approach. In a real organization, you might have groups like 'Developers', 'DBAdmins', 'ReadOnlyAuditors'. When a new developer joins, you add them to the Developers group and they immediately get the right permissions. When they leave, you remove them. This is much cleaner than managing per-user policies. Note the 10-group limit per user, which is rarely hit in practice.

🎭 IAM Roles

Roles provide temporary credentials via STS — no long-term keys needed.

Common Use Cases

  • EC2 Instance Role: access S3/DynamoDB without embedded keys
  • Cross-Account: access resources in another account
  • Federation: corporate directory via SAML/OIDC
  • Service Roles: Lambda, ECS call APIs on your behalf
Best Practice: Use roles instead of access keys. Temporary credentials rotate automatically.
Trust Policy: Every role has a trust policy that defines who can assume the role (the principal). Roles are the preferred mechanism for granting permissions in modern AWS architectures. The key benefit is that temporary credentials expire, reducing the blast radius of a compromise. For the employee directory application we'll build, we'll attach an IAM role to our EC2 instance so the application code can access DynamoDB and S3 without hardcoded credentials. This is a critical best practice that many organizations get wrong early on.

📜 IAM Policy Structure

JSON documents with one or more statements:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3ReadAccess",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ]
    }
  ]
}

Statement Elements

ElementPurpose
EffectAllow or Deny
ActionWhich API operations (e.g., s3:GetObject)
ResourceWhich AWS resource(s) via ARN
ConditionOptional constraints (IP, time, MFA, etc.)
PrincipalWho (used in resource-based policies)
Default Deny: If a policy doesn't explicitly allow an action, it is denied. An explicit Deny always overrides any Allow. Walk through the JSON policy line by line. Version is always 2012-10-17 (this is the policy language version, not a date you set). Sid is optional but recommended for readability. Effect is either Allow or Deny. Action uses the format service:APIOperation. Resource uses ARNs to specify exactly which resources the policy applies to. Note the two resource ARNs in the example: one for the bucket itself (for ListBucket) and one with /* for objects inside the bucket (for GetObject). This is a common pattern. Stress that explicit Deny always wins.

🔐 Root User vs. IAM Users

⚠️ Root User

  • Unrestricted access, cannot be limited
  • Only for: closing account, changing support plan
  • Enable MFA immediately
  • Never create access keys for root

✅ IAM Users

  • Permissions defined by policies
  • Can be restricted, monitored, audited
  • Use for all day-to-day operations
  • Create an IAM admin user instead
Rule: Lock root credentials in a secure vault. Never use root for daily tasks.
The root user is the most powerful and most dangerous identity. Very few tasks require it: closing the account, changing the support plan, restoring IAM permissions. Everything else should use IAM users or roles. Enable MFA on root immediately, do not create access keys for it, and store credentials in a secure vault. These are non-negotiable baseline security requirements.

🔐 Multi-Factor Authentication (MFA)

Requires a one-time code from a device + password. Compromised credentials alone cannot access the account.

Virtual MFA

Authenticator app (Google, Authy, Microsoft). Free, most common.

Hardware MFA

Physical key fob (e.g., Gemalto). Independent of smartphone.

FIDO2 Key

USB key (e.g., YubiKey). Phishing-resistant, multi-account.

Requirement: At minimum, enable MFA on the root user and all IAM users with console access. You can also enforce MFA in IAM policies using the aws:MultiFactorAuthPresent condition key. MFA is the single most impactful security control you can enable. AWS recommends FIDO2 security keys as the strongest option because they're phishing-resistant. Virtual MFA is the most common in practice due to convenience. You can also require MFA for specific API operations through IAM policy conditions. For example, you might allow users to list resources freely but require MFA to delete anything.

🔒 Principle of Least Privilege

Grant Only What Is Needed

Start with zero permissions, add as required. Easier to grant more later than revoke excess.

❌ Anti-Pattern

{
  "Effect": "Allow",
  "Action": "*",
  "Resource": "*"
}
Full admin to everything. Never use in production.

✅ Least Privilege

{
  "Effect": "Allow",
  "Action": [
    "dynamodb:GetItem",
    "dynamodb:PutItem",
    "dynamodb:Query"
  ],
  "Resource":
    "arn:aws:dynamodb:us-east-1:
     123456789012:table/Employees"
}
Scoped to 3 actions on 1 table.
This is one of the most important security principles. Compare the two policies side by side. The wildcard policy is a ticking time bomb. If those credentials leak, the attacker has full access to everything. The scoped policy limits damage to three operations on a single DynamoDB table. AWS provides tools to help: IAM Access Analyzer can generate least-privilege policies based on observed usage. CloudTrail logs show which API calls an identity actually makes, helping you trim excess permissions.

💫 IAM Best Practices Summary

Identity

  • Lock away root credentials
  • Enable MFA everywhere
  • No shared accounts

Access

  • Least privilege always
  • Roles for apps/services
  • Rotate and remove unused keys

Monitoring

  • CloudTrail for API logging
  • Access Advisor + Analyzer
  • Billing alerts
Remember: IAM configuration errors are the #1 cause of AWS security incidents. Invest time in getting IAM right from the start. This slide is a quick reference card. Emphasize that these aren't aspirational goals but baseline requirements. Organizations that skip these steps regularly end up in security incidents. The most common mistakes: using the root user for daily tasks, not enabling MFA, using overly permissive policies, and sharing IAM user credentials between team members. Each of these creates audit and security risks.
