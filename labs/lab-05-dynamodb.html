Step 1

Create a DynamoDB Table

In this step, students create the Employees table. Emphasize the importance of choosing the right primary key design up front, as it cannot be changed later.

🔌 Navigate to DynamoDB

  1. In the AWS Management Console, click the Services menu (or use the search bar at the top)
  2. Type DynamoDB in the search bar
  3. Click DynamoDB from the results to open the DynamoDB dashboard
💡 Tip: Verify your AWS Region in the top-right corner of the console. All resources will be created in this Region.
Make sure all students are in the same Region. DynamoDB is a regional service, so the table will only be visible in the Region where it is created.

📝 Create the Employees Table

  1. Click Create table
  2. For Table name, enter: Employees
  3. For Partition key, enter: id and select type String
  4. Leave Sort key empty — we are using a simple primary key
Key Concept: A simple primary key (partition key only) uniquely identifies each item. A composite primary key adds a sort key for finer-grained access patterns.
Explain the difference between simple and composite primary keys. For this lab we use a simple key for clarity. In production, composite keys are common when you need to model one-to-many relationships within a single table.

⚙️ Configure Table Settings

  1. Under Table settings, select Customize settings
  2. For Table class, select DynamoDB Standard
  3. For Read/write capacity settings, select On-demand
  4. Under Encryption at rest, select Owned by Amazon DynamoDB
  5. Click Create table
✅ Expected Result: The table will appear in the Tables list with status Creating. Wait until the status changes to Active (typically 10–30 seconds).
On-demand capacity mode means DynamoDB automatically scales to handle your workload. You pay per request instead of provisioning capacity units. This is ideal for unpredictable workloads or labs where you do not know the traffic pattern in advance.

Step 4

Query and Scan the Table

This step teaches the critical difference between Scan and Query operations. Scans read every item in the table; Queries are targeted and far more efficient.

🔎 Scan the Table

  1. In Explore table items, ensure the Scan tab is selected
  2. Leave all filters empty
  3. Click Run
✅ Expected Result: All 3 items are returned. You should see John Smith, Jane Doe, and Bob Johnson with all their attributes.
⚠️ Important: A Scan reads every item in the table. On large tables, this is expensive and slow. Avoid Scans in production whenever possible.
Emphasize that a Scan operation consumes read capacity proportional to the entire table size, not the number of results returned. This is fine for 3 items but disastrous for millions.

🔍 Query by Partition Key

  1. Switch to the Query tab
  2. For Partition key (id), enter: 1
  3. Click Run
✅ Expected Result: Only one item is returned — John Smith's record with all attributes (name, location, department, email, badge_number).
Query vs. Scan: Queries target a specific partition key and are significantly more efficient. They only read the items that match, consuming minimal read capacity. Always prefer Query over Scan.
This is one of the most important DynamoDB concepts. Queries go directly to the correct partition, while Scans touch every item. In a table with millions of items, this difference translates to orders-of-magnitude cost and latency savings.

Step 5

Edit an Item

This step demonstrates DynamoDB's schema flexibility. Students will modify an existing attribute and add a new one that does not exist on other items.

✏️ Modify John Smith's Record

  1. In the items list, click on the item with id = 1 (John Smith)
  2. Change the location value from Seattle to San Francisco
  3. Click Add new attribute → String
  4. Attribute name: phone — Value: 555-0100
  5. Click Save changes
✅ Verification: Run a Scan again. John Smith's record should now show location: San Francisco and a new phone attribute. The other two items do not have a phone attribute.
This is a great moment to discuss schema flexibility. Unlike relational databases, DynamoDB does not require all items to have the same attributes. John Smith now has 7 attributes while the other items have 6. This is perfectly valid and is a core feature of NoSQL design.

💡 Schema Flexibility in DynamoDB

Key Takeaway

DynamoDB is schema-flexible. Only the primary key attributes are required on every item. All other attributes are optional and can vary from item to item.

✅ Advantages

  • Easy to evolve your data model
  • No costly ALTER TABLE migrations
  • Different item types in one table

⚠️ Considerations

  • Application must handle missing attributes
  • No built-in schema enforcement
  • Careful design is still essential
Reinforce that schema-flexible does not mean schema-less. Good data modeling is still critical. The application layer is responsible for validation. Many teams use libraries or middleware to enforce schemas at the application level.

Step 6

Add a Global Secondary Index (GSI)

GSIs are one of the most powerful DynamoDB features. They allow you to query the table using an alternate key, enabling different access patterns without duplicating data.

📌 Create the GSI (1/2)

  1. Navigate to the Employees table → Indexes tab
  2. Click Create index
  3. For Partition key, enter: department (String)
  4. For Sort key, enter: name (String)
While students wait for the index to become active, explain that a GSI is essentially a separate table managed by DynamoDB that is kept in sync with the base table. It has its own partition and sort key, allowing completely different query patterns. 'All' projected attributes means every attribute from the base table is copied to the index.

📌 Create the GSI (2/2)

  1. For Index name, enter: department-name-index
  2. For Projected attributes, select: All
  3. Click Create index
⏳ Wait: The index status will show Creating. Wait until it changes to Active before proceeding. This may take 1–2 minutes.
While students wait for the index to become active, explain that a GSI is essentially a separate table managed by DynamoDB that is kept in sync with the base table. It has its own partition and sort key, allowing completely different query patterns. 'All' projected attributes means every attribute from the base table is copied to the index.

🔍 Query Using the GSI

  1. Go to Explore table items
  2. In the table/index dropdown, change it from Employees to department-name-index
  3. Select the Query tab
  4. For Partition key (department), enter: Engineering
  5. Click Run
✅ Expected Result: The query returns John Smith's record (the only employee in the Engineering department). All attributes are visible because we projected All.
Why GSIs matter: Without this index, finding employees by department would require a full table Scan with a filter. The GSI makes this a fast, efficient Query operation. Highlight that the GSI allows querying by department without touching the entire table. This is how DynamoDB supports multiple access patterns. In real applications, you might have several GSIs for different query needs. Note the sort key (name) allows results within a department to be returned in alphabetical order.

Step 7

Enable DynamoDB Streams

DynamoDB Streams captures a time-ordered sequence of item-level changes. This is the foundation for event-driven architectures, such as triggering Lambda functions when data changes.

🔄 Enable DynamoDB Streams

  1. Navigate to the Employees table → Exports and streams tab
  2. Under DynamoDB stream details, click Enable
  3. For View type, select: New and old images
  4. Click Enable stream
✅ Expected Result: The stream status changes to Enabled. You will see a Stream ARN and the view type set to NEW_AND_OLD_IMAGES.
Explain the four view types: KEYS_ONLY, NEW_IMAGE, OLD_IMAGE, NEW_AND_OLD_IMAGES. We chose NEW_AND_OLD_IMAGES so that stream consumers can see both the before and after states of each changed item. This is useful for auditing, replication, and triggering business logic.

💡 DynamoDB Streams Use Cases

⚡ Event-Driven

Trigger AWS Lambda functions on every insert, update, or delete

🔁 Replication

Replicate changes to another table, Region, or data store

📋 Auditing

Capture old and new images for compliance and change tracking

Stream records are retained for 24 hours. Consumers must process them within that window or the records expire.
DynamoDB Streams combined with Lambda is one of the most common serverless patterns on AWS. For example, a stream could trigger a Lambda function that sends a welcome email when a new employee record is created, or updates a search index whenever data changes.

Step 8

Enable Point-in-Time Recovery

PITR provides continuous backups of DynamoDB table data. It is a critical feature for production workloads that need protection against accidental writes or deletes.

🔒 Enable Point-in-Time Recovery (PITR)

  1. Navigate to the Employees table → Backups tab
  2. Under Point-in-time recovery, click Edit
  3. Check the box to Enable Point-in-time recovery
  4. Click Save changes
✅ Expected Result: PITR status changes to Enabled. You will see the Earliest restore date/time and Latest restore date/time fields.
PITR provides continuous backups for the last 35 days. You can restore your table to any second within that window. Restores always create a new table.
PITR is an essential feature for production tables. It protects against accidental deletes or overwrites. The restore creates a new table, so the original table remains unchanged. There is an additional cost for PITR based on table size.

Step 9

Create an On-Demand Backup

On-demand backups are full snapshots of the table. Unlike PITR, they are retained indefinitely until you delete them.

💾 Create an On-Demand Backup

  1. In the Backups tab, click Create backup
  2. For Backup name, enter: employees-backup-lab
  3. Click Create backup
✅ Expected Result: The backup appears in the Backups list with status Available. You will see the backup ARN, table name, and creation timestamp.

On-Demand vs. PITR

  • PITR: Continuous, automatic, last 35 days, restore to any second
  • On-Demand: Manual snapshot, retained indefinitely, restore to that exact point
On-demand backups are useful before major data migrations or application deployments. They do not affect table performance or availability. Both PITR and on-demand restores create a new table rather than overwriting the existing one.

Step 10

Delete Items and Clean Up

Students will delete a single item and learn about TTL as an alternative for automatic expiration. The table itself is kept for potential use in subsequent labs.

🗑️ Delete an Item

  1. Go to Explore table items and run a Scan to see all items
  2. Select the checkbox next to the item with id = 3 (Bob Johnson)
  3. Click Actions → Delete items
  4. Confirm the deletion by clicking Delete
✅ Expected Result: The items list now shows only 2 items: John Smith (id: 1) and Jane Doe (id: 2).
Deleting items is immediate and permanent. In production, consider using TTL (Time to Live) to automatically expire items after a specified timestamp. TTL is useful for session data, temporary records, or compliance-driven retention policies.

🧹 Clean Up

⚠️ Clean-Up Actions:
  1. Keep the Employees table — it may be used by subsequent labs or applications
  2. Delete the on-demand backup (optional, to save costs):
    • Go to Backups tab
    • Select employees-backup-lab
    • Click Delete and confirm
Production Tip: Use Time to Live (TTL) to automatically expire items instead of manually deleting them. TTL deletes are free and do not consume write capacity.
Remind students that DynamoDB on-demand pricing means they only pay for actual reads and writes. An idle table with a few items has negligible cost. The on-demand backup does incur a small storage charge, so deleting it is good practice in a lab environment.
