📝 Lab Details

Review objectives. DynamoDB is a fully managed NoSQL database. This lab focuses on core CRUD operations and the schema-flexible nature of the service. Ensure everyone has console access.

📊 Create the Employees Table

  1. Navigate to DynamoDB in the console
  2. Click Create table
  3. Table name: Employees
  4. Partition key: id (String)
  5. Table settings: Customize settings
  6. Capacity mode: On-demand
  7. Leave all other settings as default
  8. Click Create table
✅ Expected: Table status changes to Active within a few seconds.
On-demand capacity mode is simpler for labs — no need to estimate read/write capacity units. The partition key (id) uniquely identifies each item. DynamoDB does not require you to define other attributes upfront — that is the schema flexibility.

📝 Add Employee Items

  1. Click on the Employees table, then Explore table items
  2. Click Create item

Add these three items (click Add new attribute for each extra field):

idnamedepartmentlocation
E001John SmithEngineeringSeattle
E002Jane DoeMarketingNew York
E003Bob JohnsonSalesAustin
💡 Tip: For each item, click Add new attribute → String to add name, department, and location fields.
Walk students through creating the first item step by step. For subsequent items they should be able to follow the pattern. Each attribute must be added individually since DynamoDB has no predefined schema beyond the key.

🔍 Scan and Query the Table

Scan (returns all items):

  1. In Explore table items, select Scan
  2. Click Run — all 3 items appear

Query (search by partition key):

  1. Switch to Query
  2. Partition key value: E001
  3. Click Run — only John Smith's record appears
✅ Expected: Scan returns all 3 items. Query returns only the matching item.
Scans read every item in the table (expensive at scale). Queries use the partition key for efficient lookups. In production, always prefer queries over scans. You can also add filters to narrow results, but filters are applied after the read.

🔄 Edit an Item — Schema Flexibility

  1. Click on item E003 (Bob Johnson)
  2. Click Add new attribute → String
  3. Add: phone = 512-555-0199
  4. Click Save changes
  5. Run a Scan again — notice E003 has a phone attribute that other items do not have
✅ Expected: E003 now has 5 attributes while E001 and E002 still have 4. Each item can have different attributes.
This is one of the key differences between DynamoDB and relational databases. Items in the same table can have different attributes. Only the partition key (and sort key, if defined) are required. This flexibility is powerful for evolving application schemas.

🗑 Clean Up

  1. Go to DynamoDB → Tables
  2. Select the Employees table
  3. Click Delete
  4. Type confirm and click Delete table
✅ Expected: The Employees table is deleted. DynamoDB deletes all items and indexes with the table.
Deleting the table removes all data permanently. In production, you would enable Point-in-Time Recovery or take on-demand backups before deletion. On-demand tables do not incur charges when idle.
