🏗️ What We Are Building

Architecture Overview

In this lab you will deploy a simple web application on a single EC2 instance:

  • EC2 Instance — Amazon Linux 2023, t2.micro, running a Python Flask application on port 80
  • Security Group — Allows inbound HTTP (port 80) from anywhere and SSH (port 22) from your IP
  • IAM Role — Grants the instance permission to access S3 (photos) and DynamoDB (employee data)
  • User Data Script — Automatically installs dependencies and starts the application at launch
💡 Key Concept: User data scripts run only once at first boot, automating instance configuration without manual SSH access.
Walk through the architecture so students understand the full picture before diving into the steps. The Flask app reads from DynamoDB and stores photos in S3.

Step 1

Navigate to EC2 and Launch an Instance

Step 1: Navigate to EC2 and begin the instance launch wizard.

🚀 Navigate to EC2

  1. Sign in to the AWS Management Console
  2. In the search bar at the top, type EC2
  3. Click EC2 under Services to open the EC2 Dashboard
  4. Click the orange Launch instance button
✅ Expected: You should see the "Launch an instance" configuration page with the Name and tags section at the top.
Make sure students are in the correct region before launching. The region selector is in the top-right corner of the console.

📝 Name and AMI Selection

  1. Under Name and tags, enter: 
    employee-directory-app
  2. Under Application and OS Images (Amazon Machine Image):
    • Ensure Amazon Linux is selected (it is the default)
    • From the Amazon Machine Image dropdown, select Amazon Linux 2023 AMI
    • Architecture: 64-bit (x86)
  3. Under Instance type, select t2.micro
    • You should see the Free tier eligible label
💡 Note: Amazon Linux 2023 is the latest generation of Amazon Linux, optimized for AWS with long-term support.
t2.micro provides 1 vCPU and 1 GiB of memory, sufficient for this lab application. Ensure students select the correct AMI.

Step 2

Configure Key Pair

Step 2: Create a key pair for SSH access to the instance.

🔑 Create a Key Pair

  1. In the Key pair (login) section, click Create new key pair
  2. In the dialog that appears, configure:
    SettingValue
    Key pair nameemployee-dir-key
    Key pair typeRSA
    Private key file format.pem
  3. Click Create key pair
  4. The .pem file will automatically download to your computer
⚠️ Important: Save this file in a known location. You cannot download this key again. If you lose it, you will need to create a new key pair.
✅ Expected: The key pair employee-dir-key should now appear selected in the Key pair dropdown.
On macOS/Linux, students may need to run chmod 400 on the .pem file before using it for SSH. On Windows, they can use PuTTY with the .pem converted to .ppk format. For this lab, we primarily need the key pair to exist.

Step 3

Configure Network Settings

Step 3: Configure the VPC, subnet, public IP, and security group settings.

🌐 Network Settings

  1. In the Network settings section, click Edit
  2. Configure the following:
    SettingValue
    VPCSelect the default VPC
    SubnetSelect any public subnet (or leave as "No preference")
    Auto-assign public IPEnable
💡 Why a public IP? We need a public IP address so that we can access the Employee Directory web application from our browser. Without it, the instance would only be reachable from within the VPC.
Explain that the default VPC has public subnets already configured with an internet gateway. Auto-assign public IP gives the instance a public IPv4 address at launch.

🔒 Security Group Configuration

  1. Under Firewall (security groups), select Create security group
  2. Security group name: 
    employee-directory-sg
  3. Description: 
    Security group for Employee Directory EC2 instance
  4. Configure Inbound security group rules:
    RuleTypePortSource
    Rule 1 (default)SSH22My IP
    Rule 2 (click Add security group rule)HTTP80Anywhere (0.0.0.0/0)
⚠️ Security Note: SSH is restricted to "My IP" for security. HTTP is open to the internet so anyone can view the application. In production, you would place a load balancer in front and restrict direct instance access.
Discuss why SSH should be limited to the student's IP. Emphasize that opening port 22 to 0.0.0.0/0 is a security anti-pattern. The "My IP" option auto-detects the student's current public IP.

Step 4

Configure IAM Instance Profile

Step 4: Attach an IAM role so the application can access S3 and DynamoDB without embedding credentials.

👤 Create the IAM Role

Before selecting the role on the EC2 launch page, we need to create it in IAM.

  1. Open a new browser tab and navigate to the IAM Console (search for "IAM")
  2. In the left navigation, click Roles, then Create role
  3. Trusted entity type: AWS service
  4. Use case: Select EC2, then click Next
  5. Search for and select these policies:
    • ✅ AmazonS3FullAccess
    • ✅ AmazonDynamoDBFullAccess
  6. Click Next
  7. Role name: 
    EmployeeDirectoryAppRole
  8. Click Create role
Explain that IAM roles are the recommended way to grant AWS permissions to EC2 instances. This avoids storing access keys on the instance. The trust policy allows EC2 to assume this role.

🔗 Attach the IAM Role to EC2

  1. Switch back to the EC2 Launch instance browser tab
  2. Scroll down to Advanced details and expand it
  3. In the IAM instance profile dropdown, select EmployeeDirectoryAppRole
    • If you do not see it, click the refresh icon next to the dropdown
✅ Expected: The IAM instance profile dropdown should show EmployeeDirectoryAppRole selected.

💡 Why Use an IAM Role?

An IAM instance profile passes temporary credentials to the EC2 instance via the instance metadata service. The application uses these credentials automatically through the AWS SDK — no access keys need to be stored on the instance.

Stress the security benefit of roles vs. access keys. Temporary credentials are rotated automatically. This is an AWS best practice.

Step 5

Add User Data Script

Step 5: Add the bootstrap script that will install and start the application automatically when the instance launches.

📜 User Data Bootstrap Script

Still in the Advanced details section, scroll down to User data.

Paste the following script into the User data text box:

#!/bin/bash -ex
wget https://aws-tc-largeobjects.s3-us-west-2.amazonaws.com/DEV-AWS-MO-GCNv2/FlaskApp.zip
unzip FlaskApp.zip
cd FlaskApp/
yum -y install python3-pip
pip install -r requirements.txt
yum -y install stress
export PHOTOS_BUCKET=${SUB_PHOTOS_BUCKET}
export AWS_DEFAULT_REGION=<REGION>
export DYNAMO_MODE=on
FLASK_APP=application.py /usr/local/bin/flask run --host=0.0.0.0 --port=80
⚠️ Replace placeholders before pasting:
  • Replace ${SUB_PHOTOS_BUCKET} with the name of your S3 bucket (created in a later lab, or provided by your instructor)
  • Replace <REGION> with your AWS region (e.g., us-east-1, us-west-2)
Walk through each line of the script. If students haven't created the S3 bucket yet, they can leave the placeholder for now and the app will still load, just without photo functionality.

🔍 Understanding the User Data Script

LineWhat It Does
#!/bin/bash -exRuns as Bash; -e exits on error, -x prints each command for debugging
wget ... FlaskApp.zipDownloads the Employee Directory application from S3
unzip FlaskApp.zipExtracts the application files
yum -y install python3-pipInstalls pip (Python package manager)
pip install -r requirements.txtInstalls Python dependencies (Flask, boto3, etc.)
yum -y install stressInstalls the stress tool (used in a later scaling lab)
export ...Sets environment variables for the S3 bucket, region, and DynamoDB mode
FLASK_APP=... flask runStarts the Flask web server on port 80
Explain each line so students understand what happens at boot. The stress utility will be used in the Auto Scaling lab. The Flask app listens on 0.0.0.0 so it accepts connections from any IP, and port 80 is the standard HTTP port.

Test the Application

🌐 Test the Employee Directory Application

    http://<PUBLIC_IPv4_ADDRESS>

If the page doesn't load after several minutes, have students check: 1) Security group allows HTTP on port 80, 2) They're using http:// not https://, 3) The public IP is correct, 4) The user data script ran without errors.

🚧 Troubleshooting

If the application does not load, check the following:

❌ Common Issues:
  • Using HTTPS instead of HTTP — The app runs on HTTP only. Make sure the URL starts with http://
  • Security group missing HTTP rule — Verify port 80 is open to 0.0.0.0/0
  • User data script errors — Check instance system log: Select instance → Actions → Monitor and troubleshoot → Get system log
  • Wrong public IP — Refresh the Instances page and verify the IP address
  • Auto-assign public IP was not enabled — If no public IP is listed, you will need to terminate and re-launch
Walk around and help students who are having issues. The system log is invaluable for debugging user data script failures. Look for Python errors or package installation failures.

📄 View Instance Metadata

  1. Select your instance on the Instances page
  2. Click Actions → Instance settings → Edit instance metadata options
  3. Review the metadata settings:
    • IMDSv2 — the token-based metadata service (more secure)
    • The instance metadata service provides information about the instance to applications running on it

💡 What is Instance Metadata?

The instance metadata service (IMDS) allows code running on the instance to access information such as instance ID, public IP, IAM role credentials, region, and more — all without needing to make external API calls. The Flask app uses IMDS to get the IAM role credentials for accessing S3 and DynamoDB.

IMDSv2 requires a session token, which protects against SSRF attacks. This is the default on Amazon Linux 2023. The metadata endpoint is 169.254.169.254.

Step 9

Stop and Start the Instance

Step 9: Demonstrate what happens when an EC2 instance is stopped and started, particularly the public IP behavior.

⏹️ Stop the Instance

  1. Select your employee-directory-app instance
  2. Click Instance state → Stop instance
  3. In the confirmation dialog, click Stop
  4. Wait for the Instance state to change to Stopped
  5. Observe: the Public IPv4 address field is now empty (the public IP has been released)
💡 Key Observation: When you stop an instance, its public IP address is released back to the AWS pool. The private IP address is retained.
Emphasize that stopping an instance does not delete it or its EBS volume. You are not charged for compute while stopped, but you are still charged for the EBS storage.

▶️ Start the Instance

  1. With the instance still selected, click Instance state → Start instance
  2. Wait for the Instance state to change to Running
  3. Note the new Public IPv4 address — it is likely different from before
  4. Try accessing the application at the new IP: 
    http://<NEW_PUBLIC_IP>
✅ Expected: The application should load again at the new public IP address (after the user data script completes, which may take 2–3 minutes).

💡 Why Does the Public IP Change?

Dynamic public IPs are assigned from AWS's pool at launch. To keep a fixed public IP, you would use an Elastic IP address. Alternatively, placing a load balancer in front of your instances provides a stable DNS endpoint.

This is a great teaching moment about Elastic IPs and load balancers. In later labs, students will set up a load balancer, which solves this problem. Note that user data only runs on first boot by default, but the app process may need to restart.
