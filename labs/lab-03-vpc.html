📚 Lab 3

Create a VPC and Relaunch the Corporate Directory Application in Amazon EC2

AWS Technical Essentials • Lab 3 of 6

Lab 3 focuses on Amazon VPC. Students will create a custom VPC with public and private subnets, then relaunch the application from Lab 2 inside the new VPC.

🎯 Lab Overview

Objective: Create a custom Amazon VPC with public and private subnets across two Availability Zones, then launch the Employee Directory application in the new VPC.

  • Duration: ~30 minutes
  • Services used: Amazon VPC, EC2, Internet Gateway, NAT Gateway
  • Prerequisite: Lab 2 completed (IAM role and key pair created)

⚠️ Important: You will need the IAM role (EmployeeDirectoryAppRole) and key pair (employee-dir-key) from Lab 2.

Review the objectives with students. Emphasize that this lab demonstrates network isolation, a fundamental security concept. The app itself is the same as Lab 2 -- the difference is where it runs.

🛠️ What You Will Build

VPC Architecture

  • A custom VPC with CIDR 10.1.0.0/16
  • 2 public subnets across 2 Availability Zones
  • 2 private subnets across 2 Availability Zones
  • An Internet Gateway for public subnet internet access
  • A NAT Gateway for private subnet outbound access
  • Appropriate route tables for each subnet type
Walk through the architecture before students start building. Make sure they understand the difference between public and private subnets and the role of each gateway type. Key Concept: Public subnets route to the Internet Gateway. Private subnets route to the NAT Gateway. This separation is the foundation of secure AWS architectures.

Step 1

Create a Custom VPC

This step uses the "VPC and more" wizard, which creates the VPC along with subnets, route tables, gateways, and associations in a single operation.

🚀 Navigate to the VPC Dashboard

  1. Sign in to the AWS Management Console
  2. In the top search bar, type VPC
  3. Click VPC under Services to open the VPC Dashboard

🔎 Observe: The dashboard shows your existing default VPC and its associated resources. You are about to create a brand-new custom VPC alongside it.

Point out that every AWS account comes with a default VPC in each region. The default VPC is fine for quick experiments, but production workloads should use a custom VPC with deliberate CIDR planning.

📝 Create VPC — Settings

  1. Click the "Create VPC" button (orange button, top right)
  2. Under Resources to create, select "VPC and more"
  3. In Name tag auto-generation, enter: employee-directory-vpc
  4. Set IPv4 CIDR block to: 10.1.0.0/16

⚠️ Why "VPC and more"? This option automatically creates subnets, route tables, an Internet Gateway, and NAT Gateway — saving you many manual steps.

The /16 CIDR gives us 65,536 IP addresses. The "VPC and more" wizard is relatively new and dramatically simplifies VPC creation. Make sure students select this option and not "VPC only."

📝 Create VPC — Subnets & Gateways

Continue configuring the remaining settings:

  1. Number of Availability Zones: 2
  2. Number of public subnets: 2
  3. Number of private subnets: 2
  4. NAT gateways: In 1 AZ
  5. VPC endpoints: None

💰 Cost note: We select NAT gateway in only 1 AZ to reduce costs during the lab. In production, you would typically deploy a NAT gateway in each AZ for high availability.

NAT Gateways are billed per hour and per GB of data processed. For a lab environment, one NAT Gateway is sufficient. In production, each AZ should have its own NAT Gateway to avoid cross-AZ traffic and provide redundancy.

✅ Create and Wait

  1. Review the Preview panel on the right side — it shows a visual map of all resources that will be created
  2. Click "Create VPC"
  3. Wait for all resources to show a green ✔ Success status

✅ Expected result: You should see green checkmarks next to: VPC, Subnets (4), Route Tables (3), Internet Gateway, and NAT Gateway. This may take 1–2 minutes.

🚨 Do not proceed until all resources show a green checkmark. The NAT Gateway typically takes the longest to provision.

The NAT Gateway can take up to 2 minutes to become active. Students should wait for all green checkmarks before moving on. If any resource fails, they should delete the VPC and retry.

Key difference: The public route table sends internet-bound traffic (0.0.0.0/0) to the Internet Gateway. The private route table sends it to the NAT Gateway. This is the most important concept in VPC networking. The route table determines whether a subnet is public or private. A subnet is public only if its route table has a route to an Internet Gateway. Emphasize this distinction clearly.

🔎 Verify the Gateways

🌐 Internet Gateway

  1. Click Internet gateways in the left nav
  2. Find the gateway associated with your VPC
  3. Verify the State column shows Attached
  4. Verify it is attached to employee-directory-vpc

🔒 NAT Gateway

  1. Click NAT gateways in the left nav
  2. Find the NAT gateway in your VPC
  3. Verify the State column shows Available
  4. Note it is deployed in a public subnet

✅ Checkpoint: Internet Gateway is Attached and NAT Gateway is Available. Your VPC networking is ready.

The NAT Gateway itself must reside in a public subnet because it needs to reach the internet via the Internet Gateway. It then provides outbound-only internet access to resources in private subnets.

Step 3

Create a Security Group for the Application

Security groups act as virtual firewalls at the instance level. We need a new security group in the new VPC because security groups are VPC-specific.

🛡️ Create a New Security Group

  1. In the left navigation, click Security groups (under Security)
  2. Click "Create security group"
  3. Configure the basic details:
    • Security group name: employee-directory-app-sg
    • Description: Security group for Employee Directory App
    • VPC: Select employee-directory-vpc from the dropdown

⚠️ Important: Make sure you select the correct VPC! Security groups are tied to a specific VPC and cannot be used across VPCs.

A common mistake is leaving the default VPC selected. Have students double-check that they select employee-directory-vpc. If they create the security group in the wrong VPC, they will not be able to use it with the EC2 instance later.

🛡️ Add Inbound Rules

Under Inbound rules, click "Add rule" three times to create these rules:

TypePortSourcePurpose
HTTP800.0.0.0/0 (Anywhere-IPv4)Web application traffic
HTTPS4430.0.0.0/0 (Anywhere-IPv4)Secure web traffic
SSH22My IPRemote administration

🔒 Security best practice: SSH access is restricted to "My IP" only. Selecting "My IP" auto-fills your current public IP address. Never open SSH to 0.0.0.0/0 in any environment.

Outbound rules: Leave the default outbound rule as-is. It allows all outbound traffic, which the application needs to reach DynamoDB and S3.

Review each rule with students. HTTP on port 80 is required because the Flask app listens on port 80. HTTPS is included for completeness. SSH is restricted to the student's IP only. Outbound rules default to allow all.

✅ Finalize the Security Group

  1. Scroll down and review all your settings
  2. Click "Create security group"

✅ Expected result: You should see a success banner: "Security group sg-xxxxxxxx was created successfully."

Security Group Summary

  • Name: employee-directory-app-sg
  • VPC: employee-directory-vpc
  • Inbound: HTTP (80), HTTPS (443), SSH (22 — My IP only)
  • Outbound: All traffic (default)
Security groups are stateful -- if you allow inbound traffic on a port, the response traffic is automatically allowed outbound regardless of outbound rules. This is different from network ACLs, which are stateless.

Step 4

Launch EC2 Instance in the New VPC

Now we launch a new EC2 instance in the custom VPC. The configuration is similar to Lab 2, but with the critical difference of placing it in our new VPC.

🚀 Launch Instance — Basic Settings

  1. Navigate to EC2 > Instances > "Launch instances"
  2. Set the Name: employee-directory-app-v2
  3. Under Application and OS Images (AMI):
    • Select Amazon Linux 2023 (should be pre-selected)
  4. Under Instance type:
    • Select t2.micro (Free Tier eligible)
  5. Under Key pair:
    • Select the existing key pair: employee-dir-key
These settings mirror Lab 2. The key difference is in the network settings on the next slide. Make sure students select the existing key pair from Lab 2 rather than creating a new one.

🖧️ Launch Instance — Network Settings

Click "Edit" next to Network settings, then configure:

  1. VPC: Select employee-directory-vpc
  2. Subnet: Select one of the public subnets
    • Look for a subnet name containing public
  3. Auto-assign public IP: Select Enable
  4. Firewall (security groups): Select "Select existing security group"
  5. Choose: employee-directory-app-sg

🚨 Critical: You must select a public subnet and enable Auto-assign public IP. Without both, the instance will not be reachable from the internet.

This is the most error-prone step. Common mistakes: selecting a private subnet, forgetting to enable auto-assign public IP, or using the wrong security group. Verify each setting with students before proceeding.

🛠️ Launch Instance — Advanced Details

Expand "Advanced details" at the bottom of the page:

  1. IAM instance profile: Select EmployeeDirectoryAppRole
  2. Scroll all the way down to User data
  3. Paste the following bootstrap script:
#!/bin/bash -ex
wget https://aws-tc-largeobjects.s3-us-west-2.amazonaws.com/DEV-AWS-MO-GCNv2/FlaskApp.zip
unzip FlaskApp.zip
cd FlaskApp/
yum -y install python3-pip

This is the same user data script from Lab 2. Remind students to replace SUB_PHOTOS_BUCKET with their actual S3 bucket name and REGION with their current region (e.g., us-east-1). The IAM role provides the instance with permissions to access DynamoDB and S3.

Before launching, update these placeholders in the User data script:

Replace with your actual S3 bucket name from the earlier labs.

Example: employee-photo-bucket-abc123

Once the replacements are made, click "Launch instance". ✅ Expected result: A success message: "Successfully initiated launch of instance i-xxxxxxxx."

Step 6

Verify Network Isolation

This is a conceptual step to reinforce the value of VPC isolation. Guide students through comparing the two instances.

🔒 Verify Network Isolation

  1. Go to EC2 > Instances
  2. You should see both instances:
    • employee-directory-app (Lab 2 — default VPC)
    • employee-directory-app-v2 (Lab 3 — custom VPC)
  3. Click each instance and compare the VPC ID in the Details tab
💡 Network Isolation: These two instances are in different VPCs. They cannot communicate with each other directly — even though they are in the same AWS account and region. This is network isolation in action.
This is a fundamental concept. VPCs are logically isolated. To connect two VPCs, you would need VPC Peering or Transit Gateway. This isolation is the foundation of multi-tenant architectures and defense-in-depth security strategies.

🛠️ Compare Route Tables

Navigate back to VPC > Route tables and review:

🌐 Public Subnet Route Table

  • 10.1.0.0/16 → local
  • 0.0.0.0/0 → Internet Gateway

Allows inbound and outbound internet traffic

🔒 Private Subnet Route Table

  • 10.1.0.0/16 → local
  • 0.0.0.0/0 → NAT Gateway

Allows outbound-only internet traffic

Why This Matters

The route table is what makes a subnet "public" or "private." A subnet associated with a route table that has a route to an Internet Gateway is a public subnet. Everything else is private.

Spend time on this slide. The route table concept is tested on the AWS Cloud Practitioner and Solutions Architect exams. Make sure students understand that it is the route table, not the subnet name, that determines public vs. private behavior.

Step 7

Understand Private Subnet Behavior

This is a discussion-oriented step. No hands-on changes are required, but it reinforces architectural best practices.

🔒 Private Subnet Behavior

Instances in Private Subnets

  • Cannot be reached directly from the internet (no inbound access)
  • Can reach the internet via the NAT Gateway (for updates, patches, API calls)
  • Can communicate with other resources in the same VPC

💡 Architectural best practice: Place only resources that need direct internet access (web servers, load balancers) in public subnets. Everything else — databases, application servers, caches — belongs in private subnets.

This is a discussion-oriented slide. Ask students: "If you had a database for this application, which subnet type would you place it in?" (Answer: private). "Why?" (Answer: databases should not be directly accessible from the internet. They only need to communicate with the application tier.)

🏗️ Architectural Best Practices

🌐 Public Tier

  • Load balancers
  • Bastion hosts
  • NAT Gateways
  • Web servers (if no ALB)

💻 Application Tier

  • App servers
  • Container tasks
  • Lambda (VPC-connected)
  • Internal APIs

🗃️ Data Tier

  • RDS databases
  • ElastiCache
  • Elasticsearch
  • Redshift
Multi-AZ: Deploy across multiple Availability Zones for high availability. Our VPC is already set up with subnets in 2 AZs, ready for a highly available architecture. This three-tier architecture is the standard pattern for AWS deployments. The public tier faces the internet, the application tier handles business logic, and the data tier stores persistent data. Each tier should be in its own subnet group with appropriate security groups.

Step 8

Stop the Old EC2 Instance

Cleanup step. We stop the old instance from Lab 2 to avoid unnecessary charges and to keep the environment tidy.

🚫 Stop the Old EC2 Instance

  1. Navigate to EC2 > Instances
  2. Select the employee-directory-app instance (from Lab 2)
  3. Click Instance state > Stop instance
  4. In the confirmation dialog, click "Stop"

⚠️ Stop vs. Terminate: Stopping preserves the instance and its EBS volume (you can restart it later). Terminating permanently deletes it. For this lab, stop is sufficient.

✅ Expected result: The old instance state changes to Stopping and then Stopped. The new employee-directory-app-v2 remains Running.

Stopping the old instance prevents additional charges for compute time. The EBS volume will still incur a small charge. If students want to fully clean up, they can terminate instead, but stopping is safer in case they need to reference it later.
