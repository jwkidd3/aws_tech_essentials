🔐 Lab 1: Introduction to AWS IAM

AWS Technical Essentials

Identity and Access Management

Welcome students to Lab 1. This is the first of six labs in the AWS Technical Essentials course. Ensure all students have their lab credentials ready before starting.

📋 Lab Overview

🎯 Objectives

  • Navigate the IAM Dashboard
  • Create and manage IAM Users
  • Create IAM Groups and assign policies
  • Author a custom IAM Policy using JSON
  • Explore IAM Roles and their trust relationships
  • Test IAM permissions by signing in as a new user

📝 Lab Details

  • Duration: ~20 minutes
  • Level: Introductory
  • Services: IAM, S3, EC2

✅ Prerequisites

  • AWS lab account credentials
  • Web browser (Chrome or Firefox recommended)
  • Access to an incognito/private browsing window
Review the objectives with students. Emphasize that IAM is a foundational service that underpins all access control in AWS. Every other lab in this course will build on the concepts covered here.

Step 1

Sign in to the AWS Management Console

This step gets everyone logged in and oriented. Wait for all students to complete this before proceeding.

🔓 Sign in to the AWS Management Console

  1. Open your web browser and navigate to:
    https://console.aws.amazon.com/
  2. Enter the Account ID or account alias provided by your instructor
  3. Enter the IAM user name and password from your lab credentials
  4. Click Sign in
⚠️ Important: Verify you are in the correct AWS Region. Check the Region selector in the top-right corner of the console. Your instructor will specify which Region to use (e.g., US East (N. Virginia) — us-east-1).
✅ Expected Result: You should see the AWS Management Console home page with service shortcuts and a search bar at the top.
Distribute lab credentials if not already done. Confirm the Region with students. If any student has trouble signing in, verify they are using the correct Account ID and IAM credentials, not a personal AWS account.

Step 2

Explore the IAM Dashboard

This step familiarizes students with the IAM Dashboard. Encourage them to take their time exploring.

📈 Navigate to IAM

  1. In the AWS Management Console, click the Search bar at the top (or press Alt+S)
  2. Type IAM and select IAM from the results
  3. You will land on the IAM Dashboard
💡 Note: IAM is a global service — it is not tied to any specific AWS Region. You will notice there is no Region selector when viewing IAM resources.
Point out that IAM is global. This is a common exam topic. Resources created in IAM are available across all Regions.

📈 Review the IAM Dashboard

On the IAM Dashboard, take note of the following information:

  1. AWS Account section: Find the Account ID and Account Alias (if configured). Note the Sign-in URL for IAM users — you will use this later.
  2. IAM Resources summary: Review the counts for:
    • Users
    • User groups
    • Roles
    • Policies
    • Identity providers
  3. Security recommendations: Review any recommendations displayed, such as enabling MFA on the root account or rotating access keys.
✅ Expected Result: You should see a summary dashboard with resource counts and the IAM sign-in URL for your account. Copy the sign-in URL — you will need it in Step 8.
Have students copy the IAM sign-in URL to a text file or notepad. They will need it when testing permissions in Step 8. Discuss the security recommendations briefly.

Step 3

Explore Existing IAM Users

This step helps students understand the existing IAM configuration before they start making changes.

👤 Explore Existing IAM Users

  1. In the left navigation pane, click Users
  2. Review the list of existing users in the account
  3. Click on any existing user name to view their details
  4. Explore each tab:
    • Permissions — Shows directly attached policies and group-inherited policies
    • Groups — Shows which groups the user belongs to
    • Security credentials — Shows console access, access keys, MFA devices
Walk through the user detail page with students. Highlight the difference between directly attached policies and those inherited through group membership.

👤 Console vs. Programmatic Access

Understanding Access Types

Console Access:
  • Username + password
  • Used for AWS Management Console
  • Optional MFA for additional security
Programmatic Access:
  • Access Key ID + Secret Access Key
  • Used for AWS CLI, SDKs, and APIs
  • Best practice: use IAM Roles instead
💡 Note: On the Security credentials tab, observe whether the user has console access enabled, any active access keys, or MFA configured. These are important security considerations.
Emphasize that best practice is to use IAM Roles for applications rather than long-lived access keys. Access keys should be rotated regularly if used.

Create an IAM User

📝 Create an IAM User (1/4)

  3. For User name, enter: lab-user
  4. Check the box: Provide user access to the AWS Management Console
Have students use a simple but memorable password. They will need it for Step 8 when testing permissions. If the console shows the Identity Center option, make sure students select the IAM user option for this lab.

📝 Create an IAM User (2/4)

Console Access Settings

  2. Under Console password, select Custom password and enter a password you will remember (e.g., Lab@User123!)
  3. Uncheck the box: “Users must create a new password at next sign-in”
  4. Click Next
Have students use a simple but memorable password. They will need it for Step 8 when testing permissions. If the console shows the Identity Center option, make sure students select the IAM user option for this lab.

📝 Create an IAM User (4/4)

    • Console sign-in URL

Step 5

Create an IAM Group

Groups are the recommended way to manage permissions at scale. Emphasize this best practice.

👥 Create an IAM Group

  1. In the left navigation, click User groups, then click Create group
  2. For User group name, enter: EC2-Admins
  3. Under Add users to the group, check the box next to lab-user
  4. Under Attach permissions policies, search for AmazonEC2FullAccess and check the box next to it
  5. Click Create user group
Explain that by adding lab-user to this group, the user will inherit the AmazonEC2FullAccess policy in addition to the S3 read-only policy that was directly attached. This demonstrates how permissions can come from multiple sources.

👥 Verify the Group

  1. After creation, click on the EC2-Admins group name
  2. Review the Users tab — confirm lab-user is listed as a member
  3. Review the Permissions tab — confirm AmazonEC2FullAccess is attached
✅ Expected Result: The EC2-Admins group should show:
  • 1 user (lab-user)
  • 1 attached policy (AmazonEC2FullAccess)
Best Practice: In production environments, always assign permissions to groups rather than individual users. This makes permission management scalable and auditable.
This is a good moment to discuss the principle of least privilege and how groups help manage permissions across teams. Ask students: what permissions does lab-user now have in total? Answer: S3 read-only (direct) + EC2 full access (via group).

Step 6

Create a Custom IAM Policy

This step introduces policy authoring. Go slowly and explain each element of the JSON policy document.

📜 Create a Custom IAM Policy (1/2)

  1. In the left navigation, click Policies
  2. Click Create policy
  3. On the policy editor page, click the JSON tab to switch to the JSON editor
  4. Delete any existing content in the editor and paste the JSON from the next slide
Walk through each element: Version is always 2012-10-17, Effect is Allow or Deny, Action lists the specific API calls permitted, Resource specifies which AWS resources the policy applies to. Note the two Resource entries: one for the bucket itself (ListBucket) and one for objects within the bucket (GetObject).

📜 Custom Policy JSON

Paste this policy into the JSON editor:

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "s3:ListBucket",
                "s3:GetObject"
            ],
            "Resource": [
                "arn:aws:s3:::employee-directory-app-bucket",
                "arn:aws:s3:::employee-directory-app-bucket/*"
            ]
        }
    ]
}
Two Resource ARNs are needed: one for the bucket itself (ListBucket operates on the bucket) and one for objects within the bucket (GetObject operates on objects with the /* wildcard).

📜 Understanding the Policy

Policy Breakdown

Element Value Purpose
Version 2012-10-17 Policy language version (always use this date)
Effect Allow Grants the specified permissions
Action s3:ListBucket, s3:GetObject Specific S3 API operations permitted
Resource Bucket ARN + Object ARN Scoped to a single bucket and its contents
Key Concept: Two Resource ARNs are needed because s3:ListBucket operates on the bucket (arn:aws:s3:::bucket-name) while s3:GetObject operates on objects (arn:aws:s3:::bucket-name/*).
This is a common point of confusion. The bucket-level ARN and object-level ARN are different. ListBucket needs the bucket ARN; GetObject needs the object ARN with the wildcard.

📜 Create a Custom IAM Policy (2/2)

  1. Click Next
  2. For Policy name, enter: EmployeeDirectoryS3ReadPolicy
  3. Optionally add a description: Allows read access to the employee directory S3 bucket
  4. Review the Summary section — it should show:
    • Service: S3
    • Access level: List, Read
    • Resource: Specific bucket
  5. Click Create policy
✅ Expected Result: A success banner appears confirming the policy EmployeeDirectoryS3ReadPolicy was created. You can find it by searching for “EmployeeDirectory” in the Policies list with the Customer managed filter.
After creation, have students search for their policy and review it. Point out the difference between AWS managed policies and customer managed policies in the filter options.

Step 7

Explore IAM Roles

Roles are a critical concept. Students do not create a role in this lab, but understanding them is essential for the remaining labs in this course.

🎭 Explore IAM Roles

  1. In the left navigation, click Roles
  2. Browse the list of existing roles (these are created by AWS services and the lab environment)
  3. Click on any role to examine its details
  4. Review the following tabs:
    • Permissions — The policies attached to this role
    • Trust relationships — Which entities (services, accounts, users) can assume this role
Look for a role with ec2.amazonaws.com in the trust policy to demonstrate how EC2 instances can assume roles. This is a key pattern students will use in later labs.

🎭 Roles vs. Users

👤 IAM Users

  • Long-term credentials (password, access keys)
  • Assigned to a specific person or application
  • Credentials do not expire automatically
  • Directly owns permissions

🎭 IAM Roles

  • Temporary credentials (auto-rotated)
  • Assumed by users, services, or applications
  • Credentials expire and are refreshed automatically
  • Defined by a trust policy (who can assume it)
Best Practice: Use IAM Roles instead of long-term access keys whenever possible. For EC2 instances, Lambda functions, and other AWS services, always assign a role rather than embedding credentials.
Roles are preferred over users for machine-to-machine access because temporary credentials reduce the risk of credential exposure. Roles are also used for cross-account access and federation scenarios.

Step 8

Test IAM Permissions

This is the most engaging step. Students get to see IAM permissions in action by testing what lab-user can and cannot do.

🔎 Test IAM Permissions — Sign In

  1. Open a new incognito / private browser window
    • Chrome: Ctrl+Shift+N (Windows/Linux) or Cmd+Shift+N (Mac)
    • Firefox: Ctrl+Shift+P (Windows/Linux) or Cmd+Shift+P (Mac)
  2. Navigate to the IAM sign-in URL you saved from Step 2 or Step 4
  3. Sign in with:
    • User name: lab-user
    • Password: The password you set in Step 4
✅ Expected Result: You should successfully sign in and see the AWS Management Console as lab-user. Check the top-right corner to confirm the user name.
🚨 Troubleshooting: If sign-in fails, verify you are using the correct IAM sign-in URL (not the root account URL). The URL should contain your Account ID or alias. If you forgot the password, return to the admin session and reset it from the user's Security credentials tab.
It is important to use an incognito window to avoid session conflicts. If students have trouble, verify they are using the IAM user sign-in URL, not the standard console URL.

🔎 Test Read Access

  1. In the lab-user session, navigate to S3 using the search bar
  2. You should be able to see the list of S3 buckets
  3. Click on any bucket to view its contents
✅ Expected Result: You can view S3 buckets and their contents. The S3 console loads successfully and displays bucket listings.
This demonstrates that the read-only policy allows List and Get operations. The S3 console should load successfully and display bucket listings.

🔎 Test Write Access Denied

Now try to create a new bucket:

  1. Click Create bucket
  2. Enter any bucket name and click Create bucket
❌ Expected Result: You should see an Access Denied or insufficient permissions error. This confirms that lab-user has read-only access to S3 — they cannot create, modify, or delete resources.
This clearly demonstrates how IAM policies control access. The read-only policy allows List and Get operations but denies Put, Create, and Delete operations.

🔎 Test EC2 Access (Full Access via Group)

  1. Still in the lab-user session, navigate to EC2 using the search bar
  2. You should be able to view EC2 instances, security groups, and other resources
  3. Explore the EC2 Dashboard — note that you have full access
✅ Expected Result: The EC2 console loads fully. You can view instances, security groups, key pairs, and other EC2 resources. Lab-user has full EC2 access because they are a member of the EC2-Admins group, which has the AmazonEC2FullAccess policy attached.

💡 Understanding Permission Sources

lab-user currently has permissions from two sources:

  • Direct: AmazonS3ReadOnlyAccess (attached to the user)
  • Group: AmazonEC2FullAccess (inherited from EC2-Admins group)

IAM evaluates permissions as the union of all attached policies (unless there is an explicit Deny).

This is a great teaching moment. Ask students to try accessing another service like RDS or Lambda. They should see access denied, since lab-user only has S3 read and EC2 full access. Reinforce the concept that IAM defaults to deny unless explicitly allowed.

Step 9

Clean Up

Clean up is important to avoid confusion in later labs and to practice good cloud hygiene.

🗑 Clean Up — Sign Out and Return

  1. In the incognito window (lab-user session), click the account name in the top-right corner and select Sign out
  2. Close the incognito/private browser window
  3. Return to your original browser window with the admin session
  4. Navigate to the IAM service
⚠️ Important: Ensure you are signed in as the admin user (not lab-user) before proceeding with the cleanup steps. Check the user name in the top-right corner.
Verify students are back in their admin session before proceeding with deletion steps.

🗑 Clean Up — Policy and Group

Delete resources in the following order:

  1. Delete the custom policy:
    • Go to Policies, search for EmployeeDirectoryS3ReadPolicy
    • Select the policy, click Actions → Delete, type the name to confirm
  2. Remove lab-user from the group:
    • Go to User groups, click EC2-Admins
    • On the Users tab, select lab-user and click Remove users
The order matters: delete the policy first (since it is not attached to anything, it can be deleted directly). Remove the user from the group before deleting the group.

🗑 Clean Up — Group and User

  1. Delete the group:
    • Go to User groups, select EC2-Admins
    • Click Delete, type the group name to confirm
  2. Delete the user:
    • Go to Users, select lab-user
    • Click Delete, type the user name to confirm
Finally delete the user. If the policy was attached to a user or group, it would need to be detached first.

🗑 Verify Clean Up

Confirm all lab resources have been removed:

  1. Users: lab-user should no longer appear in the Users list
  2. User groups: EC2-Admins should no longer appear in the Groups list
  3. Policies: EmployeeDirectoryS3ReadPolicy should no longer appear in the Customer managed policies
✅ Expected Result: All three resources (user, group, policy) have been successfully deleted. The IAM Dashboard resource counts should reflect the removals.
Give students a moment to verify. If any resources remain, help them complete the cleanup. Reinforce that in production, you would also review CloudTrail logs to confirm the actions.
