📦 Lab 4: Create an Amazon S3 Bucket

AWS Technical Essentials

Duration: ~25 minutes

Lab 4 of 6. Students will create an S3 bucket, explore object storage concepts, configure static website hosting, versioning, and lifecycle policies. This builds on the employee directory app theme.

🎯 Lab Overview

In this lab, you will work with Amazon S3 to understand object storage fundamentals:

  • Create and configure an S3 bucket with best-practice settings
  • Upload objects and explore their properties
  • Work with presigned URLs for secure temporary access
  • Configure static website hosting with a bucket policy
  • Explore versioning, storage classes, and lifecycle rules
⚠️ Important: Use the same AWS Region you selected for your EC2 instances in previous labs.
Give students an overview of what they will accomplish. Emphasize that S3 is one of the most foundational AWS services and is used in nearly every AWS architecture.

🛠️ Where S3 Fits

Employee Directory Application

In our employee directory app, S3 will serve as the storage layer for:

  • Employee photos and profile images
  • Static website assets (HTML, CSS, JS)
  • Application backups and archives
Key Concept: S3 provides virtually unlimited, highly durable (99.999999999% — 11 nines) object storage with multiple storage classes for cost optimization.
Explain how S3 fits into the broader architecture. The employee directory app needs a place to store employee photos and potentially host a static frontend. S3 is the natural choice.

Step 1: Create an S3 Bucket

Setting up your object storage

Walk students through creating their first S3 bucket. Emphasize the global uniqueness requirement for bucket names.

📦 Step 1a: Navigate to S3

  1. In the AWS Management Console, type S3 in the search bar at the top
  2. Click S3 under Services
  3. You will see the S3 dashboard showing any existing buckets
  4. Click the Create bucket button (orange button)
💡 Note: S3 is a global service, but each bucket is created in a specific AWS Region. Your bucket data resides in the Region you select.
S3 appears as a global service in the console, but buckets are always regional. Data never leaves a Region unless you explicitly configure cross-region replication.

📦 Step 1b: Configure Bucket Settings

On the Create bucket page, enter the following:

  1. Bucket name: employee-directory-app-bucket-<UNIQUE_ID>
    Replace <UNIQUE_ID> with your initials and a random number (e.g., employee-directory-app-bucket-jw-8374)
  2. AWS Region: Select the same Region as your EC2 instances
  3. Object Ownership: ACLs disabled (recommended)
⚠️ Bucket names must be globally unique across all AWS accounts worldwide. If your name is taken, try a different combination.
Bucket names must be globally unique, DNS-compliant, 3-63 characters, lowercase only. Explain that ACLs disabled is the modern best practice — use IAM policies and bucket policies instead.

📦 Step 1c: Security & Versioning

Continue configuring the bucket:

  1. Block Public Access: Keep all four settings enabled (checked)
    This is the secure default — we will modify this later in the lab
  2. Bucket Versioning: Select Enable
  3. Default encryption:
    • Encryption type: Server-side encryption with Amazon S3 managed keys (SSE-S3)
    • Bucket Key: Enable (default)
  4. Click Create bucket
✅ Expected Result: You should see a green success banner: "Successfully created bucket employee-directory-app-bucket-<UNIQUE_ID>"
Emphasize that Block Public Access is a safety net that prevents accidental public exposure. Versioning protects against accidental deletes. SSE-S3 is free encryption at rest.

Step 2: Explore Bucket Properties

Understanding your bucket configuration

Now that the bucket is created, have students explore the various tabs and settings to understand what was configured.

🔍 Step 2a: Review Properties Tab

  1. Click on your newly created bucket name in the bucket list
  2. Click the Properties tab
  3. Scroll through and verify the following settings:

✅ Configured Settings

  • Bucket Versioning: Enabled
  • Default encryption: SSE-S3
  • Bucket Key: Enabled

🔴 Default Settings

  • Static website hosting: Disabled
  • Requester pays: Disabled
  • Object Lock: Disabled
Walk through each property. Explain what each setting does. Object Lock is for compliance/WORM requirements. Requester pays shifts data transfer costs to the requester.

🔍 Step 2b: Review Permissions Tab

  1. Click the Permissions tab
  2. Review the following sections:

Permissions Overview

  • Block public access: All four settings should be On (blocking all public access)
  • Bucket policy: Currently empty — no custom access policies defined yet
  • Access control list (ACL): Disabled (as configured during creation)
  • CORS: No configuration — used for cross-origin web requests
Key Concept: S3 uses a layered security model: IAM policies, bucket policies, Block Public Access, ACLs, and encryption all work together.
Emphasize the defense-in-depth approach. Block Public Access is an account-level or bucket-level override that prevents any policy from granting public access. This was introduced after several high-profile S3 data exposures.

Step 3: Upload Objects

Adding files to your bucket

Students will upload their first objects. Any small files will work — images, text files, etc.

📤 Step 3: Upload Objects

  1. Click the Objects tab in your bucket
  2. Click the Upload button
  3. Click Add files and select 2–3 small image files from your computer
    (Any files will work — JPG, PNG, or even text files)
  4. Under Properties, verify the storage class is set to Standard
  5. Leave all other settings as default
  6. Click Upload
✅ Expected Result: You should see a success message showing all files uploaded successfully. Click Close to return to the bucket, and verify your objects appear in the object list.
If students do not have image files handy, they can create simple text files or download sample images from the internet. The file type does not matter for this exercise.

Step 4: Explore Object Properties

Understanding S3 objects and presigned URLs

This step teaches students about object metadata and presigned URLs, which are essential for providing temporary access to private objects.

📄 Step 4a: Review Object Properties

  1. Click on one of your uploaded objects (click the object name)
  2. Review the Object overview section:

Object Metadata

  • Key: The object name (file path within the bucket)
  • Size: The object size in bytes
  • Type: The content type (e.g., image/jpeg)
  • Storage class: Standard
  • Object URL: The direct URL to the object
  • Version ID: A unique identifier for this version
Explain that every object has a unique key within the bucket. The combination of bucket name and key uniquely identifies every object in S3 globally.

🔗 Step 4b: Test the Object URL

  1. Copy the Object URL from the object details page
  2. Open it in a new browser tab
  3. Go back to the object in the S3 console
❌ Expected Result: You will see an Access Denied XML error message. This is expected because Block Public Access is enabled.
Presigned URLs are critical for applications that need to provide temporary access to private objects. They are commonly used for file downloads, image previews, and upload workflows. The URL expires after the specified duration.

🔗 Step 4c: Generate a Presigned URL

  1. Select the object using the checkbox, then click Actions → Share with a presigned URL
  2. Set the time interval to 5 minutes
  3. Click Create presigned URL
  4. Copy the presigned URL and open it in a new browser tab
✅ Expected Result: The object should load successfully! The presigned URL contains temporary authentication credentials embedded in the query string.
Presigned URLs are critical for applications that need to provide temporary access to private objects. They are commonly used for file downloads, image previews, and upload workflows. The URL expires after the specified duration.

Step 5: Create a Folder Structure

Understanding S3 key prefixes

This step demonstrates how S3 simulates folders using key prefixes. It is important that students understand S3 is a flat namespace.

📁 Step 5: Create a Folder

  1. In your bucket, click Create folder
  2. Folder name: employee-photos
  3. Click Create folder
  4. Click on the employee-photos/ folder to navigate into it
  5. Click Upload and upload an image file into this folder
  6. After uploading, note the object key: employee-photos/yourfile.jpg
This is a commonly misunderstood concept. The folder is actually a zero-byte object with the key ending in a forward slash. Applications use the delimiter and prefix parameters in API calls to simulate folder navigation.

📁 S3 Flat Namespace

Key Concept: S3 does not actually have folders or directories. It is a flat namespace. What appears as a folder is simply a shared key prefix. The object key employee-photos/image.jpg is a single string — the / is just part of the name. The console displays prefixes as folders for convenience.

Step 6: Enable Static Website Hosting

Serving web content from S3

Static website hosting is a powerful S3 feature that allows you to serve HTML, CSS, and JavaScript directly from a bucket. This is the foundation for many serverless web architectures.

🌐 Step 6a: Enable Static Website Hosting

  1. Go to your bucket's Properties tab
  2. Scroll down to Static website hosting
  3. Click Edit
  4. Select Enable
  5. Index document: index.html
  6. Error document: error.html
  7. Click Save changes
✅ Expected Result: Static website hosting is now enabled. Note the Bucket website endpoint URL displayed — copy this for later use. It will look like: http://BUCKET-NAME.s3-website-REGION.amazonaws.com
Point out the difference between the S3 REST endpoint and the website endpoint. The website endpoint supports index documents and redirects. The REST endpoint does not.

🌐 Step 6b: Create index.html

On your local computer, create a file named index.html with this content:

<!DOCTYPE html>
<html>
<head><title>Employee Directory</title></head>
<body>
  <h1>Employee Directory Application</h1>
  <p>Welcome to the Employee Directory static site
     hosted on S3!</p>
</body>
</html>
Have students try the endpoint to see the 403 error. This reinforces that enabling static website hosting alone is not enough — you also need to configure permissions.

🌐 Step 6c: Upload index.html

  1. In your S3 bucket, click Upload
  2. Upload the index.html file
  3. Click Upload, then Close
⚠️ Note: If you try the website endpoint URL now, you will still get a 403 Forbidden error. Block Public Access is still enabled, so the website is not publicly accessible yet.
Have students try the endpoint to see the 403 error. This reinforces that enabling static website hosting alone is not enough — you also need to configure permissions.

Step 7: Configure Bucket Policy

Enabling public website access

This step involves modifying security settings. Emphasize that this is only appropriate for public website content and should be done carefully in production.

🔓 Step 7a: Disable Block Public Access

  1. Go to your bucket's Permissions tab
  2. Under Block public access (bucket settings), click Edit
  3. Uncheck the Block all public access checkbox
  4. Click Save changes
  5. Type confirm in the confirmation dialog and click Confirm
⚠️ Production Warning: Disabling Block Public Access should only be done when you intentionally want to serve public content (such as a static website). In production, use CloudFront with Origin Access Control instead of making buckets directly public.
Stress the importance of this being a deliberate action. AWS requires the typed confirmation specifically because this is a sensitive security change. In real-world scenarios, CloudFront + OAC is the recommended pattern.

🔓 Step 7b: Add Bucket Policy

  1. Still on the Permissions tab, scroll to Bucket policy
  2. Click Edit
  3. Paste the following policy (replace BUCKET_NAME with your actual bucket name):
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "PublicReadGetObject",
            "Effect": "Allow",
            "Principal": "*",
            "Action": "s3:GetObject",
            "Resource": "arn:aws:s3:::BUCKET_NAME/*"
        }
    ]
}
  1. Click Save changes
Walk through the policy structure: Version, Statement array, Sid for identification, Effect Allow, Principal * means everyone, Action is limited to GetObject (read only), Resource uses the bucket ARN with /* for all objects. Emphasize that this only allows reading objects, not listing or writing.

🔓 Step 7c: Test the Website

  1. Go to your bucket's Properties tab
  2. Scroll down to Static website hosting
  3. Click the Bucket website endpoint URL
✅ Expected Result: Your browser should display the Employee Directory page with the heading "Employee Directory Application" and the welcome message. Congratulations — you are hosting a static website on S3!
Key Concept: Two things were required to make this work: (1) disabling Block Public Access and (2) adding a bucket policy that grants s3:GetObject to everyone. Both are necessary.
Make sure every student can see their website. If anyone gets errors, check that the bucket policy has the correct bucket name and that Block Public Access is disabled. Common mistake: forgetting the /* at the end of the Resource ARN.

Step 8: Explore Versioning

Protecting against accidental changes

Versioning is a critical data protection feature. This step demonstrates how it works in practice.

🔄 Step 8a: Upload a New Version

  1. Modify your local index.html file — change the content to: 
    <h1>Employee Directory Application - v2</h1>
<p>Updated version of the Employee Directory!</p>
  2. Upload the modified index.html to your bucket (same process as before)
  3. Refresh your website endpoint — you should see the updated text
  4. In the S3 console, click on index.html
Show students both versions and explain that the latest version is served by default. Discuss the cost implications — all versions count toward storage costs. Mention that lifecycle rules can be used to manage old versions.

🔄 Step 8b: Review Versions

  1. Click the Versions tab on the index.html object
  2. You should see two versions listed, each with a unique Version ID
  3. Back in the Objects list, toggle Show versions to see version IDs for all objects
Key Concept: Versioning protects against accidental overwrites and deletes. When you delete a versioned object, S3 inserts a delete marker rather than permanently removing the data. You can restore any previous version.
Show students both versions and explain that the latest version is served by default. Discuss the cost implications — all versions count toward storage costs. Mention that lifecycle rules can be used to manage old versions.

Step 9: Explore Storage Classes

Optimizing storage costs

Storage classes are how AWS provides different cost tiers for different access patterns. This is a review-only step — students should not change any settings.

💲 Step 9: Review Storage Classes

  1. Select one of your uploaded objects using the checkbox
  2. Click Actions → Edit storage class
  3. Review the available storage classes (do not change the setting):
Storage ClassUse CaseRetrieval
StandardFrequently accessed dataImmediate
Intelligent-TieringUnknown or changing access patternsImmediate
Standard-IAInfrequent access, rapid retrieval neededImmediate
Glacier Flexible RetrievalArchive, minutes to hours retrieval1–12 hours
Glacier Deep ArchiveLong-term archive, rare access12–48 hours
  1. Click Cancel — do not change the storage class
Walk through each storage class and its pricing model. Emphasize that Standard-IA has a minimum storage duration of 30 days and a per-GB retrieval fee. Glacier classes have even longer minimums. Intelligent-Tiering automatically moves objects between tiers based on access patterns.

Step 10: Create a Lifecycle Rule

Automating cost optimization

Lifecycle rules automate the transition of objects between storage classes and eventual deletion. This is essential for cost management at scale.

♻️ Step 10a: Create the Rule

  1. Go to your bucket's Management tab
  2. Under Lifecycle rules, click Create lifecycle rule
  3. Lifecycle rule name: MoveToIA
  4. Choose a rule scope: Select Apply to all objects in the bucket
  5. Check the acknowledgement box that this applies to all objects
Lifecycle rules can be scoped to specific prefixes or tags, but for this exercise we apply it to the entire bucket. In production, you would typically scope rules to specific prefixes or object tags.

♻️ Step 10b: Configure Transitions

Under Lifecycle rule actions, check the following:

  1. ☑ Transition current versions of objects between storage classes
    • Add transition: Standard-IA — after 30 days
    • Add transition: Glacier Flexible Retrieval — after 90 days
  2. ☑ Expire current versions of objects
    • After 365 days
  3. Click Create rule
✅ Expected Result: The lifecycle rule MoveToIA appears in the lifecycle rules list with status Enabled.
Explain the cost savings at each tier. Standard costs about $0.023/GB, Standard-IA about $0.0125/GB, Glacier about $0.004/GB. Over the lifecycle of data, this can represent significant savings. The 30-day minimum for IA transitions is enforced by S3.

♻️ Lifecycle Rule Timeline

What This Rule Does

Day 0–30: Standard → Day 30: Standard-IA → Day 90: Glacier Flexible Retrieval → Day 365: Deleted

This automates cost optimization as data ages and is accessed less frequently.

Step 11: Clean Up

Restoring secure defaults

Clean up is important to restore security settings. Students should keep the bucket and objects for later labs but remove the public access configuration.

🚐 Step 11: Clean Up

Restore the bucket to a secure state while keeping it for later labs:

  1. Delete the lifecycle rule:
    Management tab → Select MoveToIA → Delete → Confirm
  2. Remove the bucket policy:
    Permissions tab → Bucket policy → Edit → Delete all policy text → Save changes
  3. Re-enable Block Public Access:
    Permissions tab → Block public access → Edit → Check Block all public access → Save changes → Type confirm
  4. Disable static website hosting:
    Properties tab → Static website hosting → Edit → Select Disable → Save changes
📝 Keep the bucket! Do not delete the bucket or its objects. We will use this bucket in later labs.
Verify that students have completed all cleanup steps. The bucket should be private again with no public access. The objects and versioning should remain intact for use in later labs.
