📦 Lab 4: Create an Amazon S3 Bucket

AWS Technical Essentials

Duration: ~15 minutes

Lab 4 of 6. Students will create an S3 bucket, upload objects, test presigned URLs, and configure static website hosting.

🎯 Lab Overview

📋 Objectives

  • Create an S3 bucket with versioning
  • Upload objects and test access
  • Generate a presigned URL
  • Enable static website hosting
  • Add a public bucket policy

📝 Lab Details

  • Duration: ~15 minutes
  • Region: Same as previous labs
⚠️ Bucket names must be globally unique across all AWS accounts.
Give students an overview. S3 is one of the most foundational AWS services. This lab covers the core operations they need to understand.

📦 Create a Bucket with Versioning

  1. Navigate to S3 in the console
  2. Click Create bucket
  3. Bucket name: employee-directory-app-bucket-<YOUR_INITIALS>-<RANDOM>
  4. Region: Same as previous labs
  5. Block Public Access: Keep all checked (for now)
  6. Bucket Versioning: Enable
  7. Encryption: SSE-S3 (default)
  8. Click Create bucket
✅ Expected: Green success banner confirming bucket creation.
Bucket names must be globally unique, DNS-compliant, 3-63 characters, lowercase. If the name is taken, try a different combination. Versioning protects against accidental deletes. SSE-S3 is free encryption at rest.

📤 Upload Objects

  1. Click on your bucket name to open it
  2. Click Upload
  3. Click Add files and select 2-3 small files from your computer
  4. Click Upload, then Close
✅ Expected: Files appear in the objects list with their size and storage class (Standard).
Any files work — images, text files, etc. If students do not have files handy, they can create a simple text file. The file type does not matter for this exercise.

🔗 Test Access: Object URL vs Presigned URL

  1. Click on any uploaded object name
  2. Copy the Object URL and open it in a new tab
❌ Expected: Access Denied — Block Public Access is enabled.
  1. Back in the console, select the object checkbox
  2. Click Actions → Share with a presigned URL
  3. Set duration to 5 minutes, click Create presigned URL
  4. Open the presigned URL in a new tab
✅ Expected: The object loads. Presigned URLs include temporary credentials in the query string.
This demonstrates the difference between direct object URLs (blocked) and presigned URLs (temporary authenticated access). Presigned URLs are commonly used for file downloads, image previews, and upload workflows. They expire after the specified duration.

🌐 Enable Static Website Hosting

  1. Go to your bucket's Properties tab
  2. Scroll to Static website hosting, click Edit
  3. Select Enable
  4. Index document: index.html
  5. Click Save changes
  6. Note the Bucket website endpoint URL

Create a file named index.html on your computer with:

<!DOCTYPE html>
<html>
<head><title>Employee Directory</title></head>
<body>
  <h1>Employee Directory Application</h1>
  <p>Welcome to the Employee Directory static site
     hosted on S3!</p>
</body>
</html>
  1. Upload index.html to your bucket
If students try the website endpoint now, they will get a 403 Forbidden because Block Public Access is still on. This is intentional — the next slide adds the bucket policy to enable public access.

🔓 Add Bucket Policy for Public Access

  1. Go to Permissions tab
  2. Under Block public access, click Edit, uncheck all, Save, type confirm
  3. Under Bucket policy, click Edit and paste:
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "PublicReadGetObject",
            "Effect": "Allow",
            "Principal": "*",
            "Action": "s3:GetObject",
            "Resource": "arn:aws:s3:::BUCKET_NAME/*"
        }
    ]
}
  1. Replace BUCKET_NAME with your actual bucket name
  2. Click Save changes
⚠️ Production note: Only disable Block Public Access for intentionally public content. Use CloudFront + OAC in production.
Two things are required: (1) disable Block Public Access and (2) add the bucket policy. Both are necessary. The policy only allows GetObject — no listing, writing, or deleting. Common mistake: forgetting the /* at the end of the Resource ARN.

🌐 Test the Website Endpoint

  1. Go to Properties tab
  2. Scroll to Static website hosting
  3. Click the Bucket website endpoint URL
✅ Expected: Your browser displays the "Employee Directory Application" page. You are hosting a static website on S3.
Make sure every student can see their website. If anyone gets errors, check: bucket policy has correct bucket name, Block Public Access is disabled, index.html is uploaded to root of bucket (not inside a folder). The website endpoint URL format is: http://BUCKET-NAME.s3-website-REGION.amazonaws.com

🗑 Clean Up

Delete the S3 bucket and its contents:

  1. Go to the S3 bucket list
  2. Select your bucket, click Empty, type permanently delete, confirm
  3. Select your bucket again, click Delete, type the bucket name, confirm
⚠️ Note: You must empty a bucket before you can delete it. With versioning enabled, "Empty" removes all object versions and delete markers.
Emptying the bucket removes all current and previous versions. This is required before deletion. If students skip the empty step, the delete will fail.
