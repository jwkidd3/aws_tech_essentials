Step 1

Create an Application Load Balancer

This is the first major component. The ALB will sit in front of the application instances and distribute incoming HTTP traffic across multiple Availability Zones.

🔧 Step 1a: Create the Load Balancer

  1. Navigate to EC2 in the AWS Management Console
  2. In the left navigation, scroll down to Load Balancers
  3. Click “Create load balancer”
  4. Under Application Load Balancer, click “Create”
Why ALB? Application Load Balancers operate at Layer 7 (HTTP/HTTPS) and are ideal for web applications. They support path-based routing, host-based routing, and integrate with Auto Scaling.
Explain the difference between ALB, NLB, and CLB briefly. ALB is the right choice for HTTP-based web applications like our Flask app.

🔧 Step 1b: Basic Configuration

Configure the load balancer settings:

SettingValue
Load balancer nameemployee-directory-alb
SchemeInternet-facing
IP address typeIPv4
Internet-facing means the ALB gets a public DNS name and accepts traffic from the internet. An internal ALB would only be accessible from within the VPC.
The scheme is important — internet-facing gives us a public endpoint. Internal ALBs are used for microservices communication within a VPC.

🔧 Step 1c: Network Mapping

Under Network mapping:

  1. VPC: Select employee-directory-vpc
  2. Availability Zones: Check BOTH public subnets
⚠️ Important: You must select subnets in at least two different Availability Zones. This is what provides high availability — if one AZ goes down, the ALB routes traffic to the other.
Key Concept: The ALB itself is a managed, highly available service. By selecting multiple AZs, AWS deploys ALB nodes in each zone automatically.
Emphasize that the ALB requires at least two AZs. This is a fundamental requirement for high availability. Each AZ gets its own ALB node, and Route 53 handles DNS-based load balancing across them.

🔧 Step 1f: Health Check Configuration

Configure the health check settings:

How health checks work: The ALB sends an HTTP GET to / every 30 seconds. After 2 successful responses (HTTP 200), the target is marked healthy. After 5 failures, it is marked unhealthy and removed from rotation.
Health checks are critical for high availability. The ALB will only send traffic to healthy targets. The path "/" works because our Flask app returns a 200 on the root endpoint.

Step 3

Update the Application Security Group

Now we lock down the application so it only accepts traffic through the ALB. This is an important security best practice.

🔒 Step 3: Restrict Traffic (1/2)

  1. Navigate to EC2 > Security Groups
  2. Select employee-directory-app-sg
  3. Click “Edit inbound rules”
This is a security group chaining pattern. The app SG trusts the ALB SG, not a CIDR block. This is more secure and dynamic — if the ALB IPs change, the rule still works.

🔒 Step 3: Restrict Traffic (2/2)

  1. Find the HTTP (port 80) rule
  2. Change the Source from 0.0.0.0/0 to the ALB security group:
    Type employee-directory-alb-sg and select it from the dropdown
  3. Click “Save rules”
Security Best Practice: By referencing the ALB security group as the source, only traffic originating from the ALB can reach the application. Direct access via the instance’s public IP on port 80 is now blocked.
This is a security group chaining pattern. The app SG trusts the ALB SG, not a CIDR block. This is more secure and dynamic — if the ALB IPs change, the rule still works.

🔒 Step 3: Verify the Security Change

✅ Should Work

Access the app via the ALB DNS name:

http://<ALB_DNS_NAME>

The Employee Directory should load normally.

❌ Should NOT Work

Access the app via the instance public IP:

http://<INSTANCE_IP>

The connection should time out.

✅ Checkpoint: ALB access works. Direct IP access on port 80 is blocked. Traffic must flow through the load balancer.
Have students test both paths. The direct IP access should time out (not get a connection refused). This confirms the security group change is working correctly.

Step 5

Create an Auto Scaling Group

The Auto Scaling Group is the core component that manages the fleet of instances. It maintains desired capacity, replaces unhealthy instances, and scales based on demand.

📈 Step 5a: ASG Name and Launch Template

  1. Navigate to EC2 > Auto Scaling Groups
  2. Click “Create Auto Scaling group”
SettingValue
Auto Scaling group nameemployee-directory-asg
Launch templateemployee-directory-lt
  1. Click Next
The ASG references the launch template we just created. The ASG wizard walks through several pages of configuration — take it step by step.

📈 Step 5b: Network Configuration

On the Choose instance launch options page:

  1. VPC: Select employee-directory-vpc
  2. Availability Zones and subnets: Select BOTH public subnets
  3. Click Next
💡 Multi-AZ Deployment: By selecting subnets in two different Availability Zones, Auto Scaling will distribute instances evenly across both AZs. If one AZ experiences issues, the other AZ continues serving traffic.
This is the same multi-AZ pattern used by the ALB. The ASG will try to balance instances evenly across the selected subnets. This is the foundation of high availability on AWS.

📈 Step 5c: Attach Load Balancer

On the Configure advanced options page:

  1. Load balancing: Select “Attach to an existing load balancer”
  2. Choose “Choose from your load balancer target groups”
  3. Select: employee-directory-tg

Health Checks:

SettingValue
Health check type☑ Turn on Elastic Load Balancing health checks
Health check grace period300 seconds
  1. Click Next
Enabling ELB health checks means the ASG will use the ALB's health check results in addition to EC2 status checks. The 300-second grace period gives new instances time to boot and start the application before health checks begin.

📈 Understanding Health Checks

EC2 Health Checks

  • Default for all ASGs
  • Checks if the instance is running
  • Detects hardware/hypervisor failures

ELB Health Checks

  • Enabled when attached to a load balancer
  • Checks if the application responds
  • Detects app crashes, port not listening
Grace Period (300s): New instances get 5 minutes to boot and start the Flask application before health checks begin. Without this, instances could be terminated before they finish initializing.
The combination of EC2 and ELB health checks gives comprehensive coverage. EC2 checks detect infrastructure issues, while ELB checks detect application-level problems.

📈 Step 5d: Configure Group Size and Scaling

On the Configure group size and scaling page:

Group Size:

SettingValue
Desired capacity2
Minimum capacity2
Maximum capacity4
What these mean:
  • Desired: How many instances the ASG tries to maintain right now
  • Minimum: The ASG will never scale below this number
  • Maximum: The ASG will never scale above this number
With min=2, we always have at least 2 instances for redundancy. With max=4, we can handle traffic spikes by doubling capacity. The desired starts at 2.

📈 Step 5e: Scaling Policies

Still on the same page, configure the scaling policy:

  1. Select “Target tracking scaling policy”
SettingValue
Scaling policy nameTarget Tracking Policy
Metric typeAverage CPU utilization
Target value60
  1. Click Next through the remaining pages (notifications, tags)
  2. Review the configuration and click “Create Auto Scaling group”
How Target Tracking Works: Auto Scaling automatically adjusts the number of instances to keep average CPU utilization at 60%. If CPU exceeds 60%, it scales out (adds instances). If CPU drops well below 60%, it scales in (removes instances). Target tracking is the simplest and most effective scaling policy. AWS manages the CloudWatch alarms and scaling actions automatically. 60% CPU is a good target — it leaves headroom for traffic spikes while keeping utilization reasonable.

Step 7

Test High Availability (Self-Healing)

This is the most exciting part of the lab — students will intentionally break something and watch the infrastructure heal itself automatically.

💥 Step 7a: Terminate an Instance

  1. Go to EC2 > Instances
  2. Identify one of the instances launched by the Auto Scaling group
    (check the “Auto Scaling group” column or instance tags)
  3. Select the instance → Instance state > Terminate instance
  4. Confirm the termination
👀 Now observe what happens: Keep the instances page open and watch closely over the next 1–2 minutes.
Tell students to watch carefully. This demonstrates the self-healing nature of Auto Scaling. The terminated instance will be replaced automatically.

💥 Step 7b: Observe Self-Healing

After terminating the instance, watch for these events:

  1. The ALB immediately stops sending traffic to the terminated instance
  2. The Auto Scaling group detects that the instance count has dropped below desired capacity
  3. A new instance is automatically launched to replace the terminated one
  4. The new instance passes health checks and the ALB begins routing traffic to it
✅ Checkpoint: After a few minutes, you should see 2 healthy instances again in the target group. The application continued working through the ALB during the entire process.
This is self-healing! No manual intervention was required. Auto Scaling maintained the desired capacity automatically. Emphasize that during this entire process, the application remained available to end users. The ALB routed all traffic to the remaining healthy instance while the replacement launched. This is the power of combining ALB with Auto Scaling.

Step 8 (Optional)

Test Scaling with a Stress Test

This step is optional but highly recommended if time permits. It demonstrates dynamic scaling based on CPU utilization — the target tracking policy in action.

🔥 Step 8: Stress Test (Optional)

  1. SSH into one of the Auto Scaling instances: 
    ssh -i employee-dir-key.pem ec2-user@<INSTANCE_PUBLIC_IP>
  2. Run the stress tool to spike CPU utilization: 
    stress --cpu 2 --timeout 300
  3. This runs for 5 minutes, consuming CPU on 2 cores
What to watch:
  • CloudWatch CPU metric increases above 60%
  • After a few minutes, Auto Scaling launches additional instances (up to max of 4)
  • When the stress test ends, CPU drops and instances eventually scale back in
The stress tool was installed by the user data script. The target tracking policy will detect CPU above 60% and scale out. Scale-in happens more slowly (typically 5-15 minutes after load drops) to avoid flapping.
