This script is similar to Lab 2 but adds the stress tool and embeds the Instance ID and AZ in the page. When students refresh the ALB DNS, they will see different instance IDs, proving load balancing works. No manual edits needed — the script detects instance metadata automatically.

📊 Create Auto Scaling Group

  1. Navigate to EC2 → Auto Scaling Groups
  2. Click Create Auto Scaling group
  3. Name: employee-directory-asg
  4. Launch template: employee-directory-lt
  5. Click Next
  6. VPC: employee-directory-vpc
  7. Subnets: Select both public subnets
  8. Click Next
The ASG uses the launch template to know what kind of instances to create. Selecting both subnets means instances will be distributed across AZs for high availability.

Attaching the ASG to the target group means new instances are automatically registered with the ALB. ELB health checks mean unhealthy instances get replaced. Desired=2 means 2 instances launch immediately, one in each AZ.

🌐 Verify — Test the ALB

  1. Go to EC2 → Instances — confirm 2 new instances are Running
  2. Go to Load Balancers → select your ALB
  3. Copy the DNS name
  4. Open in a browser: http://<ALB_DNS_NAME>
  5. Refresh the page several times
✅ Expected: The Employee Directory page loads. Refreshing shows different Instance IDs and AZs — the ALB is distributing traffic.
It takes 2-3 minutes for instances to pass health checks and be registered. If the page does not load, check: ALB status is Active, target group shows healthy targets, instances are in Running state. The instance ID changes on refresh prove load balancing is working.

🔨 Test Self-Healing

  1. Go to EC2 → Instances
  2. Select one of the ASG instances
  3. Click Instance state → Terminate instance
  4. Watch the Auto Scaling Groups activity tab
✅ Expected: Within 1-2 minutes, Auto Scaling launches a replacement instance to maintain the desired count of 2. The ALB continues serving traffic through the remaining healthy instance.
This is the key demonstration of self-healing. The ASG detects the terminated instance, launches a replacement, and the ALB automatically registers it. Users experience no downtime because the other instance continues serving traffic. Check the Activity tab to see the scaling event.

📊 View CloudWatch Metrics

  1. Go to EC2 → select an instance → Monitoring tab
  2. Review: CPU utilization, network in/out, status checks
  3. Go to Load Balancers → select ALB → Monitoring tab
  4. Review: request count, healthy host count, response time
💡 Key metrics: The ALB Healthy Host Count dropped to 1 when you terminated an instance and returned to 2 when the replacement launched.
CloudWatch metrics may take a few minutes to populate. Point out that these metrics are available automatically — no agent installation needed. In production, you would set CloudWatch Alarms to notify on unhealthy hosts or high error rates.

🗑 Clean Up — Step 1

⚠️ Delete in this order to avoid dependency errors:
  1. Auto Scaling Group: EC2 → Auto Scaling Groups → select → Delete (set desired/min to 0, then delete)
  2. Wait for all ASG instances to terminate
  3. Load Balancer: EC2 → Load Balancers → select → Delete
  4. Target Group: EC2 → Target Groups → select → Delete
Order matters. Delete the ASG first so it does not launch replacement instances. Wait for instances to fully terminate before deleting the ALB and target group. This can take 2-3 minutes.

🗑 Clean Up — Step 2

  1. Launch Template: EC2 → Launch Templates → select → Delete
  2. Security Groups: Delete employee-directory-alb-sg and employee-directory-app-sg
  3. VPC: VPC → Your VPCs → select employee-directory-vpc → Delete (this removes subnets, route tables, and IGW)
  4. Key Pair: EC2 → Key Pairs → delete employee-dir-key
  5. Any remaining EC2 instances from earlier labs → Terminate
⚠️ If VPC deletion fails, check for remaining ENIs or instances. All dependencies must be removed first.
The VPC delete action will automatically clean up subnets, route tables, and the internet gateway if there are no remaining dependencies. If it fails, look for lingering network interfaces or instances. Security groups must be deleted before the VPC.
